Milestone 03 / Metadata Model / Lesson 05

describe:
one call, ready to write

Four lessons, four separate metadata reads. This verb folds them into a single brief — the exact view an agent reads before building any create or update.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. You can now find a table's names, its writable columns, its option codes, and its lookup binds — but that's four calls and a lot of stitching. When Claude Code constructs a payload for you, it doesn't want four raw metadata dumps; it wants one consolidated, payload-shaped answer. metadata describe is that answer — and it's the natural close to "read the schema before you change it," because the very next milestone is the change.

Today's win: read a single describe brief and see every M03 idea — names, writability, options, binds — fused into a create-ready shape.

01 The one-shot brief

metadata describe <logical> makes one read-only call and returns exactly what you need to build a valid write — nothing more:describe

agent-cloud · exit 0crm --json metadata describe contact

{ "ok": true, "data": {
    "logical_name":    "contact",        # L01 — the canonical name
    "entity_set_name": "contacts",       # L01 — the URL path for the POST
    "primary_id":      "contactid",      # L02 — IsPrimaryId column
    "primary_name":    "fullname",       # L02 — IsPrimaryName column
    "writable_attributes": [            # L02 — only IsValidForCreate/Update columns
      { "logical_name": "gendercode", "attribute_type": "Picklist",
        "required_level": "None",
        "options": [ {"value":1,"label":"Male"}, {"value":2,"label":"Female"} ] },   # L03
      { "logical_name": "msa_managingpartnerid", "attribute_type": "Lookup",
        "required_level": "None",
        "bind_key": "msa_managingpartnerid@odata.bind",                        # L04
        "targets": [ { "logical": "account", "set_name": "accounts" } ] }
    ] } }

Every field traces to a lesson you just did — but pre-digested. The labels are already flattened to plain {value, label} (no localized-label nesting), and the list is pre-filtered to writable columns only. (Contact has 191 of them — the block above shows two.)

02 Why an agent loves this verb

Read the brief top-to-bottom and you can assemble a create payload without a second call:

describe fieldanswers the questionfrom
entity_set_nameWhat URL do I POST to?L01
primary_nameWhat's the one field I almost always set?L02
writable_attributes[]Which columns may I set at all?L02
required_levelWhich ones must I set?L02
options / global_optionset_idWhat code does this label map to?L03
bind_key + targets[]How do I write this lookup, and to what?L04
✓
This is the agentic contract in one call. --help teaches a human the command tree (M01 L07); describe teaches the agent the data shape. When you ask Claude Code to "create a contact named Dana, female, managed by the ITWorx account," this is the read it makes first — then it builds the payload from the brief, no guessing.

03 One more read: alternate keys

A table can also carry alternate keys — business-unique columns (an email, an order number) you can address a record by, instead of its GUID. metadata keys <logical> lists them:

agent-cloud · exit 0crm --json metadata keys contact

{ "ok": true, "data": [], "meta": { "count": 0 } }   # contact has none defined

An empty list is the common case — most stock tables have no alternate key. When one does exist, it powers upsert by business key: "create-or-update the contact whose email is X" without first looking up its GUID. You met the entity upsert verb in M02; alternate keys are what make that key-based upsert possible. We only read keys here; defining one is M04 territory.

04 Do it yourself ~5 min

All read-only. Treat describe as the payload spec it is:

# 1 · the full write-readiness brief for contact
crm --json metadata describe contact

# 2 · same for account — compare entity_set_name and primary_name
crm --json metadata describe account

# 3 · any alternate keys on either table?
crm --json metadata keys contact
crm --json metadata keys account
  1. From query 1, write out (on paper) the JSON body to create a female contact named "Dana Okoro": which fields, which gendercode value, what primary_name?
  2. Find a Lookup in query 1's writable_attributes. Using its bind_key and targets[].set_name, write the @odata.bind line to point it at an account.
  3. In query 2 (account), which writable attribute has a non-None required_level? That's the field you can't skip on create (recall L02's account name).
  4. Both keys calls return []. In your own words: what capability would a non-empty result unlock?

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · What does metadata describe give you that the individual metadata verbs don't?
Correct. describe is curated for writing: it filters to writable attributes and pre-resolves option codes and lookup binds into one create-ready view. It reads less than the raw verbs, not more.
It's a consolidation, not a data dump or a permission. One read returns the entity-set, primary id/name, and writable columns with their options and bind keys already resolved — the shape a payload needs.
2 · In a describe brief, a Lookup attribute carries bind_key and targets[]. What are they for?
Correct. That's L04 pre-digested: bind_key = the navigation-property bind name, targets[] = the valid referenced table(s) and their entity-set names — everything you need to write the lookup.
They're the lookup-write recipe: bind_key is the @odata.bind name (from the nav property), and targets[] tells you which table(s) it may reference and each one's set name.
3 · Why does describe list only writable attributes?
Correct. A payload may only contain IsValidForCreate/Update columns (L02). Listing shadow/computed/system columns would just be noise you'd have to filter out, so describe filters them for you.
By design: a write payload can only contain writable columns, so describe omits the read-only ones (shadows, computed, system) to hand you exactly the candidate fields.
4 · metadata keys contact returns []. What does that tell you?
Correct. Alternate keys are optional and separate from the GUID primary key (which always exists). Empty means no business-key upsert path — you address records by contactid. Defining one is M04.
Alternate keys are optional extras, not the primary key. Empty just means no business-key (e.g. by-email) upsert is available — the GUID primary key still works fine for everything else.
