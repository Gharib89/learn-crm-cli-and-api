Columns
are typed

A column is never just a name. It carries a data type and a set of flags that decide whether — and how — you're allowed to write it.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. In M02 you wrote gendercode as a number and bound parentcustomerid to an account — but you took on faith that those were legal moves. The attribute metadata is what tells you: this column is a Picklist, that one's a read-only mirror, this one is required on create. Read it first and a write in M04 becomes mechanical instead of trial-and-error.

Today's win: list a table's columns, then read one column's definition — its type, its writability, and whether it's required.

01 List the columns

metadata attributes <logical-name> returns every column of a table as a compact row — each with its two names (L01 applies to columns too), its type, and three writability flags:attrmeta

agent-cloud · exit 0crm --json metadata attributes contact

{ "ok": true, "data": [
    { "LogicalName": "accountid",    "AttributeType": "Lookup",
      "IsValidForCreate": false, "IsValidForRead": true, "IsValidForUpdate": false,
      "RequiredLevel": "None" },
    { "LogicalName": "accountidname", "AttributeType": "String",
      "IsValidForCreate": false, "IsValidForRead": true, "IsValidForUpdate": false },  # shadow — §03
    { "LogicalName": "gendercode",   "AttributeType": "Picklist",
      "IsValidForCreate": true,  "IsValidForRead": true, "IsValidForUpdate": true,
      "RequiredLevel": "None" }
  ] }

Three columns, three different stories — and you can read all three off the flags before touching a single write.

02 The three things every column tells you

AttributeType
The shape of the value. String, Integer, DateTime, Money, Boolean, Picklist (option-set), Lookup (foreign key), Uniqueidentifier (a GUID). This decides how you encode the value in a payload — a Picklist takes a code (L03), a Lookup takes an @odata.bind (M02 L02).
IsValidForCreate / Update
Whether you may set this column on a create / on an update. false means the platform owns it — attempting to write it errors. (IsValidForRead is almost always true; the interesting signal is the write pair.)
RequiredLevel
Whether you must supply it. None (optional) · ApplicationRequired (the app insists) · SystemRequired (the platform insists — you can't even make it optional). Skipping a required column fails the create.

For the full picture of one column, metadata attribute <logical> <column> returns the complete definition — dozens of fields including IsPrimaryId, IsPrimaryName, MaxLength, IsCustomAttribute, and the option-set when it's a Picklist:

agent-cloud · exit 0crm --json metadata attribute contact gendercode

{ "ok": true, "data": {
    "LogicalName": "gendercode", "SchemaName": "GenderCode",
    "AttributeType": "Picklist", "EntityLogicalName": "contact",
    "IsPrimaryId": false, "IsPrimaryName": false,
    "IsValidForCreate": true, "IsValidForUpdate": true,
    "IsCustomAttribute": false, "IsManaged": true,
    "RequiredLevel": { "Value": "None", "CanBeChanged": true } .. } }

✓ The single-attribute view nests RequiredLevel as an object — {Value, CanBeChanged} — where CanBeChanged:false is how SystemRequired locks itself in. The compact attributes list (§01) flattens it to just the string value.

✓
Recognise two columns instantly. IsPrimaryId:true marks the GUID key (contactid — M02 L01's PrimaryIdAttribute). IsPrimaryName:true marks the title column (fullname — what shows in lookups and lists). Every table has exactly one of each.

03 Shadow columns: the read-only mirrors

Look again at the list in §01. The accountid lookup is followed by accountidname — a String with IsValidForCreate:false and IsValidForUpdate:false. You'll see one (sometimes two — a …yominame for phonetic sort) trailing every lookup column.

These are shadow attributes: read-only mirrors the platform maintains so a query can show the related record's name without a join. They explain what you saw back in M02 L01 — a lookup surfaces as a _x_value GUID plus annotations carrying the name. The shadow column is where that name lives in storage.

ColumnTypeWrite?What it is
accountidLookupvia bindThe real foreign key. Set it with @odata.bind (M02 L02).
accountidnameStringneverRead-only mirror of the parent's name — for display.
accountidyominameStringneverPhonetic (yomi) mirror, for Japanese-style sorting.

Rule of thumb: if a String column's name is another column's name with name/yominame tacked on, and it's not writable — it's a shadow. You read it, you never set it.

04 Do it yourself ~4 min

All read-only. List, inspect, and spot the shadows:

# 1 · every column of contact — scan the AttributeType column
crm --json metadata attributes contact

# 2 · the full definition of one column — note IsPrimaryName / RequiredLevel
crm --json metadata attribute contact fullname

# 3 · a required column? inspect the primary name on account
crm --json metadata attribute account name
  1. In query 1, find fullname. Is it IsValidForCreate:true? (Surprise — it's a computed full name; check what its flags say.)
  2. Query 2: which flag marks fullname as the table's title column? What's its RequiredLevel?
  3. Query 3: account's name — what RequiredLevel does it carry, and what would happen if you tried to create an account without it?
  4. Back in query 1, count the shadow columns: how many …name strings sit right after a Lookup?

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · A column shows IsValidForCreate:false and IsValidForUpdate:false. What does that mean for your payload?
Correct. Both write flags false = a system-maintained / read-only column. Leave it out of create and update payloads; trying to set it is an error.
It's read-only — the platform maintains it. IsValidForRead is still true (you can query it), but it must stay out of write payloads. Shadow columns and computed fields look exactly like this.
2 · RequiredLevel: "SystemRequired" on a column means…
Correct. SystemRequired is the platform's own mandate (vs ApplicationRequired, which a maker sets and can lift). A create without it fails.
It's the platform's hard requirement — stronger than ApplicationRequired, which a maker can toggle. You must supply a value; you can't make it optional.
3 · You see contactidname (String, not writable) right after a Lookup column. It's…
Correct. Every lookup spawns a non-writable …name (and often …yominame) String that mirrors the parent's name so queries can show it without a join. You read it; you set the lookup itself via @odata.bind.
It's a shadow attribute — the read-only name-mirror the platform maintains alongside every lookup. The real FK is the Lookup column; the shadow is just for display.
4 · Which flag tells you a column is the table's title (what shows in a lookup picker)?
Correct. IsPrimaryName:true marks the title column (e.g. fullname on contact, name on account). IsPrimaryId is the GUID key, a different column.
It's IsPrimaryName — the title/display column. IsPrimaryId marks the GUID primary key; the two are always separate columns.
