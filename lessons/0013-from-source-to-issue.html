The mission isn't to read crm's source for its own sake — it's to drive D365 through a reviewable, agentic workflow, and that workflow gets better every time you file a precise upstream issue instead of a vague one.

Today's win: take a surprising result and produce a fileable issue — minimal repro, the --json evidence, the suspected layer, and expected-vs-actual — with the org details scrubbed.

01 What a fileable report contains

A maintainer can't fix what they can't reproduce or locate. Six parts turn a complaint into a work item — the first three are evidence, the last three are your analysis, which is exactly what M11 trained:

  1. Minimal repro — the exact command, with --json. One line, no surrounding script. evidence
  2. The envelope — paste the {ok,data,meta} output: meta.category, meta.retryable, any code. evidence · L03/M01
  3. Environment — crm --version, cloud vs on-prem, api_version, and the output of crm --json connection doctor. evidence · L05
  4. Suspected layer — name the file: commands/, core/, transport, or cli.py — and say why. analysis · L01/L04
  5. Expected vs actual — what you believed the request would do, and what it did. analysis
  6. Privacy — scrub the real org URL to the example env before posting. always
✕
Never paste a real org. Replace your cloud host with https://orgexample.crm.dynamics.com and any on-prem host with http://crm.contoso.local/Contoso. A GitHub issue is public and indexed forever — internal hostnames don't belong there.

02 Localise the layer before you file

Part 4 — the suspected layer — is where your deep-dive pays off. You don't guess; you run one diagnostic and read one signal. crm connection doctor probes the transport edge end-to-end (DNS → TCP → TLS → api_version → auth → rate-limit) and returns a per-check verdict:doctor

localise firstcrm --json connection doctor
# -> {ok, data:{checks:[{check, ok, detail, hint}, ...]}}

The logic: doctor red ⇒ it's the connection/transport, not your command — say so. doctor green but exit 2 ⇒ a usage error, so the command layer's arg parsing. doctor green, request ran but the result is wrong ⇒ the core layer built the wrong OData. You've turned "it's broken" into "the bug is in core/<group>.py."

03 Vague vs fileable — the same bug, twice

Suppose crm query odata accounts --orderby revenue returns rows in no obvious order. Here's the report most people write:

✕
"crm query sorting is broken — --orderby doesn't work. Please fix." No command shown, no output, no version, no idea which layer. The maintainer's only possible reply is a list of questions.

And here's the same bug, run through the rubric — note every claim is backed by a command output, and the analysis points at one file:

paste into the issue## Repro
crm --json query odata accounts --orderby revenue --top 3

## Actual (envelope)
{"ok": true, "data": {"value": [ ...rows NOT sorted by revenue... ]},
 "meta": {"count": 3}}

## Expected
Rows ascending by revenue (OData default for $orderby).

## Environment
crm 3.10.0 · cloud · api_version v9.2
crm --json connection doctor -> all checks ok

## Suspected layer
core/query.py — looks like --orderby isn't being mapped to the
$orderby OData param (request built wrong, not a transport issue:
doctor is green and the call succeeded with ok:true).

## Org
https://orgexample.crm.dynamics.com   (scrubbed)
✓
File it with the gh CLI against the crm repo: gh issue create --repo Gharib89/crm --title "…" --body-file issue.md. The maintainer can reproduce in one paste and already knows where to look.

04 Your turn ~8 min

A scenario. Draft the issue — then score yourself against the six-part rubric.

?
Scenario. You run crm --json entity create accounts --data '{"name":"Acme"}' and get ok:false with meta.category: "forbidden" (HTTP 403). But you can create accounts perfectly well in the web UI with the same user. Write the bug report.

Self-check — a strong draft has all six:

  1. Repro — the exact --json entity create line.
  2. Envelope — the ok:false + category:"forbidden" + any code.
  3. Environment — version, cloud/on-prem, and crucially crm --json connection whoami + doctor (does the CLI authenticate as the same user as the UI? impersonation via CRM_AS_USER?).
  4. Suspected layer — 403 + doctor-auth-green points away from transport. Likely a privilege/impersonation difference, not a crm bug — or a missing header. Naming this is the analysis.
  5. Expected vs actual — "create succeeds in UI as this user; CLI returns 403."
  6. Privacy — org scrubbed to the example host.
✓
The senior move here: a great report often concludes it isn't a crm bug at all — it's an impersonation or privilege gap your investigation surfaced. That's still a win: you saved a round-trip and you know your own stack better. Paste your draft to me and I'll score it against the rubric.

4 · A request returns ok:true and a 200, but the rows ignore your --orderby. Doctor is green. Best "suspected layer" line?
