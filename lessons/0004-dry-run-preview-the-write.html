--dry-run:
Preview the Write

See the exact request a mutation would send — checked against your live org — before a single byte is written.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. Everything so far was read-only. Your next lesson writes to the org for real — and a wrong write (wrong org, wrong field, wrong record) is the expensive mistake this whole mission is built to avoid. --dry-run is the seatbelt: it shows you the precise plan, validated against the live server, with the write held back. Build the reflex now, before you ever need it.

Today's win: take a create and an update, preview each with --dry-run, and read straight off the JSON exactly what HTTP request would hit which record in which org — having changed nothing.

01 What --dry-run actually does

Add --dry-run to any mutating command and the CLI does all the work up to the write, then stops and hands you the request it would have sent.skill You met its fingerprint in L03 — here it is in full, previewing a contact create:

agent-cloud · exit 0 · NOTHING writtencrm --json --dry-run entity create contacts --data '{"firstname":"Ada","lastname":"Lovelace"}'

{
  "ok": true,
  "data": {
    "_dry_run": true,
    "method": "POST",
    "url": "https://orgexample.crm.dynamics.com/api/data/v9.2/contacts",
    "body": { "firstname": "Ada", "lastname": "Lovelace" }
  },
  "meta": { "dry_run": true }
}

Same envelope you already know how to read: ok:true, exit 0. But the meta.dry_run:true flag changes what that success means — it's "this request would be valid," not "a record now exists." (Recall L03: never trust ok:true alone to mean a write landed.)

02 The rule that makes the preview trustworthy

The trap newcomers fall into: assuming --dry-run means "do nothing / talk to no one." It doesn't.

!
--dry-run = "no writes" — not "no traffic." Read requests (GETs) still run for real. So a preview runs the command's full pre-flight against the live org — resolving lookups, validating metadata, checking what already exists — and reports live facts, not guesses.records

That's exactly why the preview is worth trusting. When you preview a clone, the CLI actually retrieves the source record to resolve its lookups, then shows you the resolved would_create body it'd write.records When you add --validate (next section), it issues real metadata reads. The plan you're shown is the plan that would run — assembled by hitting the server, just stopping short of the one write.

The flip side, worth saying once: if a command's only effect is a read (a query, an entity get), --dry-run changes nothing about it — it runs normally and returns real data. Dry-run only ever holds back writes.

03 Reading the plan

Three fields in the preview answer the three questions you must ask before any write. Here's a create (left) next to an update of an existing account (right):

create"method": "POST"
"url": ".../v9.2/contacts"
"headers": {
  "Prefer": "return=representation"
},
"body": {
  "firstname": "Ada"
}
update"method": "PATCH"
"url": ".../accounts(a1b2c3d4-…-abc)"
"headers": {
  "If-Match": "*"
},
"body": {
  "telephone1": "+20-…"
}
method
Which verb. POST = create a new row; PATCH = update an existing one. If you meant to edit a record and see POST, stop — you're about to make a duplicate.
url
Which target — and which org. The entity set (contacts), the specific record GUID for an update, and the host. This is your wrong-org seatbelt: confirm the host is the org you intended (L02's lesson, now at write-time).
body
Exactly what changes. The fields you're sending — nothing more. An update sends only the keys you named, so you can see at a glance you're not clobbering anything else.

04 Its partner: --validate

--dry-run shows you the request; it doesn't check your field names. --validate does — it runs a few read-only metadata lookups and blocks unknown fields with a clean, actionable envelope instead of cryptic server noise. Note the position: --validate is a per-verb flag (it goes after the verb, beside --data), while --dry-run is global (before the group) — get them backwards and you'll just hit a "No such option" usage error:records

agent-cloud · exit 1 · --validate caught a typocrm --json --dry-run entity create contacts --data '{"firstnam":"Ada"}' --validate

{
  "ok": false,
  "error": "Unknown field(s) for contacts: firstnam",
  "meta": {
    "unknown_fields": ["firstnam"],
    "did_you_mean": { "firstnam": "firstname" },
    "dry_run": true
  }
}

That's ok:false at exit 1 (an operational "no," not a usage error — the field name was checked and rejected), and meta.dry_run:true confirms nothing was written. The two compose: the global --dry-run plus the per-verb --validate is the safest possible look at a write. For agent-driven writes, validate-first is the recommended default — without it, an unknown field comes back as raw OData noise an agent can't act on.records You'll lean on this pairing constantly once you start customizing.

05 Do it yourself ~3 min

Still no real writes — every line below is a preview. Run them against your active agent-cloud org:

# preview a create
crm --json --dry-run entity create contacts --data '{"firstname":"Ada","lastname":"Lovelace"}'

# preview a create with a deliberately misspelled field + validate
# (--validate is a per-verb flag — it goes AFTER the verb, like --data)
crm --json --dry-run entity create contacts --data '{"firstnam":"Ada"}' --validate

Then confirm, from your own output:

  1. In preview #1: what is method, and does the url host match the org you intend to write to?
  2. Did either command create a record? How do you know from the JSON alone? (Point to the field.)
  3. In preview #2, what did --validate put in meta — and what does did_you_mean suggest?
✓
The reflex you're building: preview → read method / url / body → confirm verb, target, org → then re-run without --dry-run to commit. Next lesson, you take that last step for real.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You run a command with --dry-run. Which best describes what hits the network?
Correct. --dry-run = "no writes," not "no traffic." The pre-flight reads (lookup resolution, validation, existence checks) run for real — which is exactly why the previewed plan reflects the live org and not a guess.
Not quite. It's not offline and nothing is written-then-rolled-back. The rule is "no writes, not no traffic": reads still run for real, so the preview is built from live server facts.
2 · A dry-run preview comes back ok:true, exit 0, meta.dry_run:true. Was a record created?
Correct. Under dry-run, ok:true means "this request would be valid," not "it ran." meta.dry_run:true is the definitive signal that nothing was written — you don't need to re-query.
Re-read §1. meta.dry_run:true is conclusive: this was a preview, so no record was created. Success here means "valid request," not "write landed."
3 · You meant to edit an existing account, but the preview shows "method": "POST". What does that warn you about?
Correct. POST = create, PATCH = update. Seeing POST when you intended an edit means the command would insert a new row — the duplicate-record footgun the preview just saved you from.
Not quite. The method is meaningful: POST creates, PATCH updates. POST when you wanted an edit = an impending duplicate, which is precisely what reading the plan catches.
4 · For an unattended, agent-driven create with a payload you assembled, what's the recommended safest look before committing?
Correct. --validate turns an unknown field into a clean unknown_fields/did_you_mean envelope (instead of cryptic OData noise), and --dry-run shows the exact request — together, the safest preview. Validate-first is the documented default for agent-driven writes.
Not the safest. Raw runs surface unknown fields as cryptic server noise; --yes only skips prompts; cleanup-after is reactive. The documented default for agent writes is --validate --dry-run — names checked and request shown, nothing written.
