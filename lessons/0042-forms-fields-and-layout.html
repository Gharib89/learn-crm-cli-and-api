Forms: fields
and layout

The M07 capstone: inspect and modify the auto-generated form, then run the full M07 round-trip and clean everything up.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads writes cleanup
Why this matters: A table's main form is the data-entry screen for individual records. D365 auto-creates an 'Information' form, but it only contains the primary-name field and ownerid. Your custom columns (ag_status, ag_priority) are in the schema but invisible on the form until you add them. The CLI handles field placement without the drag-and-drop form designer.

A form in D365 is an XML document (formXml) describing tabs, sections, and field controls. The auto-generated 'Information' form is sparse. form add-field appends a control to the first section; form remove-field removes it; form export lets you inspect the raw XML — all without opening Power Apps maker.

01 Form types and the list command

Every table can have multiple forms. Each form has a type that determines when D365 renders it. Run form list to see what exists:

shellcrm --json form list ag_ticket
envelope · live-verified 2026-06-18{
  "ok": true,
  "data": [{
    "name": "Information",
    "type": 2,
    "formid": "a1b2c3d4-0042-0000-0000-000000000001",
    "isdefault": false
  }]
}

One form: type 2 = Main. isdefault: false is expected — there's only one main form so it's the effective default in practice. D365 auto-provisioned it when the entity was created.

Form type reference:

TypeNameWhen rendered
2MainData entry / display — the form users open when clicking a record
6Quick ViewEmbedded read-only panel on another entity's form
7Quick CreateThe slide-out create form (accessible from the global nav)
8DialogDeprecated; legacy dialog forms
11CardCompact card view in activity feeds

By default form list returns main forms only. To see every type:

shell# List every form type
crm --json form list ag_ticket --all

# List only quick-create forms
crm --json form list ag_ticket --type quickcreate

--type accepts: dashboard | main | quickview | quickcreate | dialog | card.

02 Add and remove fields

form add-field <entity> <attribute> appends a control cell to the form. The CLI resolves the classid automatically from the attribute's type — you don't supply it.

TIP
One field at a time. form add-field appends to the end of the first section by default. To insert between existing fields or to lay out in a 2-column grid, open the Power Apps form designer (GUI-only for positional control).

03 Inspect the form XML

form export returns the raw formXml — the underlying XML document that D365 stores. After adding ag_status, you can see its control cell:

shellcrm --json form export ag_ticket "Information"
envelope · live-verified 2026-06-18{
  "ok": true,
  "data": {
    "entity": "ag_ticket",
    "form": "Information",
    "formxml": "<form><tabs>...</tabs></form>"
  }
}

After form add-field ag_ticket ag_status, the formXml contains a control cell like this inside a <row>:

<row>
  <cell>
    <control
       id="ag_status"
       classid="{3EF39988-22BB-4F0B-BBBE-64B5A3748AEE}"
       datafieldname="ag_status"
    />
  </cell>
</row>

The classid ties the control to its renderer; datafieldname binds it to the attribute. This is the raw structure that add-field and remove-field manipulate.

Two other form subcommands exist but aren't demonstrated here — they follow the same pattern:

  • form set-field — move an existing form field to a different tab or section
  • form clone — copy a named form to another entity

Run crm form --help for their full flag reference.

04 The M07 capstone — full round-trip writes cleanup

Done-when: create a view and a model-driven app exposing the custom table, and add a ribbon button.

Run this script start-to-finish. Each numbered comment marks a milestone step.

capstone · full round-trip · crm v1.0.0# Prerequisites: ag_ticket entity exists (from M04 pattern)

# 1. Add custom columns (if not present)
crm --json metadata add-attribute ag_ticket --kind picklist \
    --display "Status" --option "100000000:Open" --option "100000001:In Progress" \
    --option "100000002:Closed" --solution itworx_dev
crm --json metadata add-attribute ag_ticket --kind integer \
    --display "Priority" --solution itworx_dev

# 2. Get the ObjectTypeCode for view create
crm --json metadata entity ag_ticket   # note ObjectTypeCode

# 3. Create a public view   ← done-when ①
crm --json view create ag_ticket --name "Open Tickets" --otc <OTC> \
    --column ag_name:200 --column ag_status:150 --column ag_priority:100 \
    --filter-active --solution itworx_dev
#   → save savedqueryid

# 4. Create a model-driven app   ← done-when ②
crm --json app create --name "Ticket Tracker" --unique-name "itworx_tickettracker" \
    --solution itworx_dev
#   → save appmoduleid

# 5. Build a sitemap
crm --json app build-sitemap "TicketNav" \
    --area "Tickets:Ticket Management" --group "Tickets/Main:Tickets" \
    --subarea "Tickets/Main:entity=ag_ticket:Open Tickets" \
    --unique-name itworx_tickettracker --solution itworx_dev
#   → save sitemapid

# 6. Bind sitemap + view to the app
crm --json app add-components <appmoduleid> \
    --component "sitemap:<sitemapid>" --component "view:<savedqueryid>"

# 7. Add custom fields to the form
crm --json form add-field ag_ticket ag_status --solution itworx_dev
crm --json form add-field ag_ticket ag_priority --solution itworx_dev

# 8. Create a JS web resource
echo 'function showTicketId(pc){Xrm.Navigation.openAlertDialog({text:"ID: "+pc.data.entity.getId()});}' \
    > /tmp/ticket_ribbon.js
crm --json webresource create --name "itworx_/scripts/ticket_ribbon.js" \
    --file /tmp/ticket_ribbon.js --solution itworx_dev

# 9. Add ribbon button   ← done-when ③
crm --json ribbon add-button ag_ticket --label "Show ID" --location form \
    --webresource "itworx_/scripts/ticket_ribbon.js" \
    --function "showTicketId" --param PrimaryControl --solution itworx_dev

# ── CLEANUP ──────────────────────────────────────────────────────────────────
# 10. Remove form fields and ribbon button (in any order)
crm --json form remove-field ag_ticket ag_priority --solution itworx_dev
crm --json form remove-field ag_ticket ag_status --solution itworx_dev
crm --json ribbon remove ag_ticket \
    --button-id "ag_ticket.form.ShowID.CustomAction" --yes --solution itworx_dev

# 11. Delete web resource, app, view
crm --json webresource delete "itworx_/scripts/ticket_ribbon.js" --yes
crm --json app delete itworx_tickettracker --yes
crm --json entity delete savedqueries <savedqueryid> --yes   # delete the view

# 12. Delete the entity (metadata + all its records)
crm --json metadata delete-entity ag_ticket --yes --check-dependencies

Done-when checklist

GoalLessonSatisfied by
Create a view L02 + capstone step 3 view create ag_ticket --name "Open Tickets"
Create a model-driven app exposing the custom table L03 + capstone steps 4–6 app create + build-sitemap + add-components
Add a ribbon button L04 + capstone step 9 ribbon add-button ag_ticket --label "Show ID" --location form
WARN
Teardown order matters. Delete form fields and ribbon buttons before deleting the web resource (the ribbon still references it). Delete the entity last — it carries the schema that the view and form reference. You can delete the view before or after the entity; D365 cascades the savedquery delete when the entity is deleted.

05 GUI-only limits in M07

The CLI covers field placement, view columns, app structure, and basic ribbon buttons. These tasks remain GUI-only:

TaskWhy it's GUI-only
Form designer drag-and-drop (multi-column layout, tab reorder) formXml positional placement requires the canvas editor in Power Apps maker
Ribbon Workbench (complex rules, enable/display rules per button) Multi-rule RibbonDiffXml is too complex to express as per-flag CLI parameters
Business Process Flows on forms BPF editor is canvas-based; no CLI verb exposes BPF creation
App validation & publishing UI Power Apps make.powerapps.com checker — a separate quality service

Lesson check

4 questions · click to reveal

1. You run crm --json form list ag_ticket on a brand-new custom entity. What do you expect to see?

Correct. create-entity auto-provisions one main form (type=2) named 'Information'. It contains the primary name field and ownerid — your custom columns aren't on it yet.
D365 auto-provisions one main form (type=2) named 'Information' when you create an entity. It's sparse — just ag_name and ownerid. Custom columns must be added via form add-field.

2. Which command adds ag_status to the Information form of ag_ticket?

Correct. form add-field <entity> <attribute> appends the field to the first section of the main form (by default). The classid is resolved from the attribute type automatically.
Use crm form add-field ag_ticket ag_status. metadata add-attribute creates the schema column (M04 verb); form set-field moves a column that's already on the form to a different tab/section.

3. After running the full M07 capstone script, a colleague wants to see the final structure of the form before cleanup. Which command shows the raw formXml?

Correct. form export <entity> <form-name> returns the formXml showing all control cells — classid, datafieldname, tab/section structure.
form export ag_ticket "Information" returns the formXml. metadata describe shows the schema's writable attributes; view list shows views; ribbon export shows the command bar XML.

4. What is the correct teardown order for M07 artefacts?

Correct. The safe order: remove form fields and ribbon button (which reference the web resource), then delete the web resource, then delete the app (sweeps dependent rows), then delete the view, then delete the entity last (it carries the schema).
Order matters. Remove the ribbon button before deleting its web resource (the button still references it). Delete the entity last — it's the anchor everything else points to. The app delete doesn't cascade to the entity, views, or forms.
