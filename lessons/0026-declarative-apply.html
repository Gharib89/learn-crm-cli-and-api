Milestone 04 / Schema Authoring / Lesson 04

Declare it:
apply a spec

Write the whole desired state — table, columns, and a shared option set — as one reviewable file, then let the CLI converge the org to it.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. Scaffold packs a table into a command; apply -f packs it into a file. A file can be reviewed in a pull request, committed to git, and re-applied to dev/test/prod to get the same result. It's the form that fits the mission best: Claude Code writes the spec, you read the diff, then apply. This is declarative, desired-state schema authoring.

Today's win: author a YAML spec with a global option set and a table that uses it, apply it, then re-apply and watch the org say "already there — nothing to do."

01 The spec file

A spec declares entities and, optionally, shared optionsets. Here's Project with a global Priority option set it references:apply

project.yamloptionsets:
- name: itworx_priority
  display_name: Priority
  options:
  - { value: 100000000, label: Low }
  - { value: 100000001, label: Medium }
  - { value: 100000002, label: High }
entities:
- schema_name: itworx_Project
  display_name: Project
  display_collection_name: Projects
  ownership: UserOwned
  primary_attr: { schema_name: itworx_Name, label: Name }
  attributes:
  - { kind: string,   schema_name: itworx_Code,    display_name: Code, required: ApplicationRequired, max_length: 30 }
  - { kind: money,    schema_name: itworx_Cost,    display_name: Cost, precision: 2 }
  - { kind: picklist, schema_name: itworx_Priority, display_name: Priority, optionset_name: itworx_priority }
agent-cloud · exit 0crm --json apply -f project.yaml --solution itworx_dev

{ "ok": true, "data": { "applied": [
    {"kind":"entity",    "name":"itworx_Project"},
    {"kind":"optionset", "name":"itworx_priority"},    # created before the column that needs it
    {"kind":"attribute", "name":"itworx_Code"},
    {"kind":"attribute", "name":"itworx_Cost"},
    {"kind":"attribute", "name":"itworx_Priority"}
  ], "skipped":[], "failed":[] }, "meta": { "staged": false } }
!
The top-level solution must be a mapping, not a string. Writing solution: itworx_dev fails with "solution must be a mapping." The simplest path is to leave it out of the file and pass --solution itworx_dev on the command — which also overrides any solution the spec declares.

02 Desired state, in dependency order

Read the applied list again: the option set was created before the picklist column that references it. You didn't order that — apply works out the dependency graph and creates resources in the right sequence, then runs one PublishAllXml at the end.apply Each resource is created with if-exists=skip, exactly like scaffold — so the spec describes the state you want, and apply makes reality match it.

03 Re-apply: convergence, not repetition

The defining property: applying an unchanged spec is a no-op. Everything already exists, so everything is skipped — apply converges the org to the spec rather than blindly re-running steps:

agent-cloud · exit 0 · second apply, unchangedcrm --json apply -f project.yaml --solution itworx_dev

{ "ok": true, "data": {
    "applied": [],                              # nothing to do
    "skipped": [ "itworx_Project", "itworx_priority", "itworx_Code",
                 "itworx_Cost", "itworx_Priority" ],
    "failed": [] } }
✓
This is the agentic sweet spot. Ask Claude Code to "add a Project table with a code, a cost, and a priority" and it can write this file. You review the YAML like any code change, apply it, and if you re-run it nothing breaks. The spec is the reviewable artifact the GUI never gives you — and the same file deploys to every org. Editing a spec and applying it is the loop you'll use constantly from here.

04 Global vs local option sets

In L02 you built a picklist with inline options — that creates a local option set, private to one column. The spec above instead defines a global one (itworx_priority) and points the column at it. The difference matters:

local
Belongs to one column on one table. Simple, but not reusable. (L02's inline --option.)
global
A standalone, named, reusable set — many columns/tables can share itworx_priority. Defined once, referenced by optionset_name.

This is the M03 L03 read-side distinction, now from the write side — and the issue's create-optionset in declarative form. When a picklist references a pre-existing global, apply's --include-referenced-optionsets (on by default) makes sure that global is pulled into your target solution so the export is complete.apply Prefer global whenever the same choice list appears in more than one place.

05 Do it yourself ~8 min writes

Save the spec from §01 as project.yaml (drop the top-level solution; you'll pass it as a flag), then:

# 1 · preview the plan — no writes
crm --json --dry-run apply -f project.yaml --solution itworx_dev

# 2 · apply for real
crm --json apply -f project.yaml --solution itworx_dev

# 3 · apply again, unchanged — everything skips
crm --json apply -f project.yaml --solution itworx_dev

# 4 · clean up (the global option set survives the table — delete it too)
crm --json metadata delete-entity itworx_project --yes
crm --json metadata delete-optionset itworx_priority --yes
  1. In step 1, note the order of planned: where does optionset sit relative to the itworx_Priority column? Why must it come first?
  2. In step 2, count the entries in applied. In step 3, count applied vs skipped. What changed, and what does that prove?
  3. Edit project.yaml: add a { kind: boolean, schema_name: itworx_Active, display_name: Active } column, then apply again. Which buckets do the old vs new resources land in?
  4. After step 4's delete-entity, run metadata get-optionset itworx_priority. It still exists — that's why step 4 deletes it separately. Why does a global outlive the table?

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You apply an unchanged spec a second time. What's in applied?
Correct. Apply converges the org to the spec. If the org already matches, there's nothing to apply — all resources are skipped. Re-running is safe and idempotent.
An unchanged re-apply is a no-op: applied is empty and everything is skipped. That idempotency is the whole point of desired-state.
2 · Your spec has a picklist column referencing a global option set defined in the same file. What ensures the option set exists first?
Correct. Apply orders resources by dependency automatically — the applied list shows the option set created before the picklist. You just declare; apply sequences.
Apply builds a dependency graph and creates in order — option set before the column that references it. No manual ordering or depends_on needed.
3 · You wrote solution: itworx_dev as a top-level string in the spec and apply failed. Fix?
Correct. The error is "solution must be a mapping." The clean fix is the --solution flag, which also overrides whatever the spec declares.
Top-level solution must be a mapping, not a bare string. Easiest: drop it and use the --solution flag.
4 · Why prefer a global option set in a spec over an inline local one?
Correct. Global = a standalone, named set referenced by optionset_name; reuse it everywhere the same choices appear. Local belongs to one column. (And a global outlives the table — delete it separately.)
A global option set is standalone and reusable across columns and tables (referenced by name), where a local one is private to a single column. That reuse is the reason to define it once in the spec.
