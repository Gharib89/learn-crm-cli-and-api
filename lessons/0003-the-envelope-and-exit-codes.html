The Envelope
& Exit Codes

Every --json command answers in the same shape and sets an exit code. Read both, and you always know what happened.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. In L02 you saw "ok": true and moved on. But an agent can't act on a screenshot or a feeling — it needs a contract it can branch on. This lesson is that contract: the one shape every result takes, and the exit code that goes with it. It's how you'll tell a clean success from "the server said no" from "you typed the command wrong" — every single time, the same way.

Today's win: run four commands and, from the JSON shape and the exit code alone, classify each one as success, operational failure, or usage error — without reading a word of prose.

01 One shape, every time

The crm CLI talks two ways. To a human it prints friendly text. To an agent — you, driving it — it prints --json, and that mode guarantees a single, stable envelope.skill Two top-level shapes, and only two:

{ "ok": true,  "data": ..., "meta": {...} }   # it worked
{ "ok": false, "error": "...", "meta": {...} }   # it didn't

Four fields, and the rules between them never change:

ok
The branch. A boolean — true means the operation took effect, false means it didn't. Always read this first.
data
The payload — present only when ok:true. A record, a list under value, whoami's GUIDs, a dry-run preview.
error
A human-readable message — present only when ok:false. Says what went wrong.
meta
An optional sidecar of structured facts about the result — status code, category, row count, dry-run flag, warnings. Present when there's something to report; absent when there isn't.

Here's the live success you already met in L02 — notice it carries no meta at all (nothing structured to report):

agent-cloud · exit 0{
  "ok": true,
  "data": {
    "@odata.context": "https://orgexample.crm.dynamics.com/api/data/v9.2/$metadata#...WhoAmIResponse",
    "BusinessUnitId": "4a617aa7-5a3d-f111-88b5-000d3a3b279b",
    "UserId":         "bf40e4d2-ee3e-f111-88b4-6045bd001b10",
    "OrganizationId": "9102ced1-5c3e-f111-b31d-000d3a302819"
  }
}

And a query — same ok:true, but now meta appears, because a list has facts worth reporting (which set, how many rows):

agent-cloud · exit 0{
  "ok": true,
  "data": {
    "@odata.context": "https://orgexample.crm.dynamics.com/api/data/v9.2/$metadata#accounts(name)",
    "value": [
      { "name": "Contoso Ltd", "accountid": "a1b2c3d4-1111-2222-3333-000000000abc" }
    ]
  },
  "meta": { "entity_set": "accounts", "count": 1 }
}
✓
The mental model: ok is the question "did it work?"; data/error is the answer; meta is the footnotes. You branch on ok, you act on data or error, and you scan meta for the details that explain or qualify the result.

02 The exit code says the same thing — to $?

The envelope is for you to parse. But the CLI also sets a Unix exit code, readable as $? — so a plain shell script can branch without parsing any JSON at all. The two are redundant on purpose: ok:false always rides with a non-zero exit.skill

codemeaning
0success — ok:true. The operation took effect.
1operational failure — the request was well-formed, but the server (or a safety guard) said no: not-found, validation, permission, a declined destructive op.
2usage error — the command itself was malformed: bad/unknown flag, missing argument, bare crm. The server was never even asked.

Non-zero = the operation did not take effect. But which non-zero matters enormously, because it tells you whose problem it is:

exit 1 — you asked correctly; the answer was "no"

A valid command reached the server and got a real, negative answer. Here's a get on a GUID that doesn't exist:

agent-cloud · exit 1{
  "ok": false,
  "error": "Entity 'account' With Id = 00000000-... Does Not Exist",
  "meta": {
    "status": 404, "code": "0x80040217",
    "category": "not_found", "retryable": false
  }
}

On an operational failure, meta carries the structured diagnosis: the HTTP status, the Dataverse code, a coarse category (not_found, auth, validation…), and retryable — whether trying again could ever help.skill That's the channel an agent branches on, not the prose in error.

exit 2 — the command was wrong; the server never heard it

Misspell a flag and you never leave your own machine:

exit 2{ "ok": false, "error": "No such option '--no-such-flag'." }
!
The distinction that matters: both 1 and 2 are ok:false — but on a 2 you fix your invocation (a retry of the same string fails identically), while on a 1 you handle the server's answer (a retry might help — check meta.retryable). Conflating them is the classic agent bug: retrying a typo forever, or giving up on a transient error that just needed another go.

03 meta in the wild: the dry-run flag

One more meta field worth meeting now, because it's your future seatbelt. A --dry-run previews a write without issuing it — and marks the envelope with meta.dry_run: true, the canonical "this was only a preview" signal.skill

agent-cloud · exit 0 · NO write happenedcrm --json --dry-run entity create contacts --data '{"firstname":"Test"}'

{
  "ok": true,
  "data": { "_dry_run": true, "method": "POST", "url": ".../contacts", ... },
  "meta": { "dry_run": true }
}

ok:true, exit 0 — yet no record was created. The success here means "this request would be valid," not "it ran." That's exactly why you check meta.dry_run before trusting an ok:true to mean a write landed. (The full --dry-run reflex is the next lesson — for now just clock that meta is where it announces itself.)

04 Do it yourself ~3 min

Run these four, each followed by echo $? so you see the exit code right after the JSON:

crm --json connection whoami; echo $?
crm --json entity get accounts 00000000-0000-0000-0000-000000000000; echo $?
crm --json connection whoami --no-such-flag; echo $?
crm --json --dry-run entity create contacts --data '{"firstname":"Test"}'; echo $?

Now fill this in from your own output — that's the whole skill:

  1. For each: what is ok, what is $?, and is it success / operational failure / usage error?
  2. The two that printed ok:false — one was exit 1, one was exit 2. Which is which, and why does that map to "server said no" vs "I typed it wrong"?
  3. Two printed ok:true. Did both actually change the org? (Look at meta on the dry-run one.)
✓
All safe to repeat. A bogus-GUID get is read-only, a bad flag never reaches the server, and --dry-run issues no write. Run them as many times as you like.

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · A command returns {"ok":false,...} with exit code 2. What does that tell you about where the problem is?
Correct. Exit 2 = usage error — the CLI rejected your invocation locally and never made the HTTP call. Re-running the same string fails identically; fix the command. (Server-side rejections are exit 1.)
Not quite. Exit 2 means the CLI never reached the server — the command was malformed (bad flag, missing arg). A server-side "no" (auth, validation, not-found) is exit 1, not 2.
2 · You're scripting against crm and need to know whether an operation actually took effect. What do you branch on first?
Correct. ok is the designed branch point, mirrored by the exit code. data/error tell you the payload or reason, and meta the details — but the yes/no answer is ok.
No. data, error, and meta are downstream of the decision. The single field designed to answer "did it work?" is ok — and the exit code says the same thing for a shell.
3 · crm --json entity get accounts <bogus-guid> returns ok:false, meta.status:404, exit 1. Is this a usage error?
Correct. The command was perfectly valid — it reached the server, which truthfully reported the row doesn't exist. That's an operational failure (exit 1), and meta.category:"not_found" / retryable:false tells you re-running won't change the answer.
Re-read §2. The command was well-formed and did reach the server (hence the 404 and exit 1) — that's an operational failure, not a usage error. Usage errors (exit 2) never reach the server at all.
4 · A create returns "ok": true, exit 0, and meta: {"dry_run": true}. Did a new record get created in the org?
Correct. Under --dry-run, ok:true means "this request would be valid," not "it ran." meta.dry_run:true is the canonical signal that nothing was written — which is exactly why you check meta and never trust ok:true alone to mean a mutation took effect.
Not quite. meta.dry_run:true is the giveaway: this was a preview, so no record was created. It shows that ok:true means "valid request," not necessarily "write happened" — always check meta.
