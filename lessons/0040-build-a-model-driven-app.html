Build a model-driven app

Wire a table, its views, and a sitemap into a navigable D365 app — three CLI commands.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads writes cleanup
Why this matters: A table and its views are invisible until you wire them into an app. A model-driven app is the container end-users log into — it bundles a sitemap (which areas to show), the forms, and the views into one navigable experience. Three CLI steps: create the app, build the sitemap, bind the components.

Model-driven apps are D365's answer to 'how do I put my tables in front of users?' Unlike canvas apps (code-first, blank-canvas), model-driven apps are schema-driven: you declare which tables to expose and the platform renders the forms and views it already knows about.

01 App anatomy

A model-driven app is a layered container. From the outside in:

App (appmodule: ag_tickettracker) └── Sitemap (M07Sitemap) └── Area: Ticket Management └── Group: Tickets └── Subarea: ag_ticket → "Open Tickets" view └── binds to: Main form, Public views

The app (appmodule) is the top-level container — it's what appears in the D365 app-switcher. The sitemap defines what areas, groups, and subareas appear in the left navigation pane. Each subarea points to an entity; when a user clicks it, D365 renders the entity's default public view in the list pane and opens records in the main form.

add-components binds specific views, forms, charts, or dashboards to the app. Without it the app uses the entity's default resources automatically — so binding the sitemap is usually the only required step.

02 Step 1 — Create the app

Use crm app create to register a new appmodule record and publish it into a solution.

FlagPurpose
--nameDisplay name shown in the app-switcher (e.g. "Ticket Tracker").
--unique-namePublisher-prefixed schema name (e.g. ag_tickettracker). Used in all subsequent commands.
--descriptionOptional description string.
--if-existserror (default) or skip. Use skip in idempotent scripts.
--icon-webresourceOptional web resource logical name for the app icon. Defaults to the platform icon.
--solutionTarget solution unique name. The app record is created inside this solution.
 
shellcrm --json app create \
    --name "Ticket Tracker" \
    --unique-name "ag_tickettracker" \
    --description "M07 demo app" \
    --solution agsol

WARN
The app_lookup_error is a known read-back timing gap — the app was created (the POST succeeded), but the CLI's immediate GET to read it back raced the platform's replication. The appmoduleid in the envelope is still correct. Save it for the next two commands — there is nothing to re-run.

03 Step 2 — Build the sitemap

The sitemap defines the left-nav structure. app build-sitemap generates and registers the SiteMap XML in one command. The three flag types map directly to the three nav levels:

FlagSyntaxPurpose
--area"id[:Title]"Top-level nav section. The id becomes the Area's XML attribute; Title is what's displayed.
--group"areaId/groupId[:Title]"A group within the given area. The slash-path says "this group lives under this area."
--subarea"areaId/groupId:entity=<logical>[:Title]"A table entry inside a group. entity= names the D365 logical entity name.
--unique-nameag_tickettrackerLinks the new sitemap to this app at creation time.
--solutionagsolRegisters the sitemap record in the solution.

Pass --dry-run first to preview the generated SiteMapXml before committing:

shell · dry-run (no write)crm --json app build-sitemap "M07Sitemap" \
    --area "Tickets:Ticket Management" \
    --group "Tickets/Main:Tickets" \
    --subarea "Tickets/Main:entity=ag_ticket:Open Tickets" \
    --unique-name ag_tickettracker \
    --solution agsol \
    --dry-run

The dry-run shows the sitemapxml that will be sent to the API:

dry-run · sitemapxml preview<SiteMap>
  <Area Id="Tickets" Title="Ticket Management">
    <Group Id="Main" Title="Tickets">
      <SubArea Id="ag_ticket" Entity="ag_ticket" Title="Open Tickets" />
    </Group>
  </Area>
</SiteMap>

Once the preview looks correct, run without --dry-run:

shell · livecrm --json app build-sitemap "M07Sitemap" \
    --area "Tickets:Ticket Management" \
    --group "Tickets/Main:Tickets" \
    --subarea "Tickets/Main:entity=ag_ticket:Open Tickets" \
    --unique-name ag_tickettracker \
    --solution agsol

04 Step 3 — Bind components

Use app add-components to pin specific components — sitemap, views, forms, charts, dashboards, or BPFs — to the app. Each --component flag takes the form kind:guid.

Supported component kinds: view | chart | form | dashboard | sitemap | bpf

shellcrm --json app add-components <appmoduleid> \
    --component "sitemap:<sitemapid>" \
    --component "view:<savedqueryid>"
envelope · live-verified 2026-06-18, crm v1.0.0{
  "ok": true,
  "data": {
    "added": 2,
    "app_id": "a1b2c3d4-0040-0000-0000-000000000001"
  }
}
TIP
You don't have to bind every view manually. Binding the sitemap is usually enough — the app will show all public views for the entity automatically. Use --component view:<id> to pin a specific view as the only visible option.

05 Clean up — app delete

When you're done with a demo app, app delete removes the appmodule record and sweeps its FK-dependent rows in one pass. Pass the unique-name and --yes to skip the confirmation prompt.

shellcrm --json app delete ag_tickettracker --yes
envelope · live-verified 2026-06-18, crm v1.0.0{
  "ok": true,
  "data": {
    "deleted": true,
    "appmodule": "a1b2c3d4-0040-0000-0000-000000000001",
    "dependents_deleted": [
      { "entity": "dvtablesearch", "id": "a1b2c3d4-0040-0000-0000-000000000010" },
      { "entity": "appsetting", "id": "a1b2c3d4-0040-0000-0000-000000000011" }
    ],
    "dependents_skipped": [
      { "entity": "appmodulecomponent", "id": "a1b2c3d4-0040-0000-0000-000000000012" }
    ]
  }
}

dependents_deleted lists FK rows the CLI swept automatically (e.g. dvtablesearch, appsetting). dependents_skipped lists rows that were already gone — for example appmodulecomponent records that had already been unbound from the solution.

WARN
Deleting the app does NOT delete the sitemap record. The sitemap becomes orphaned (no longer linked to any app) but remains in the org and is harmless. If you need a fully clean org, also delete the sitemap via crm entity delete sitemaps <sitemapid> --yes.

Lesson check

3 questions · click to reveal

1. The app_lookup_error field appears in the create envelope. What does it mean?

Correct. app_lookup_error is a read-back timing gap. The app was created (the POST succeeded); the CLI's GET to read the result back found the record not yet replicated. Use the appmoduleid from the envelope — it's correct.
The app was created. app_lookup_error is a timing race between the POST succeeding and the GET read-back. The appmoduleid field is correct. There's nothing to re-run.

2. You want the left-nav to show "Ticket Management" as the area, "Tickets" as a group, and ag_ticket as a table entry. Which build-sitemap flags achieve this?

Correct. The syntax is id[:Title] for areas, areaId/groupId[:Title] for groups, and areaId/groupId:entity=<logical>[:Title] for subareas. The dry-run shows the generated SiteMapXml so you can verify before committing.
The sitemap flag syntax is: --area 'id[:Title]', --group 'areaId/groupId[:Title]', --subarea 'areaId/groupId:entity=<logical>[:Title]'. Each flag carries an id, the parent path, and an optional display title.

3. Which component kind would you use to pin a specific view as the only list view shown in an app?

Correct. --component view:<savedqueryid> pins a specific saved query as a visible view in the app. Without it, the app shows all the entity's public views by default.
Use --component view:<savedqueryid>. The sitemap kind binds the navigation tree; bpf is for Business Process Flows; form is for forms. Views are bound with view:<id>.
