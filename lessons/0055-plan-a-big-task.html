Milestone 10 · Working Agentically · Lesson 03

Plan it
before you mutate

A big requirement fails when you start typing verbs. The agent's real value is turning a vague ask into an ordered, dependency-aware plan — automatable steps sequenced, GUI-only pieces flagged — that you approve before the first write.

planning  ·  running example: Deal Desk  ·  no writes yet
Why this matters: The mission's success criterion is taking a real requirement — "a custom table with columns, a 1:N to account, inside a solution, plus a plug-in step" — and driving it end-to-end. The difference between a clean delivery and a half-built mess is almost entirely in the plan: what order, what depends on what, and what can't be done at all. This lesson is the planning discipline; L04 hardens it with dependency checks.

Plan mode exists for exactly this: think before you mutate. The agent reads the requirement, drafts a plan, and shows it to you — and nothing touches the org until you say go.

01 The requirement: a "Deal Desk" approval

A sales manager asks for this, in their words:

ASK
"When an Opportunity goes over $50k, route it for Deal Desk approval. Track each review on its own record with an approval status. Show the total approved value on the Account. Give reviewers their own app, and email them when a review is submitted."

That's one sentence and at least seven moving parts across four layers — schema, UI, automation, and notifications. Typing verbs now is how you end up with a lookup pointing at a table that doesn't exist yet. The agent's first move is not to build. It's to decompose.

02 Split automatable from GUI-only — first

Before ordering anything, apply L02's boundary test to every part. This is the step that prevents a silent hole:

SURFACE
The three GUI-only rows don't disappear — the agent names them now as manual tasks for you (or a maker) to do in the designer, and plans the rest around them. A plan that quietly omits the rollup is worse than one that flags it.

03 Order the automatable work by dependency

Now sequence what's left in the canonical customization order — publisher/solution, then schema, then UI, then automation, then publish — respecting each dependency edge. This is the plan the agent presents:

  1. Publisher + solution — solution create-publisher, solution create. Everything after inherits the prefix + lands here.
  2. "Deal Review" table — metadata create-entity. prereq for the lookup & step
  3. Approval-status option set — metadata create-optionset. prereq for the status column
  4. 1:N Account → Deal Review — metadata create-one-to-many --referenced-entity account --referencing-entity ag_dealreview. Needs both tables to exist.
  5. Status column on Deal Review — metadata add-attribute --kind picklist --optionset-name …. Needs the option set.
  6. Plug-in assembly — plugin register-assembly (DLL compiled outside first). prereq for the step
  7. Register step on Opportunity Update — plugin register-step --message Update --entity opportunity. Needs the assembly + confirmed plug-in type.
  8. Reviewer app + sitemap — app create, app build-sitemap. Table is invisible in the app until a subarea references it.
  9. Form fields — form add-field for the status/lookup. Needs the columns published.
  10. Publish once — solution publish-all. Stage every write with --stage-only, publish at the end.
ORDER
Every arrow in that list is a real dependency edge: a lookup needs both tables, a step needs its assembly, a subarea needs its table, a form field needs a published column. Get the order wrong and a step fails mid-run, leaving the org half-built. The order is the plan — verified against each component in L04.

04 Approve the plan, then execute under the contract

With the plan on the table, the loop is: you approve → the agent executes step by step → each write is gated by the contract from L01. Concretely, per step:

  1. Dry-run the write (--dry-run) and read the previewed request — does the URL/body match the intent?
  2. Commit only on a clean preview; check $? + the envelope (and meta.warnings) before moving on.
  3. Stage, don't publish — carry --stage-only through the batch, then one publish-all. Only published customizations export later.
  4. Stop on the first real failure — a broken step means a wrong assumption; re-plan rather than pushing through.
PLAN MODE
In Claude Code, plan mode is where this lives: the agent drafts the whole sequence and the GUI-only hand-offs without touching the org, you review it like a PR, and execution only starts on approval. Planning is free; a half-applied schema change is not.

Lesson check

4 questions · click to reveal

1. What is the agent's first move on a big requirement?

Correct. Before any write — even the publisher — the agent decomposes the requirement and classifies each part, because that's what reveals the GUI-only hand-offs and the dependency order. Creating the publisher or dry-running comes later, once there's a plan to execute.
First decompose and classify (automatable vs GUI-only). That surfaces the manual hand-offs and the build order. Creating the publisher, dry-running, or picking an environment are execution steps that follow the plan.

2. Why must the 1:N relationship come after the "Deal Review" table in the plan?

Correct. create-one-to-many binds a referenced and a referencing entity — if "Deal Review" doesn't exist yet, the call has nothing to point at and fails. That's a dependency edge, which is exactly what dictates plan order.
It's a dependency: the relationship references both the referenced and referencing entities, so "Deal Review" must already exist. It's not about speed, an intervening publish, or system-table restrictions.

3. The requirement needs a rollup of total approved value. How does the plan handle it?

Correct. Rollup columns have no CLI path, so the plan names the rollup as a manual designer task while sequencing everything that is automatable. There's no rollup kind to add, a plug-in is a different mechanism with its own tradeoffs, and silently dropping it is the failure mode the split is designed to prevent.
The rollup is GUI-only — the plan flags it as a manual task and builds the automatable parts around it. There's no --kind rollup, substituting a plug-in changes the design, and omitting it silently leaves a hole.

4. During execution, a planned step fails with a real server error. The agent should:

Correct. A genuine failure mid-plan usually means an assumption was wrong (a missing prereq, a bad name). Pushing past it builds on a broken foundation. Stopping to re-plan — and dependency-checking, per L04 — is the disciplined move. Blind retries, skipping, or publishing a half-built change all compound the problem.
Stop and re-plan: a real failure signals a wrong assumption, and continuing builds on a broken base. Retrying, skipping, or publishing partial work all make the half-built state worse.
