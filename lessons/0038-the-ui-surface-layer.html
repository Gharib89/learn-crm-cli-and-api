Milestone 07 / Forms, Views & Apps / Lesson 01

The UI surface layer

Forms, views, apps, ribbon, and web resources — the five artefacts that turn data into UX.

reads  ·  live on agent-cloud  ·  orgexample.crm.dynamics.com
Why now. M01–M06 built the data layer: tables, columns, relationships, solutions. But users never see raw API payloads — they see forms to fill in, views to search, and apps to navigate. M07 teaches the five UI artefacts that sit between your schema and the end-user's browser.

Every table you created in M04–M05 is already accessible through the Dataverse Web API. But D365 CE is a model-driven app — the platform reads your schema and renders a UI around it. That UI is built from five artefacts: forms, views, sitemaps, apps, and ribbon/web resources.

01 The five UI artefacts

Each artefact has a distinct role in the UI stack. Together they form the complete surface layer between raw data and the end-user's browser.

ArtefactWhat it is
FormData entry / display screen for a single record. One form = one row view. A table can have many forms (main, quick-create, quick-view, card).
View (savedquery)The list lens for a table — defines which columns appear and in what order. Each table has auto-created "Active" and "Inactive" system views out of the box.
SitemapThe navigation tree (Areas → Groups → Subareas) that tells a model-driven app which tables to surface in the left-hand menu.
Model-driven app (appmodule)Container that packages a sitemap, forms, and views into a first-class app. Users see this app in the D365 menu when they log in.
Ribbon / web resourceThe command bar (ribbon) above forms and views, and the static JS/HTML/CSS assets (web resources) that power ribbon button logic and custom scripts.

These artefacts layer on top of each other in a predictable stack:

  Web API / Dataverse
      ↓  data
  Forms & Views ← render individual records and lists
      ↓  grouped into
  Sitemap ← navigation tree (areas/groups/subareas)
      ↓  packaged as
  Model-driven App ← what the user logs into
  Ribbon / Web Resources → command-bar buttons & scripts
✓
Auto-provisioned: every time you create a custom table with create-entity, D365 auto-creates an 'Information' main form, an Active <Table> default public view, and an Inactive <Table> view. M07 teaches you to inspect those defaults and add your own.

03 CLI coverage map

The crm CLI covers the most common operations for all five artefacts. A few tasks — particularly visual designers and complex rule editors — remain GUI-only.

ArtefactCLI groupKey verbsGUI-only tasks
Form crm form list, add-field, remove-field, set-field, clone, export Form designer (drag-and-drop layout), form event scripting (OnLoad/OnSave) beyond field presence
View crm view list, create Advanced filter builder (complex OR trees in the GUI), column resize persistence per user
Sitemap (via crm app build-sitemap) build-sitemap Sitemap Editor canvas in Power Apps maker portal
App crm app create, build-sitemap, add-components, delete App validation/publishing UI, icon uploads (beyond webresource)
Ribbon crm ribbon add-button, list, export, remove Classic Ribbon Workbench (visual editor for complex multi-rule buttons)
Web resource crm webresource create, list, get, update, delete Power Apps web resource editor UI
!
Ribbon add-button uses an export-modify-import cycle under the hood (export solution ZIP → patch RibbonDiffXml → import ZIP back). This is why it's async and shows progress lines. You'll see this in M07 L04.

04 Check yourself

Three questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · A user navigates to the 'Tickets' area of a model-driven app and sees a list of rows. Which artefact defines the columns displayed in that list?
Correct. The view (saved query, querytype=0 public) is the artefact that defines the column set, filter, and sort order for a table's list display.
Not quite. The list-column layout is controlled by the view (savedquery). The form controls the single-record layout; the sitemap controls navigation; the ribbon controls command-bar buttons.
2 · You call crm --json metadata create-entity --display 'Ticket' --solution itworx_dev. Which UI artefacts does D365 auto-provision for you?
Correct. The platform auto-provisions an 'Information' main form plus 'Active <Table>' (default public view) and 'Inactive <Table>' views. You'll see this with crm view list ag_ticket in L02.
The platform auto-provisions two things: an 'Information' main form and two public views ('Active Tickets' and 'Inactive Tickets'). The app, sitemap, ribbon, and web resources are not auto-created.
3 · You want to add a JavaScript 'Mark as Reviewed' button to the Ticket form's command bar — entirely from the CLI. Which TWO steps are required (in order)?
Correct. You upload the JS file as a web resource first (webresource create), then reference it in ribbon add-button --webresource ... --function .... The ribbon command calls the JS function at runtime.
A ribbon button that runs JavaScript requires a web resource for the script. The two steps are: upload the JS file via webresource create, then ribbon add-button referencing it.
