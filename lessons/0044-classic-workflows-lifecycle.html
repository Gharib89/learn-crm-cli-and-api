Classic workflows:
lifecycle

List, activate, deactivate, export, import, and trigger on-demand classic workflows via the crm CLI.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads writes
Why this matters: Classic workflows are the most common automation artefact in legacy D365 orgs. Before they migrate to Power Automate, you need to inventory them, activate/deactivate them for maintenance windows, export them for ALM, and trigger on-demand ones programmatically. The CLI handles all of this without opening the Power Apps maker.

A classic workflow is a category-0 process stored in the workflow entity. Its statecode tells you whether it fires; ondemand tells you whether you can trigger it on a specific record. Five CLI verbs cover the full operational lifecycle — plus migration-assess for the cloud migration path.

01 workflow list — fields decoded

Start every workflow operation by listing what exists. Add --category workflow to filter to category 0 only.

shellcrm --json workflow list --category workflow

Each record in the returned array carries these fields:

FieldTypeWhat it tells you
workflowidGUIDPrimary key — used for activate/deactivate/run/export
namestringDisplay name of the workflow
categoryintAlways 0 for classic workflows
statecode0 or 10 = Draft (inactive), 1 = Activated
statuscode1 or 21 = Draft, 2 = Activated (mirrors statecode)
ondemandbooltrue = can be triggered on a record via workflow run; false = fires only on record events
primaryentitystringLogical name of the entity this workflow is scoped to

statecode / statuscode reference

statecodestatuscodeMeaning
01Draft (inactive) — will not fire on record events
12Activated — fires normally
02Activating — transitional state set briefly by the platform during activation

Useful filters

shell# Filter by entity
crm --json workflow list --category workflow --entity contact

# Return only activated workflows
crm --json workflow list --category workflow --activated

02 Activate and deactivate

Toggle activation state without touching the Power Apps maker. This is the maintenance-window pattern: deactivate before a schema change, reactivate after.

Activate

shellcrm --json workflow activate 5de74dca-2211-4433-8866-99aabbccdd01

TIP
Activation-record GUIDs. When you export and import a workflow, the imported record gets a new workflowid but the underlying definition GUID (type=2) stays the same. workflow activate resolves this automatically — pass either GUID and it will find the parent definition.

03 Export and import (ALM)

workflow export extracts the full workflow definition including its XAML. This is the starting point for the ALM pattern: export from dev, commit to source control, import to staging/prod.

Export

shellcrm --json workflow export 79fcb86b-954b-4bd4-8c14-8d5fbb78b84e -o /tmp/reset-stamp.json
envelope · live-verified 2026-06-18{
  "ok": true,
  "data": {
    "workflow_id": "79fcb86b-954b-4bd4-8c14-8d5fbb78b...",
    "out_path": "/tmp/reset-stamp.json",
    "record": { /* full workflow definition including XAML */ }
  }
}

The record object contains all workflow fields including the xaml property — the serialised XAML process definition. Omit -o to get the definition inline in the JSON envelope (useful for piping or inspection).

Import

shell# Import (upsert) workflow definition from file
crm --json workflow import --file /tmp/reset-stamp.json

# Import and activate in one step
crm --json workflow import --file /tmp/reset-stamp.json --activate

workflow import upserts the definition — if a workflow with the same definition GUID already exists it is updated; otherwise a new record is created. The --activate flag activates it immediately after import, saving a second CLI call.

TIP
ALM pattern. Export from dev → git commit the JSON → CI pipeline runs workflow import --file … --activate on staging and prod. No Power Apps maker access required on downstream environments.

04 workflow run — trigger on-demand writes

workflow run POSTs the ExecuteWorkflow OData action against a specific record. Two conditions must be true before this works:

  • The workflow must have ondemand: true
  • The workflow must be activated (statecode: 1)

Auto-trigger workflows (ondemand: false) fire on record events — they cannot be triggered via workflow run.

Dry-run first

shell · dry-runcrm --dry-run --json workflow run 79fcb86b-954b-4bd4-8c14-8d5fbb78b84e \

FlagPurpose

05 migration-assess — inventory for cloud migration

workflow migration-assess reads all category-0 workflows and flags them against Microsoft's Power Automate migration capability matrix. Each workflow gets a verdict and a list of blockers.

Blocker types

BlockerMeaning
custom_activityWorkflow uses a custom workflow activity (.NET class) — no equivalent in Power Automate cloud flows
real_timeWorkflow runs synchronously (real-time mode) — cloud flows are async-only
wait_conditionWorkflow contains a wait step or timeout — needs redesign for cloud flows (use delay/schedule actions instead)
 
shellcrm --json workflow migration-assess --entity contact
live output · agent-cloud[
  {
    "id": "79fcb86b-954b-4bd4-8c14-8d5fbb78b...",
    "name": "Reset Security Stamp",
    "primaryentity": "contact",
    "state": "activated",
    "mode": "background",
    "verdict": "blocked",
    "blockers": ["custom_activity"]
  }
]

Verdict "blocked" means redesign required — not impossible. "Reset Security Stamp" uses a custom workflow activity (.NET), so a direct migration to Power Automate is not available. The team must either rewrite the logic as a Power Automate HTTP action or a plug-in, or keep the classic workflow.

Omit --entity to assess all workflows across all entities:

shellcrm --json workflow migration-assess
WARN
Plan before migrating. A "blocked" verdict is not a red card — it is a work-item. Run migration-assess early on a project to size the migration backlog, not as a last step before cutover.

Lesson check

4 questions · click to reveal

1. workflow list returns a workflow with statecode: 0, statuscode: 1. What does this mean?

Correct. statecode: 0 / statuscode: 1 = Draft (inactive). The workflow exists but will not fire until activated with workflow activate <id>.
statecode: 0 / statuscode: 1 = Draft (inactive). Activated is statecode: 1 / statuscode: 2. The transitional "activating" state is statecode: 0 / statuscode: 2 — set briefly by the platform during activation.

2. Which flag filters workflow list to show only activated workflows?

Correct. --activated filters to statecode: 1 records only. Combine with --entity <logicalname> to get all active workflows on a specific table.
The correct flag is --activated. There is no --active, --statecode, or --status flag on workflow list.

3. You want to run the "Reset Security Stamp" workflow against a contact via the CLI. What must be true?

Correct. Both conditions are required: ondemand: true (the workflow is designed to be manually triggered) and statecode: 1 (activated). A draft on-demand workflow will be rejected.
Two conditions: ondemand: true and statecode: 1. Category 5 is Power Automate cloud flows — workflow run does not apply. The owner and --bypass-plugins have no bearing on whether the workflow can be triggered.

4. What does migration-assess verdict "blocked" mean?

Correct. "Blocked" means the workflow uses features that have no direct equivalent in Power Automate cloud flows — custom activities, real-time synchronous execution, or wait conditions. Each blocker is a redesign work-item, not a permanent blocker.
"Blocked" = blockers found that require redesign — not a statement of impossibility. The activation state is separate from migration readiness. migration-assess only targets category 0 (classic workflows), not business rules.
