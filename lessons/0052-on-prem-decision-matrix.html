The M09 capstone. When a command fails on on-prem, sort it fast: VPN, auth, the version ceiling, a cloud-only feature, a missing privilege — or a genuine tool defect worth filing upstream.

live on agent-on-prem  ·  crm.contoso.local/Contoso  ·  reads capstone
Why this matters: The mission has two on-prem goals — handle the NTLM/v9.1 split correctly, and recognize when a failure is a tool defect (not user error) so you can file a high-quality issue upstream to Gharib89/crm. The first four lessons gave you the divergences. This one turns them into a triage you can run in seconds — the difference between "I know exactly which layer broke" and "let me retry and hope."

On cloud, most failures are your request. On on-prem, a failure could be the network, the build's age, or the auth realm — none of which a retry fixes. Good triage starts by asking which layer, not what did I type wrong.

01 The five on-prem failure classes

Almost every on-prem-specific failure falls into one of five buckets. Each has a distinct tell — usually in meta.category plus the error string — and a distinct fix:

ClassTellCause & fix
Network doctor dns_tcp fails; transport_error; "No route to host" VPN down or host unreachable. Connect the VPN; re-run doctor.
Version 501; "Requested API Version 'v9.2' is not available" Profile pinned above the on-prem ceiling. Re-negotiate to v9.1 (profile add without --api-version). L02
Feature "message/operation not found"; a verb that works on cloud errors on-prem A cloud-only capability (e.g. CreateMultiple) absent from the older build. Use the on-prem equivalent ($batch) or accept it's not available. L02 §04
Auth 401; auth_failed Wrong NTLM domain / username / password. Re-run profile set-password; check the domain prefix.
Privilege 403; forbidden The user authenticated but lacks the privilege/role for this operation. A security-role fix, not a connection fix.
KEY
Four of these five are environment problems, not request problems — the command and payload were fine. That's the on-prem mindset shift: before you re-read your own JSON, ask whether the connection, build, or realm is the real story. meta.category usually tells you in one field.

02 The triage, in order

Run it top to bottom — the first match is your answer:

  1. Run connection doctor first. It classifies network / TLS / version / auth in one call (L03). If a check fails, you have your class — stop here.
  2. Read meta.category + the error string (not just the HTTP status). transport_error → network. 501 + "API Version" → version. auth_failed → auth. forbidden → privilege.
  3. Ask "does this work on cloud with the same inputs?" If yes and it fails only on on-prem, suspect a feature gap (cloud-only message) before suspecting a bug.
  4. Only then consider a tool defect — when none of the above fit and the behavior is wrong, not just different (§03).
TRAP
Don't trust meta.retryable in isolation. You saw in L02 that a 501-for-version carries retryable: true — generic 5xx classification — yet retrying never helps. The error string is the deciding signal when the category is ambiguous.

03 Tool defect vs user error — and filing it well

The mission explicitly wants you to catch genuine crm defects and report them. Use this line:

It's probably not a CLI bug if…It may be a CLI bug if…
The error string names a server condition (version, auth, privilege, "not found") The CLI raises an internal traceback / unhandled exception, not a clean envelope
The same command + inputs fail the same way the server would by hand The CLI builds a wrong request for on-prem (visible in --dry-run: bad URL/version/body)
It's a documented cloud-only feature A command that should be version-agnostic behaves differently on v9.1 for no documented reason
A retryable class actually recovers on retry The envelope contract is violated (missing ok/meta, wrong category for the status)

When it is a defect, a high-quality issue to Gharib89/crm includes the reproducible facts — never the real org URL or host (scrub to the example env):

a good upstream issue carries…# 1 · the exact version
crm --version                         # → crm, version 1.0.0

# 2 · the command (scrubbed) + the FULL --json envelope it returned
crm --json --profile <on-prem> <command…>
#   → {"ok": false, "error": "...", "meta": {"status":…, "category":…}}

# 3 · what doctor says (which layer is healthy)
crm --json connection doctor

# 4 · expected vs actual, and whether cloud behaves differently
SCRUB
Before pasting anything upstream, replace the real host/org with the example env (http://crm.contoso.local/Contoso, https://orgexample.crm.dynamics.com) and mask GUIDs. The repro must be faithful in shape, not in secrets. The agent can offer to file it — but never files silently.

Lesson check

4 questions · click to reveal

1. A bulk write fails on on-prem with "message not found", but the identical command succeeds on cloud. Which failure class is this, and what's the fix?

Correct. "Works on cloud, 'message not found' on on-prem" is the feature-gap signature: a newer Dataverse message (like CreateMultiple) that the older on-prem build doesn't have. The fix is the on-prem equivalent — $batch — not a role change, a re-negotiation, or a bug report.
This is a feature gap: a cloud-only message absent from the older on-prem build (the "works on cloud, not on-prem" tell). Use the on-prem mechanism ($batch). It's not auth, not the version ceiling per se, and not a CLI defect.

2. What should you run first when any on-prem command fails unexpectedly?

Correct. connection doctor walks the whole path and tells you which layer broke — so you fix the right one. Blind retries, profile rebuilds, and bug reports all come after you've localized the failure, not before.
Run connection doctor first — it localizes the failure to a layer (network/TLS/version/auth) so you don't fix the wrong thing. Retrying, rebuilding, or filing a bug are premature until you know which layer broke.

3. Which of these is the strongest signal that you've hit a genuine crm tool defect (not a user/environment error)?

Correct. A wrong request the CLI itself composed — visible in --dry-run — or a crash that breaks the {ok,meta} envelope contract points at the tool. The other three are well-formed server responses to real environment conditions (privilege, version ceiling, throttling) — expected behavior, not bugs.
The defect signal is the CLI's own behavior being wrong: a malformed request it built (seen in --dry-run) or a traceback that violates the envelope contract. 403/501/429 are correct server responses to real conditions — not bugs.

4. A teammate says "on-prem needs a different set of crm commands than cloud." Based on M09, what's the accurate correction?

Correct. The entire milestone proved the commands, flags, and envelope are the same on both targets. What differs is the auth scheme (NTLM vs OAuth), the API version (v9.1 vs v9.2), and a few cloud-only platform features — and the profile carries the auth and version for you.
There's no separate on-prem command set and no SOAP. The verbs, flags, queries, writes, and envelope are identical; only auth (NTLM/OAuth), version (v9.1/v9.2), and a few cloud-only features differ — and the profile supplies auth + version.
