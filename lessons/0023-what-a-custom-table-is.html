What a custom
table really is

Before you create one, build the picture of what Dataverse actually makes — and why a prefix, a solution, and a publish step are all part of it.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. M03 taught you to read the schema — three names, typed columns, option sets, relationships, the describe brief. M04 flips it: you start building that schema. But a custom table isn't a config row you toggle — it's a real backing store plus a metadata definition plus a pile of auto-generated system columns, all stamped with a publisher and tracked in a solution. Get that model right first and every write verb in this milestone reads as obvious.

Today's win: explain — to a colleague — exactly what gets created when you make a custom table, and preview one with zero writes using --dry-run.

01 A table is more than the columns you name

When you create a custom table called "Project", you type one display name — but Dataverse provisions a whole apparatus behind it. You already met most of these pieces in M03 as things you read; here's what gets written for you automatically:create-table

What's createdExample (Project)You met it in
The three namesitworx_project · Itworx_Project · itworx_projectsM03 L01
Primary id (GUID)itworx_projectid — SystemRequired, autoM03 L02
Primary name columnitworx_name — String, ApplicationRequired, autoM03 L02 / L05
System columnsownerid, createdon, modifiedon, statecode/statuscode, …M02 L01
Privileges & entity setcreate/read/write/delete roles + the /itworx_projects Web API pathM01 L06

So your job on create is small: pick a display name, a primary-name label, and an ownership model. Dataverse fills in the GUID key, the system columns, the privileges, and the entity set. The custom columns you actually care about (budget, start date, status) you add after — that's L02 onward.

✓
Ownership is a create-time, near-permanent choice. UserOwned (rows have an ownerid, can be assigned + secured per-user — the default) vs OrganizationOwned (org-wide, no owner). Pick UserOwned unless the data is truly org-global reference data. You can't flip it later without recreating the table.

02 The publisher prefix & the solution

Every custom thing you create is stamped with your publisher's customization prefix. That's why the table above is itworx_project, not just project — the prefix namespaces your customizations away from Microsoft's and from other vendors'.publisher The default schema name is simply <Prefix>_<Display>:

display
Project → the UI label a user sees
prefix
itworx → comes from the publisher bound to your solution
schema_name
itworx_Project → derived, PascalCase (spaces dropped: "Unit Price" → itworx_UnitPrice)

And every customization lands inside a solution — the deployable container that tracks what you changed, so it can be exported and moved dev → test → prod (that's M06's whole topic). Your profile carries both defaults, so you rarely type them:

agent-cloud · exit 0crm --json profile list

{ "name": "agent-cloud", "active": true,
  "publisher_prefix": "itworx",          # stamps every itworx_* schema name
  "default_solution": "itworx_dev" }     # where new components are tracked
!
The prefix must match the solution's publisher. If your profile prefix doesn't match a real publisher on the org, the create fails with "solution unique name … is not valid" (a 404). Prefix and solution travel together — fix them with crm profile edit <name> --publisher-prefix … --default-solution ….

03 Preview with zero writes: --dry-run

Your safest first contact with the write surface is the same global flag you met in M01 L04 — --dry-run. On scaffold table it reports the entity and every column it would create, makes no create calls, and fires only one existence-check GET:scaffold

agent-cloud · exit 0 · dry-runcrm --json --dry-run scaffold table "Project" \
    --column "Code:string:max_length=20,required=ApplicationRequired" \
    --column "Budget:money" \
    --column "Active:boolean"

{ "ok": true, "data": {
    "applied": [], "skipped": [], "failed": [],
    "planned": [                              # nothing created — just the plan
      { "kind": "entity",    "name": "itworx_Project" },
      { "kind": "attribute", "name": "itworx_Code" },
      { "kind": "attribute", "name": "itworx_Budget" },
      { "kind": "attribute", "name": "itworx_Active" }
    ] }, "meta": { "dry_run": true } }

Read the planned list and you can already see the prefix doing its work — every name comes back itworx_*, derived from your display names. The {applied, skipped, planned, failed} shape is the standard report for all schema-building verbs (scaffold and apply); you'll see it again every lesson. "No writes" — but as M01 L04 warned, not "no traffic": one GET still goes out to check whether the table already exists.

04 The last step is always publish

A metadata change isn't live the instant the API accepts it. Dataverse keeps an unpublished staging layer; your new table and columns only reach users' forms and the app once PublishAllXml runs.publish The crm verbs handle this for you:

FlagWhat happensWhen to use
--publish (default)Runs PublishAllXml after the changeSingle change, want it live now
--no-publishStages the change, skips publishMany imperative changes — publish once at the end
--stage-only (scaffold)Builds everything, suppresses the final publish → meta.staged=trueBatch build you'll publish/review separately

Imperative verbs publish per call by default (one create = one publish), which is slow if you're adding ten columns. The batch builders (scaffold, apply) publish once at the end. That single difference is a big reason the batch tools exist — and a thread you'll pull through L02–L04.

05 Do it yourself ~5 min

All read-only or dry-run — nothing lands on the org:

# 1 · confirm your prefix + default solution
crm --json profile list

# 2 · preview a custom table — see the prefix derive every name, no writes
crm --json --dry-run scaffold table "Project" \
    --column "Code:string" --column "Budget:money" --column "Active:boolean"

# 3 · look at a stock table's auto system columns (the ones you'd inherit)
crm --json metadata describe account
  1. In query 1, note your publisher_prefix. Predict the schema_name for a table you'd display as "Service Visit". (Spaces drop, PascalCase.)
  2. In query 2's planned list, every name is <prefix>_*. Which two components did Dataverse not ask you to name, yet will create anyway? (Hint: the GUID key and the primary-name column.)
  3. In query 3, scan writable_attributes for columns you never defined — ownerid, dates, state. Those are the system columns every table inherits.
  4. One-sentence answer, out loud: why does a brand-new table already have a primary id and a name column before you add a single field?

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You run scaffold table "Project" with three columns. Beyond those three, what does Dataverse create on its own?
Correct. A custom table is a whole apparatus: the primary id GUID and primary-name column are auto, plus the standard system columns, privileges, and the Web API entity set. You named 3; you got far more.
A table is more than its named columns. Dataverse auto-provisions the primary id (GUID), the primary-name column, system columns (owner/dates/state), privileges, and the entity set — all without you asking.
2 · Your display name is "Project" and your publisher prefix is itworx. What's the default schema name?
Correct. Schema name = <Prefix>_<Display> = itworx_Project (PascalCase, prefix first). The lowercase itworx_project is its logical name; itworx_projects is the entity set.
It's <Prefix>_<Display> → itworx_Project. (itworx_project is the logical name; itworx_projects the entity set — same table, three names, from M03 L01.)
3 · Why must the customization happen inside a solution whose publisher matches your prefix?
Correct. The solution records what you changed so it can move dev→test→prod (M06), and its publisher's prefix stamps the component. Mismatch and the create 404s with "solution unique name not valid".
The solution is the tracked, deployable container, and its publisher supplies your prefix. They travel together — a prefix that matches no real publisher/solution makes create fail with a 404.
4 · The API accepted your create-entity call. Is the table live for users yet?
Correct. Metadata changes sit in an unpublished layer until publish. The crm verbs publish by default; batch builders publish once at the end, and you can stage with --no-publish/--stage-only.
Acceptance ≠ published. Changes stage in an unpublished layer until PublishAllXml runs. crm publishes by default; defer it with --no-publish (per-verb) or --stage-only (scaffold).
