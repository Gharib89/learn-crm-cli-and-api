The WhoAmI
Handshake

One read-only command proves you're talking to a live Dynamics org — and tells you who and where you are.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. In L01 you configured a profile — but a saved profile is just config on disk. Before any customization you must confirm, from the live server, which org actually answers and that auth works. Hitting the wrong org is the costliest beginner mistake; this lesson builds the reflex that prevents it.

Today's win: run one command and read three facts back from your live cloud org — proving the CLI is connected, and showing the identity it connected as.

01 What are you connecting to?

Dynamics 365 Customer Engagement doesn't run on your laptop. It runs as an environment (also called an org) — an isolated database plus its own web address. Your agent-cloud org lives at https://orgexample.crm.dynamics.com.

Every org exposes one programmatic door: the Dataverse Web API, an HTTP service at a fixed path. The crm CLI is a thin wrapper over that API — it doesn't click the GUI and it doesn't run local SQL. Each command is one HTTPS request to:

https://orgexample.crm.dynamics.com/api/data/v9.2/
└──────────── one org ───────────┘└── Web API ──┘ the environment / host versioned endpoint

That version segment matters for the whole mission: cloud (OAuth) tops out at v9.2; on-prem (NTLM) caps at v9.1 — ask for v9.2 on-prem and the server returns HTTP 501.skill You'll meet that split for real in M09; for now, just notice the version sits right there in the URL.

02 WhoAmI — the identity handshake

WhoAmI is the Web API's "are you there, and who am I?" function. It takes no input and returns three IDs identifying the calling session.webapi It's the canonical first call: cheap, read-only, and it fails loudly when auth or the URL is wrong. The CLI exposes it as:

# --json = the stable, machine-readable contract — always use it
crm --json connection whoami

● your live result   here's what your agent-cloud org returned moments ago:

agent-cloud · exit 0{
  "ok": true,
  "data": {
    "@odata.context": "https://orgexample.crm.dynamics.com/api/data/v9.2/$metadata#...WhoAmIResponse",
    "BusinessUnitId": "4a617aa7-5a3d-f111-88b5-000d3a3b279b",
    "UserId":         "bf40e4d2-ee3e-f111-88b4-6045bd001b10",
    "OrganizationId": "9102ced1-5c3e-f111-b31d-000d3a302819"
  }
}

Read it as the three facts of "where am I, as whom":

@odata.context
The URL the request actually hit. Your proof of which org + which API version (note the live v9.2). Check this before any write.skill
OrganizationId
GUID of the org (the environment) itself.
UserId
GUID of the user the CLI authenticated as — your application user.
BusinessUnitId
GUID of that user's business unit — the security / ownership container it sits in.

A GUID (e.g. bf40e4d2-ee3e-f111-…) is Dataverse's universal primary key — every record, table, and org is identified by one. You'll copy-paste these constantly, so get comfortable now.

✓
The whole win: if "ok": true comes back and the @odata.context host matches the org you intended, you're connected and pointed at the right place. That two-second check is the seatbelt you'll fasten before every mutation later in the mission.

03 Do it yourself ~2 min

Open your terminal in this workspace and run, in order:

# 1 · which profile is active? the ● marks it
crm profile list

# 2 · the handshake against the active org
crm --json connection whoami

Then confirm you can answer these from your own output before moving on:

  1. Which profile shows the ● dot? Does its URL end in .dynamics.com (cloud) — your intended target?
  2. In the whoami output, what API version appears inside @odata.context?
  3. Could you point to your OrganizationId from memory of where it sits in the JSON?
!
Read-only & safe. whoami and profile list issue no writes — run them as often as you like. (The --dry-run safety net for writes comes in a later lesson.)

04 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You're about to create a custom table and want to be certain the CLI is pointed at your cloud dev org, not on-prem. Which single field in the whoami output proves which host + which API version you're hitting?
Correct. @odata.context is a real URL echoed back by the server — host and /api/data/v9.x/ version. The skill's rule: confirm it before any mutation. (OrganizationId also names the org, but it's an opaque GUID — no host or version at a glance.)
Not quite. That field is a GUID for the user / BU / org — useful, but it won't show you the host or API version. The field that echoes the actual URL hit is @odata.context.
2 · crm entity get accounts <guid> returns a record. Under the hood, what did the CLI actually do?
Correct. crm is a thin wrapper over the Dataverse Web API — every command is one HTTPS call to /api/data/v9.x/. No local DB, no GUI, no SOAP.
No. There's no local database, no GUI automation, and (in this CLI) no SOAP. Every crm command is a single HTTPS request to the Web API at /api/data/v9.x/.
3 · Your whoami shows @odata.context: …/api/data/v9.2/… What does that version tell you about which org you're on?
Correct. Cloud (OAuth) serves v9.2; on-prem (NTLM) maxes at v9.1 and returns HTTP 501 if you ask for v9.2. So v9.2 in the context = the cloud org. You'll use this tell constantly.
Re-check §1: on-prem caps at v9.1, cloud serves v9.2. Seeing v9.2 means you're on the cloud org.
4 · A teammate says "just trust the active profile and start writing." Why is running whoami first still the better habit?
Correct. whoami is read-only and cheap, yet it proves both that auth succeeds and exactly which org answered. The active profile is just config; whoami is confirmation from the live server — and that gap is where wrong-org accidents hide.
Not the reason. whoami isn't mandatory, it locks nothing, and it doesn't cache. Its value: it cheaply confirms — read-only — both that auth works and which live org actually responded, before a write can reach the wrong place.
