Build it by hand:
create & add

Make the table for real with one verb, then grow it a column at a time — meeting the full typed-column palette as you go.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. L01 gave you the model and a no-write preview. The imperative verbs — metadata create-entity then metadata add-attribute — are the most explicit way to write it: one call per thing, each returning a clean envelope you can read. Learn the palette here and the batch tools (L03–L04) become "the same thing, faster."

Today's win: create a real custom table on agent-cloud, add three typed columns, confirm with describe — then delete it, leaving the org clean.

01 create-entity — the table itself

One call makes the table. You name the display label; the prefix, the GUID key, and the primary-name column are filled in for you (L01):create-entity

agent-cloud · exit 0crm --json metadata create-entity --display "Project" \
    --description "Customer delivery projects"

{ "ok": true, "data": {
    "created": true,
    "schema_name":      "itworx_Project",
    "logical_name":     "itworx_project",
    "entity_set_name":  "itworx_projects",
    "primary_attribute": "itworx_name",        # auto-created String, ApplicationRequired
    "solution": "itworx_dev", "published": true } }

The envelope hands back all three names at once — and confirms itworx_name was created for you as the primary-name column. published: true means this call already ran PublishAllXml. Defaults worth knowing:

FlagDefaultNote
--schema-name<prefix>_<Display>Override to control casing/wording
--ownershipUserOwnedOr OrganizationOwned (L01)
--primary-attr / --primary-label<prefix>_Name / "Name"Rename the primary-name column
--if-existserrorskip = no-op success if it exists

02 add-attribute — the typed-column palette

Columns are added one verb at a time to an existing table. The --kind picks the type; the type decides which other flags are required. The full palette:add-attribute

KindKey flagsStores
string / memo--max-length, --format Text|Email|Url|Phone|TextAreaShort text / long text
integer / bigint--min --maxWhole numbers
decimal / double / money--precision (required), --min --maxFractional / currency
boolean--true-label --false-labelTwo-option yes/no
datetime--format DateOnly|DateAndTimeDates / timestamps
picklist / multiselect--option value:label (local) or --optionset-name (global)Choice(s) from a set
lookup--target-entityA link to another table → M05
image / file--max-size-kbBinary attachment

Three columns for our Project — a required code, a currency amount, a yes/no — each its own call:

agent-cloud · exit 0# string — required, capped at 20 chars
crm --json metadata add-attribute itworx_project --kind string \
    --display "Code" --max-length 20 --required ApplicationRequired

# money — precision is mandatory for decimal/double/money
crm --json metadata add-attribute itworx_project --kind money \
    --display "Budget" --precision 2

# boolean — custom true/false labels
crm --json metadata add-attribute itworx_project --kind boolean \
    --display "Active" --true-label "Yes" --false-label "No"

# each returns ↓
{ "ok": true, "data": { "created": true, "entity": "itworx_project",
    "schema_name": "itworx_Code", "logical_name": "itworx_code",
    "attribute_type": "String", "solution": "itworx_dev", "published": true } }
!
Two traps the live CLI enforces. (1) decimal, double and money require --precision — omit it and you get "--precision is required for this kind." (2) A local picklist's --option must be value:label or :label (auto-value) — a bare --option "New" is rejected with "--option must be 'value:label' or ':label'".

A local picklist, built inline — values auto-assign from your publisher's option-value base:

agent-cloud · exit 0crm --json metadata add-attribute itworx_project --kind picklist \
    --display "Stage" --option ":Planning" --option ":Active" --option ":Closed"
# → Planning=100000000, Active=100000001, Closed=100000002 (a LOCAL option set, this table only)

03 Each call publishes — unless you say otherwise

By default every add-attribute runs its own PublishAllXml. Add ten columns and you publish ten times — correct, but slow. When you're building a table imperatively, stage with --no-publish and publish once at the end:

agent-cloud · exit 0crm --json metadata add-attribute itworx_project --kind money \
    --display "Budget" --precision 2 --no-publish
# note: no "published" key in the envelope when staged
{ "ok": true, "data": { "created": true, "attribute_type": "Money" } }
✓
This is exactly the pain the batch builders remove. Staging by hand and remembering to publish is bookkeeping. scaffold table (L03) and apply -f (L04) do all the columns in dependency order with a single publish — the imperative path is what they automate.

04 Confirm it: describe the result

You learned describe as M03's capstone — now use it to verify your own write. The custom columns sit alongside the auto-created ones:

agent-cloud · exit 0crm --json metadata describe itworx_project

{ "logical_name": "itworx_project", "primary_id": "itworx_projectid",
  "primary_name": "itworx_name", "writable_attributes": [     # custom + inherited system cols
    {"logical_name":"itworx_code",   "attribute_type":"String",  "required_level":"ApplicationRequired"},
    {"logical_name":"itworx_budget", "attribute_type":"Money",   "required_level":"None"},
    {"logical_name":"itworx_active", "attribute_type":"Boolean", "required_level":"None"},
    {"logical_name":"itworx_stage",  "attribute_type":"Picklist",
       "options":[{"value":100000000,"label":"Planning"}, …]},
    {"logical_name":"itworx_name",   "attribute_type":"String",  "required_level":"ApplicationRequired"},
    {"logical_name":"ownerid"}, {"logical_name":"itworx_projectid"}, # … the system columns you inherited
  ] }

Your four custom columns are there with the exact types and required levels you set — and so are itworx_name, ownerid, the GUID id, and the rest of the system set. That is the table you reasoned about in L01, now real.

05 Do it yourself ~8 min writes

This one creates real schema on agent-cloud — then deletes it, so the org ends clean. (Custom-table delete is reversible-by-recreation; you'll formalise cleanup in L05.)

# 1 · create the table
crm --json metadata create-entity --display "Project"

# 2 · add three typed columns (stage them, publish on the last)
crm --json metadata add-attribute itworx_project --kind string  --display "Code"   --max-length 20 --no-publish
crm --json metadata add-attribute itworx_project --kind money   --display "Budget" --precision 2 --no-publish
crm --json metadata add-attribute itworx_project --kind boolean --display "Active"
#   (the last call has no --no-publish, so it publishes all three)

# 3 · confirm
crm --json metadata describe itworx_project

# 4 · clean up
crm --json metadata delete-entity itworx_project --yes
  1. Run step 2's money line without --precision. Read the exact error. Then add --precision 2 and watch it succeed.
  2. In step 3, find your three columns. What required_level did Code get, and why? Find itworx_name — who created it?
  3. Count the columns in writable_attributes you did not define. Those are the inherited system columns from L01.
  4. Try a local picklist: add a --kind picklist --display "Stage" column with three :label options, then describe again and read the auto-assigned values.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You run create-entity --display "Project" and nothing else. Which column can you immediately set on a new row?
Correct. create-entity auto-creates the primary-name column (itworx_name, String, ApplicationRequired) — the envelope reports it as primary_attribute. It exists before you add anything.
The envelope's primary_attribute is itworx_name — the primary-name column Dataverse creates automatically. It's settable from the moment the table exists.
2 · add-attribute --kind money --display "Budget" fails. Most likely fix?
Correct. Fractional/currency kinds need a precision (decimals). The CLI rejects them with "--precision is required for this kind." until you supply --precision N.
Money, decimal and double all require --precision (number of decimals). The exact error is "--precision is required for this kind."
3 · You add 6 columns, each a separate add-attribute with defaults. How many publishes ran?
Correct. Imperative verbs publish per call. Six calls = six publishes. Stage the first five with --no-publish and let the sixth publish them all at once — or use a batch builder.
Each imperative call publishes by default → six publishes. To publish once, pass --no-publish on all but the last (or use scaffold/apply, which publish once).
4 · A local picklist --option. Which form does the CLI accept?
Correct. The form is value:label or :label (value auto-assigned from the publisher's option base). A bare label is rejected: "--option must be 'value:label' or ':label'".
It's value:label or :label (auto value). :New works; New alone doesn't — the value comes first, label second.
