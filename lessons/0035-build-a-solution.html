From scratch: publisher → solution → table

Create the full stack in four commands, then wire the component in.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  writes
Why this matters: You've been using agsol (or itworx_dev) as a ready-made bucket. Now you'll build one from first principles — publisher, solution, table — and explicitly register the table as a solution component. This is the exact sequence an agent runs when setting up a new customization context on a fresh org.

Four commands. Publisher first — it's the source of the prefix every component inherits. Then the solution container. Then the table. Then add-component to register it.

01 Create the publisher

A publisher owns a customization prefix (e.g. itworx_) and an option-value prefix that namespaces picklist integers. Every component you create inherits the publisher's prefix — so the publisher must exist before the solution.

FlagPurpose
--nameUnique name (internal key). Lowercase, no spaces.
--displayHuman-readable friendly name shown in the UI.
--prefixSchema prefix (2–8 chars, alphanumeric, starts with a letter, not mscrm). Columns, tables, and option sets get this prefix.
--option-value-prefixInteger 10000–99999. Offsets picklist option values to avoid cross-publisher collisions.
--if-existsskip or error (default). Use skip for idempotent scripts.
--no-set-defaultDon't overwrite publisher_prefix in the active CLI profile.
 
shellcrm --json solution create-publisher \
    --name itworx \
    --display "ITWorx Publisher" \
    --prefix itworx \
    --option-value-prefix 72000 \
    --if-exists skip \
    --no-set-default

TIP
If your profile already has a publisher (like itworx), skip create-publisher and pass --publisher itworx to solution create. The publisher only needs to exist once per org.

02 Create the solution container

The solution is the versioned container that travels between orgs. It references the publisher you just created.

FlagPurpose
--nameUnique name. Used as the ZIP filename base and import key.
--displayFriendly name shown in the Solutions UI.
--version4-part version string (major.minor.build.revision). Start at 1.0.0.0.
--publisherUnique name of the publisher to bind this solution to.
--if-existsskip or error.
--no-set-defaultDon't overwrite default_solution in your CLI profile. Important when creating a one-off solution without disrupting your working context.
 
shellcrm --json solution create \
    --name itworx_dev \
    --display "ITWorx Dev Solution" \
    --version 1.0.0.0 \
    --publisher itworx \
    --if-exists skip \
    --no-set-default

03 Create a table inside the solution

Pass --solution itworx_dev to metadata create-entity so Dataverse stamps the new entity with the solution context. The schema name automatically gets the publisher's prefix.

shellcrm --json metadata create-entity \
    --schema-name itworx_Project \
    --display "ITWorx Project" \
    --display-collection "ITWorx Projects" \
    --solution itworx_dev
envelope · live-verified 2026-06-18{

    "created": true,
    "schema_name": "itworx_Project",
    "logical_name": "itworx_project",
    "entity_set_name": "itworx_projects",
    "primary_attribute": "itworx_name",
    "metadata_id_url": "https://orgexample.crm.dynamics.com/api/data/v9.2/EntityDefinitions(246395a6-256b-f111-ab0c-70a8a5af9b7e)",
    "solution": "itworx_dev",
    "published": true
  }
}
TIP
The MetadataId lives in the URL. Look at metadata_id_url: the GUID at the end — 246395a6-256b-f111-ab0c-70a8a5af9b7e — is the entity's MetadataId. That's the value you pass to add-component --id in the next step.

04 Register the component with add-component

Creating the entity with --solution stamps it with the solution context, but explicitly calling add-component makes it a root component — the thing that appears in solution components output and travels in the ZIP.

FlagPurpose
--solutionTarget solution unique name.
--typeComponent type integer. 1 = Entity (table). See the component-type reference in L02.
--idThe component's MetadataId (for entities: extracted from metadata_id_url).
--no-add-requiredDon't ask the server to automatically pull in required dependent components.
--no-subcomponentsDon't include subcomponents (columns, forms, views) of the entity.
 
shellcrm --json solution add-component \
    --solution itworx_dev \
    --type 1 \
    --id 246395a6-256b-f111-ab0c-70a8a5af9b7e
envelope · live-verified 2026-06-18{
  "ok": true,
  "data": {
    "added": true,
    "solution": "itworx_dev",
    "component_id": "246395a6-256b-f111-ab0c-70a8a5af9b7e",
    "component_type": 1
  },
  "meta": {
    "note": "AddRequiredComponents was enabled: the server may have silently added required components beyond the requested entity; the response does not report them."
  }
}
TIP
The meta.note says "may have silently added required components." That's normal — when you add an Entity, the server also pulls in its forms, views, and ribbon. These show up in solution components output but aren't listed in the add-component response.

05 Verify with solution components

Run components to confirm the entity (and its silently-pulled dependents) are registered in the solution. This is the same command covered in L02.

shellcrm --json solution components itworx_dev

The output will list the root component (type 1 — the entity) along with the dependent components the server added: forms (type 60), views (type 26), ribbons (type 70), and possibly the primary column (type 9). Together they represent everything that travels in the solution ZIP.

TIP
The root component count you see here is what ends up in solution.xml as <RootComponent> entries. Everything non-root is packed into customizations.xml but doesn't appear in the manifest's component list.

Lesson check

3 questions · click to reveal

1. What does --option-value-prefix 72000 on create-publisher do?

Correct. The integer prefix namespaces all choice/picklist option values under this publisher — so two publishers' option integers never overlap in a merged org.
Not quite. The option-value prefix is about picklist integer namespacing — it offsets choice field values so publishers don't collide. Nothing to do with version numbers, sample data, or component limits.

2. You ran metadata create-entity --solution itworx_dev. The entity is created. Do you still need add-component?

Correct. --solution stamps the entity's metadata with the solution context, but explicit root component registration via add-component is what makes it appear in solution components and travel in the ZIP as a root entry.
Not quite. --solution routes creation but doesn't automatically call AddSolutionComponent. You need the separate add-component step to register it as a root component.

3. The add-component response includes a meta.note about "AddRequiredComponents". What did the server do?

Correct. When you add an Entity component, the server's AddRequiredComponents behavior silently pulls in everything the entity depends on — default forms, system views, and the ribbon XML. They appear in solution components even though the response only lists the one entity you added.
Not quite. The note is about dependent component pull-in: the server added forms, views, and ribbon automatically alongside the entity you requested. No backup, publish, or version bump happens.
