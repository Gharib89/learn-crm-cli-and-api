Option sets:
local vs global

A picklist's legal codes aren't yours to guess. Pull the real {value, label} pairs from the org — and learn which option sets live on one column and which are shared estate-wide.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. In M02 you set gendercode to 1 and trusted it meant "Male." That worked because the seed data hinted at it — but a real requirement ("set status to Escalated") gives you a label, and the API only accepts the code. The picklist metadata is the dictionary that translates one to the other. Read it and you never hard-code a magic number again.

Today's win: resolve a column's allowed options to code → label pairs, and tell a local option set from a reusable global one.

01 Resolve a column's options

L02 told you gendercode is a Picklist — but not its legal values. metadata picklist <logical> <column> returns the option set behind the column, with each option's stored code and human label:optionset

agent-cloud · exit 0crm --json metadata picklist contact gendercode

{ "ok": true, "data": {
    "LogicalName": "gendercode",
    "OptionSet": {
      "Name": "contact_gendercode",    # the option-set's own name
      "IsGlobal": false,                # ← LOCAL: belongs to this column
      "Options": [
        { "Value": 1, "Label": { .."Male".. } },
        { "Value": 2, "Label": { .."Female".. } }
      ] } } }

There's the answer M02 took on faith: 1 → Male, 2 → Female. The Label arrives as the same fat localized-label object you met in L01 (trimmed to the English string above). When you write the column, you send the code; the label is only ever for humans.

✓
Works for state/status too. The same verb resolves statecode and statuscode — the State and Status option sets every table has. That's how you'd turn "set this case to Resolved" into the right numeric statuscode.

02 Local vs global — the IsGlobal flag

Notice "IsGlobal": false above. That one boolean is the whole distinction:

Local option set
IsGlobal:false. Defined on a single column and used nowhere else. Its Name is usually <table>_<column> (contact_gendercode). Edit it and only that one column changes.
Global option set
IsGlobal:true. A standalone, named list that many columns across many tables can share. Defined once; reused everywhere. Edit it and every column bound to it changes at once.

Global sets have their own two verbs, independent of any column. list-optionsets enumerates them; get-optionset <name> retrieves one with its options:global

agent-cloud · exit 0crm --json metadata get-optionset componentstate

{ "ok": true, "data": {
    "Name": "componentstate", "IsGlobal": true,   # ← GLOBAL, reusable
    "Options": [
      { "Value": 0, "Label": { .."Published".. } },
      { "Value": 1, "Label": { .."Unpublished".. } },
      { "Value": 2, "Label": { .."Deleted".. } },
      { "Value": 3, "Label": { .."Deleted Unpublished".. } }
    ] } }

By default, picklist expands a bound global so you always get the options inline — pass --no-global to skip that expansion and just see the reference. For a column bound to a global, the full describe view (L05) even hands you the global's global_optionset_id.

03 Why the difference matters

Reach for…When…
Local option setThe choices mean something only to one column — contact's gender, this table's custom "stage." No reuse intended.
Global option setThe same list recurs across the model — a company-wide "Region," "Priority," or "Approval status" you want defined once and kept consistent everywhere it's used.

It's the same reasoning as a shared enum versus a one-off constant in code. You'll feel this directly in M04: when you author a custom picklist column, you'll choose whether to mint a local set or bind an existing global. Reading them now means that choice won't be blind.

!
Editing a global ripples. Because many columns share it, changing a global option set's members changes them everywhere at once. Convenient for consistency, dangerous if you forget the blast radius. Local sets have no such reach.

04 Do it yourself ~4 min

All read-only. Resolve some codes, then meet the globals:

# 1 · the gender codes M02 took on faith — now confirmed
crm --json metadata picklist contact gendercode

# 2 · the Status option set every record carries
crm --json metadata picklist contact statuscode

# 3 · list the org's GLOBAL option sets (IsGlobal:true)
crm --json metadata list-optionsets

# 4 · retrieve one global directly, by name
crm --json metadata get-optionset componentstate
  1. Query 1: which code is "Female"? Re-read M02 L02 — is that the value you set there?
  2. Query 2: list every statuscode option for contact as code → label. Which code means "Inactive"?
  3. Query 1 vs query 4: compare the IsGlobal flag and the Name. Which is <table>_<column>, which is a standalone name?
  4. Re-run query 1 with --no-global. Does gendercode change? (It's local — predict before you run.)

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · A requirement says "set the contact's gender to Female." What do you actually send in the PATCH?
Correct. The API stores and accepts the code. You resolve "Female" → 2 with metadata picklist, then send the number. The label is never written.
Option sets store an integer code, not the label text. metadata picklist tells you "Female" = 2; you send 2.
2 · IsGlobal: true on an option set means…
Correct. "Global" = reusable. Defined once as its own object, bound by any number of columns. Editing it changes them all. A local set (IsGlobal:false) lives on a single column.
Global is about reuse, not visibility or scope of records. It's a named option set defined once and shared by many columns/tables — edit it once, every binding updates.
3 · You want to inspect a global option set without going through any particular column. Which verb?
Correct. Globals are first-class objects with their own verbs: list-optionsets to enumerate, get-optionset <name> to retrieve one with its options — no column needed.
Globals stand on their own. list-optionsets enumerates them and get-optionset <name> fetches one directly — no table or column in the picture.
4 · By default, what does metadata picklist do when a column is bound to a global set?
Correct. picklist resolves the options whether the set is local or global — it expands a bound global by default. --no-global skips the expansion if you only want the reference.
It expands the bound global inline by default, so you always get usable {value,label} pairs. Use --no-global only when you want to skip that and see the bare reference.
