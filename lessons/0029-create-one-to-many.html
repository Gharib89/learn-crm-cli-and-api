Milestone 05 / Relationships / Lesson 02

Wire it up:
one-to-many

One atomic command makes the relationship and its lookup column together — preview it, run it, read it back.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. L01 gave you the model: a 1:N is a lookup on the child. metadata create-one-to-many builds both halves — the relationship metadata and the lookup attribute — in a single server action. That atomicity matters: you never end up with a half-made relationship. Learn this verb and you can connect any custom table to account, contact, or each other.

Today's win: create a real 1:N from account to a custom project table, previewing the exact request first, then confirm the lookup column appeared on the child.

01 Four required ideas, one call

Every 1:N is "this parent → those children, reached through this lookup." The verb asks for exactly that — four required flags map straight onto the L01 vocabulary:create-one-to-many

FlagMeans (L01 term)Our value
--referenced-entitythe "1" / parent sideaccount
--referencing-entitythe "N" / child side (gets the lookup)ag_project
--schema-namethe relationship's own name (prefixed)ag_account_ag_project
--lookup-schemaschema name of the lookup column to createag_AccountId
--lookup-displayUI label for that lookup"Account"
✓
"Atomically" is the headline. The help text says it: "Create a 1:N relationship and its lookup attribute atomically." One POST to RelationshipDefinitions creates the relationship metadata and the lookup column on the child — there's no separate "add the lookup" step, and no partial state if it fails.

02 Preview first: --dry-run

Same global flag from M01 L04 — and here it's especially worth it, because the dry-run shows the entire request body the CLI assembles, including defaults you didn't type:create-one-to-many

agent-cloud · exit 0 · dry-runcrm --json --dry-run metadata create-one-to-many \
    --schema-name ag_account_ag_project \
    --referenced-entity account \
    --referencing-entity ag_project \
    --lookup-schema ag_AccountId --lookup-display "Account" \
    --lookup-required ApplicationRequired

{ "ok": true, "data": { "_dry_run": true,
    "method": "POST", "url": "…/RelationshipDefinitions",
    "body": {
      "@odata.type": "…OneToManyRelationshipMetadata",
      "SchemaName": "ag_account_ag_project",
      "ReferencedEntity": "account", "ReferencingEntity": "ag_project",
      "CascadeConfiguration": {          # defaults you didn't type → L03
        "Assign":"NoCascade", "Delete":"RemoveLink", "Reparent":"NoCascade",
        "Share":"NoCascade", "Unshare":"NoCascade", "Merge":"NoCascade" },
      "Lookup": { "SchemaName": "ag_AccountId",
        "RequiredLevel": { "Value": "ApplicationRequired" } } },
    "_exists": false, "would_skip": false,
    "references": [          # the CLI pre-checks both ends exist
      {"kind":"referenced_entity","value":"account","_exists":true},
      {"kind":"referencing_entity","value":"ag_project","_exists":true}
    ] }, "meta": { "dry_run": true } }

Two things the preview earns you for free: the CascadeConfiguration block — the defaults that decide what happens to children when the parent is deleted or reassigned (all of L03) — and the references check, confirming both tables exist before you write. No write went out; this is pure planning.

03 Run it — read the envelope

Drop --dry-run and the relationship is born. The envelope reports the new lookup column's logical name and the relationship id:create-one-to-many

agent-cloud · exit 0crm --json metadata create-one-to-many \
    --schema-name ag_account_ag_project --referenced-entity account \
    --referencing-entity ag_project \
    --lookup-schema ag_AccountId --lookup-display "Account" \
    --lookup-required ApplicationRequired

{ "ok": true, "data": {
    "created": true, "kind": "OneToMany",
    "schema_name": "ag_account_ag_project",
    "referenced_entity": "account", "referencing_entity": "ag_project",
    "referencing_attribute": "ag_accountid",   # ← the lookup column, auto-lowercased
    "relationship_id": "2c99a64a-926a-f111-…",
    "solution": "itworx_dev", "published": true } }

The key field is referencing_attribute: ag_accountid — the lookup column now sitting on ag_project. Notice it's the lowercase of your --lookup-schema (ag_AccountId → ag_accountid): the schema name is the cased "design" name, the logical name is its lowercase form, exactly the three-names rule from M03 L01. And published: true — the call ran PublishAllXml for you.

!
The relationship needs both tables to already exist. create-one-to-many connects two existing tables — it does not create the child. If ag_project isn't there yet, build it first with create-entity/scaffold (M04). The dry-run's references[]._exists is your pre-flight check.

04 Confirm it landed

Two reads prove the write. First, the relationship itself shows up under the child's ManyToOne (L01); second, the lookup column is now a real attribute you can describe:

agent-cloud · exit 0# the relationship — from the child's viewpoint it's ManyToOne
crm --json metadata relationships ag_project
# → ManyToOne[] contains ag_account_ag_project, ReferencingAttribute: ag_accountid

# the lookup column now exists on the child table
crm --json metadata attribute ag_project ag_accountid
{ "AttributeType": "Lookup", "LogicalName": "ag_accountid",
  "SchemaName": "ag_AccountId", "IsCustomAttribute": true }

That's the whole loop: model it (L01) → preview it → write it → read it back. The lookup is real; in L04 you'll actually use it to point a project row at an account.

05 Do it yourself ~8 min writes

This creates a real table, a real relationship, then deletes both so the org ends clean. (Deleting the relationship also removes its lookup column — you'll formalise this in L05.)

# 0 · need a child table to attach to (skip if you still have one from M04)
crm --json metadata create-entity --display "Project"

# 1 · PREVIEW the 1:N — read the body + cascade defaults, no write
crm --json --dry-run metadata create-one-to-many \
    --schema-name ag_account_ag_project --referenced-entity account \
    --referencing-entity ag_project \
    --lookup-schema ag_AccountId --lookup-display "Account"

# 2 · run it for real (drop --dry-run)
crm --json metadata create-one-to-many \
    --schema-name ag_account_ag_project --referenced-entity account \
    --referencing-entity ag_project \
    --lookup-schema ag_AccountId --lookup-display "Account"

# 3 · confirm the lookup column exists on the child
crm --json metadata attribute ag_project ag_accountid

# 4 · clean up (removes relationship + lookup, then the table)
crm --json metadata delete-relationship ag_account_ag_project --yes
crm --json metadata delete-entity ag_project --yes
  1. In step 1's body, find CascadeConfiguration. What is Delete set to by default? (Hold that thought — it's the heart of L03.)
  2. In step 2's envelope, read referencing_attribute. Why is it lowercase when you passed ag_AccountId?
  3. In step 3, confirm AttributeType is Lookup and IsCustomAttribute is true. Which table is it on — and why that one?
  4. Try swapping --referenced-entity and --referencing-entity. Predict which table would get the lookup before you read it back.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You want each project to belong to one account. Which flags get account and ag_project?
Correct. Account is the "1"/parent → --referenced-entity. Project is the "N"/child that gets the lookup → --referencing-entity. Swap them and the lookup lands on account instead.
Parent (the "1", account) is referenced; child (the "N", project) is referencing and receives the lookup. Order is not symmetric — it decides which table gets the column.
2 · The envelope returns referencing_attribute: "ag_accountid". What is that?
Correct. referencing_attribute is the lookup column the call created on the referencing (child) table — the logical (lowercase) form of --lookup-schema.
It's the new lookup column on the child (ag_project). The relationship name is schema_name; the lookup column is referencing_attribute.
3 · Why is --dry-run especially useful on this verb?
Correct. The dry-run prints the assembled CascadeConfiguration defaults and the references[] existence pre-check — you see exactly what would be sent, and that both ends exist, before committing.
Dry-run writes nothing. Its value here is visibility: the cascade defaults you didn't type, plus a confirmation both tables exist. It's not required, and it never creates anything.
4 · create-one-to-many fails with "referencing entity does not exist." What's the fix?
Correct. The relationship links two tables that must already exist. Build the child with create-entity/scaffold (M04) first; the dry-run's references[]._exists catches this before you write.
The verb only connects existing tables. If the child isn't there, create it first (M04). The dry-run existence check would have flagged it.
