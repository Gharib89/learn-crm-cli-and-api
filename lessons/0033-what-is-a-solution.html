The customization container

Before you can ship a change, you need to know what wraps it.

target  agent-cloud  ·  orgexample.crm.dynamics.com  ·  read-only
Why now. Every table, column, relationship, and form you've built in M03–M05 lives somewhere — a solution. Solutions are D365's deployment unit: they're what you export from dev and import to test and prod. Get this model wrong and your changes arrive in the wrong place, overwrite the wrong things, or simply don't move at all.

A solution is a ZIP-shaped container: you put customization components in, then move the ZIP between orgs. The publisher inside the ZIP is what gives your components their prefix — the itworx_ you've been typing was always solution-scoped.

01 The three-layer model

Every customization in D365 sits inside a three-layer stack: a Publisher stamps names, a Solution groups components, and Components are the actual metadata pieces.

Publisher

A publisher has a uniquename (e.g. itworx), a component prefix (e.g. itworx_), and an option-value offset (e.g. 72000). One publisher per org per ISV/team. It stamps every component name so customizations from different vendors never collide.

Solution

A named container bound to a publisher. The solution has a uniquename (e.g. itworx_dev), a friendly name (e.g. ITWorx Dev Solution), and a version (1.0.0.0). An org can have many solutions — Microsoft ships dozens of system solutions alongside yours.

Component

A piece of metadata registered in a solution: a table (type 1), a column (type 2), a web resource (type 26), a workflow (type 29), and so on. The same component can appear in multiple solutions.

Publisher: itworx  (prefix: itworx_, option-value offset: 72000)
  └─ Solution: itworx_dev  (version 1.0.0.0, unmanaged)
       ├─ Component: itworx_project  [type 1 — Entity]
       ├─ Component: itworx_project_name  [type 2 — Attribute]
       └─ Component: itworx_status_choices  [type 59 — OptionSet]
✓
The prefix is publisher-scoped, not solution-scoped. Both itworx_dev and a future itworx_v2 solution share the same publisher and therefore the same itworx_ prefix. You don't need a new publisher per solution — just per team or ISV.

02 Managed vs Unmanaged

Every solution ZIP exists in one of two states, and the state determines how it behaves after import.

Unmanaged — the dev layer

Components are loose in the org. You can modify them in place. Export an unmanaged ZIP and import it to the same org — it updates in place. This is your dev layer. You build and iterate here.

Managed — the production layer

Read-only after import. Can be upgraded by reimporting a newer version. Can be uninstalled cleanly — removing a managed solution removes all its components from the target org. When you ship to prod you typically import as managed.

You cannot convert a managed solution to unmanaged. If you import a managed ZIP to an org where you need to edit the components, you're stuck. Always keep your unmanaged source in the dev org.

03 The ALM pipeline

A typical D365 ALM pipeline moves customizations through three environments. The CLI automates all the transfer steps — only the final merge decision in prod is ever manual.

DEV org                   TEST org                    PROD org
itworx_dev (unmanaged) ── export ──► import ──► export --managed ──► import
  1. Dev org — build and iterate with the unmanaged solution. All the work in M03–M05 happened here.
  2. Export — solution export itworx_dev produces a ZIP. Unmanaged export for test, managed export for prod.
  3. Test org — import the unmanaged ZIP. QA tests against it. You can still inspect and compare components.
  4. Export as managed — solution export itworx_dev --managed from the dev org. Same content, locked for distribution.
  5. Prod org — import the managed ZIP. Components are live and read-only. Upgrade by reimporting a newer managed version.
✓
Both your training orgs are dev orgs. agent-cloud and agent-on-prem are both unmanaged dev environments — you never need to worry about managed solutions in this course. But knowing the distinction is what lets you judge what goes where in a real deployment.

05 Check yourself

Three questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · What is a solution publisher's main job?
Correct. The publisher owns the prefix (e.g. itworx_) that is prepended to every component name — tables, columns, option sets — ensuring no two vendors' customizations collide in the same org.
The publisher's job is to stamp every component name with its unique prefix (e.g. itworx_). It has nothing to do with permissions, versions, or language settings.
2 · You export a solution ZIP and import it to the test org without the --managed flag. What lands in test?
Correct. Without --managed you export an unmanaged ZIP. Importing it creates an unmanaged solution in the target org — components are editable. This is the normal test-org flow.
No flag = unmanaged export = unmanaged import. Components land as editable. --managed is only needed when you want the locked, distributable form (typically for prod).
3 · Which of these tasks requires the GUI — not the crm CLI?
Correct. The Solution Checker calls the Power Apps Checker API — it's not wired into the crm CLI. All the others (export, create-publisher, add-component) are fully automatable.
Exporting, creating a publisher, and adding components are all CLI commands. Solution Checker is the exception — it requires the Power Apps Checker API and lives in the GUI (or a separate CI integration).
