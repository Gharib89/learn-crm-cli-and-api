The Profile
Lifecycle

Install the CLI, then create, inspect, switch and edit the saved connection it runs against. This is the doorway to everything else.

crm v3.9.0  ·  5 profiles  ·  active agent-cloud
Why this, first. The crm CLI never talks to "a server" in the abstract — it talks to whichever profile is active. A profile bundles where (URL), how (auth scheme), and the secret. Get this model right and every later command has a known target; get it wrong and you'll customize the wrong org. Everything starts here.

Today's win: read your real profile list, understand what each field means, and safely switch the active profile and back — the CLI's one piece of persistent state.

01 Install — how the binary got here

The crm binary bundles its own Python runtime — there's nothing else to install.skill One line per host:

# Linux / WSL (you're here — already on v3.9.0)
curl -fsSL https://pub-bbeb86c46454443ca76521dd4d29818e.r2.dev/install.sh | sh

# Windows (PowerShell)
irm https://pub-bbeb86c46454443ca76521dd4d29818e.r2.dev/install.ps1 | iex

# open a new shell so PATH updates, then verify
crm --version          # → crm, version 3.9.0

You already have it installed, so that's the last you'll think about install. The real work — and the part you'll repeat on every new machine or org — is the profile.

02 What a profile is

A profile is a named, saved connection. It answers three questions so you don't retype them every command:

FieldQuestionExample
URLWhich org?https://orgexample.crm.dynamics.com
Auth schemeHow do I prove who I am?oauth or ntlm
SecretThe credential itselfOAuth client secret / NTLM password
DefaultsOptional conveniences--default-solution, --publisher-prefix

The CLI is stateful: exactly one profile is active at a time, and unflagged commands run against it. The auth scheme is inferred from the URL — a *.dynamics.com host means cloud → OAuth; anything else means on-prem → NTLM.skill You can override with --auth-scheme, but the default is almost always right.

03 profile add — create & connect

crm profile add does four things in one shot: save the secret → test the connection (WhoAmI) → activate the profile.skill Run it bare for an interactive wizard, or pass flags for scripting:

# interactive wizard — easiest on a terminal
crm profile add

# cloud (OAuth) — non-interactive
crm profile add --url https://org.crm.dynamics.com --name mycloud \
  --tenant-id <aad-tenant> --client-id <app-id> --client-secret '<secret>'

# on-prem (NTLM)
crm profile add --url http://crm.contoso.local/contoso --name onprem \
  --username alice --domain CONTOSO --password '...'

Where the secret lives

The secret is stored in the OS keyring when available, or as a 0600 plaintext field in the profile file when it isn't (WSL / headless), or whenever you pass --store-password-plaintext. It's keyring or plaintext — a single store, never both.skill

!
Security note (your setup). On WSL there's no keyring, so all five of your profiles show cred=plaintext — the secret sits in a 0600 file readable by your user. That's expected here and these are mutate-safe dev orgs, but treat that file like any other credential: never commit it, and prefer per-run --password if you ever script against a sensitive org.

Credential resolution at run time is a fixed order: --password (per-run override) > stored secret > TTY prompt. There is no environment-variable fallback — no .env, no D365_*. The only env knob is CRM_HOME (where state lives).skill

04 list · use · edit — the lifecycle

crm profile list shows every saved profile; ● marks the active one. Here's your real list:

crm profile list  Profiles
  ────────
  ● agent-cloud:   cloud    https://orgexample.crm.dynamics.com   cred=plaintext
  ○ agent-on-prem: on-prem  http://crm.contoso.local/Contoso     cred=plaintext
  ○ cloud:         cloud    https://orgexample.crm.dynamics.com   cred=plaintext
  ○ crmworx:       on-prem  http://crm.contoso.local/Contoso     cred=plaintext
  ○ mocedev:       on-prem  https://devdigitalplatform.mocd.gov.ae/MoCD  cred=plaintext

Each row is name · kind · URL · where the secret is stored. The full lifecycle:

CommandDoesTouches secret?
profile addCreate, save secret, test, activatesets it
profile listList all; mark the active ●—
profile use [name]Switch active. No name → picker; --none → clear—
profile edit NAMEChange fields: url, username, domain, tenant-id, client-id, api-version, default-solution, publisher-prefixno — not the secret
profile set-password NAMEStore / replace the secretsets it
profile delete-passwordRemove the stored secretclears it
profile rmDelete the profile and its secretclears it
✓
The one trap to remember: profile edit changes everything except the secret. To rotate a password or client secret you use profile set-password — not edit. (Editing the URL between two orgs without re-checking the secret is a classic way to get a confusing auth failure.)

05 Do it yourself ~3 min

All read-only or fully reversible — your secrets are never touched:

  1. List what you've got, and read the active marker:
    crm profile list  — which row has the ●?
  2. Switch the active profile to the spare cloud one, then list again to see ● move:
    crm profile use cloud  →  crm profile list
  3. Switch back to your primary target (do this — later lessons assume it):
    crm profile use agent-cloud
  4. Inspect what fields edit could change (no changes made):
    crm profile edit --help
!
Leave these alone for now. profile rm, delete-password, and set-password change credentials/state. You don't need them today — we'll use them deliberately when there's a reason.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You run crm profile add --url https://contoso.crm.dynamics.com … The CLI picks an auth scheme. Which, and why?
Correct. A *.dynamics.com URL = cloud, so the CLI infers oauth (client-credentials). A non-dynamics host would infer ntlm. You can override with --auth-scheme, but you rarely need to.
Not quite. The scheme is inferred from the URL: *.dynamics.com → oauth; anything else → ntlm. So this cloud URL gives OAuth, no flag required.
2 · A client secret expired. You need to rotate it on your agent-cloud profile. Which command?
Correct. profile edit changes every field except the secret; the secret is rotated with profile set-password. (There's no --client-secret on edit at all.)
No. profile edit deliberately can't touch the secret — that's what profile set-password is for. use just switches active; --rotate isn't a thing.
3 · Your profile list shows cred=plaintext on every row. What does that mean?
Correct. Keyring or plaintext is the at-rest store. WSL has no keyring, so the CLI falls back to a 0600 plaintext field — expected on this setup. (Auth on the wire is still NTLM/OAuth as normal; "plaintext" is only about disk.)
Re-read §3: "plaintext" describes where the secret rests on disk, not the network. WSL has no keyring, so it uses a 0600 file. Nothing is broken, and there's no env-var store.
4 · You set CRM_PASSWORD=hunter2 in your shell, then run a command with no --password and no stored secret. What happens?
Correct. Credential resolution is --password > stored secret > TTY prompt — full stop. There is no D365_*/CRM_* password env var and no .env autoload. The only env knob the CLI reads is CRM_HOME.
No — the CLI deliberately reads no credential env var and no .env. With no --password and no stored secret, it falls through to a TTY prompt (or errors cleanly under --json/non-TTY).
