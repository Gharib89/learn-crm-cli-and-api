Create a system view

Define the column set, filter, and sort order for a table's list lens.

reads writes  ·  live on agent-cloud  ·  orgexample.crm.dynamics.com
Why now. The list UI is the first thing a user sees when they open a table. D365 auto-creates "Active" and "Inactive" views when you create an entity — but you'll almost always want a custom view: fewer columns, filtered on status, sorted by priority. That's what this lesson teaches.

A view is a D365 saved query (table: savedquery) that answers "which rows, which columns, in which order". Every list in a model-driven app is rendered from a view. The CLI lets you inspect the auto-generated views and create custom ones in a single command.

01 Read the auto-generated views

Every create-entity call auto-provisions two public views (querytype 0):

  • Active <Table> — isdefault: true, filter: statecode=0
  • Inactive <Table> — isdefault: false, filter: statecode=1

Use view list to read them back:

agent-cloud · exit 0crm --json view list ag_ticket

{ "ok": true, "data": [
    {
      "name":          "Active Tickets",
      "savedqueryid":  "a1b2c3d4-0001-0000-0000-000000000000",
      "isdefault":     true,
      "querytype":     0
    },
    {
      "name":          "Inactive Tickets",
      "savedqueryid":  "a1b2c3d4-0002-0000-0000-000000000000",
      "isdefault":     false,
      "querytype":     0
    }
  ] }

The querytype field controls where the view appears. The full set:

querytypeView kindWhere it appears
0Public viewView picker in model-driven app lists
1Advanced Find viewColumn set for the Advanced Find dialog
2Associated viewShown on a form's subgrid for related records
4Quick Find viewColumns searched by the list search bar
64Lookup viewColumns shown in lookup / relationship dialogs

In this lesson we create a public view (querytype 0) — the kind that appears in the view picker.

02 The ObjectTypeCode requirement

view create requires --otc: the entity's integer ObjectTypeCode. This is not the MetadataId GUID — it's the integer D365 embeds in the layoutxml <grid object="N"> attribute. You must read it from the metadata before running view create.

agent-cloud · exit 0crm --json metadata entity ag_ticket

{ "ok": true, "data": {
    "LogicalName":      "ag_ticket",
    "SchemaName":       "ag_ticket",
    "MetadataId":       "246395a6-b1c2-4d3e-9f0a-1b2c3d4e5f6a",
    "ObjectTypeCode":   10828,
      … other fields …
  } }

The value you need is ObjectTypeCode — an integer, not a GUID.

!
OTC is org-specific for custom entities. The value 10828 is from agent-cloud — your org will differ. Always read it from metadata entity before running view create. System entities (like account or contact) have fixed OTCs across all orgs, but custom entities do not.

03 Create a custom view

view create builds the fetchxml (filter + sort) and layoutxml (column set) for you from the flags you pass. Key flags:

Flag / ArgRequiredDescription
ENTITY (positional)yesLogical name of the table
--name TEXTyesDisplay name of the view
--otc INTEGERyesObjectTypeCode from metadata entity
--column TEXTRepeatable; logicalname[:width]; column order preserved
--order TEXTSort expression, e.g. ag_priority desc
--filter-activeAdds statecode=0 condition to fetchxml
--defaultMarks this view as the default for the entity
--query-typeDefault: public. Options: public, advanced-find, associated, quick-find, lookup
--if-exists [error|skip]Behaviour when a view with the same name already exists
--solution TEXTRegister the view in this solution

Dry-run first

Pass --dry-run to see the fetchxml and layoutxml the CLI will send — without writing anything to the org:

dry-run · no org writecrm --json --dry-run view create ag_ticket \
    --name "Open Tickets" \
    --otc 10828 \
    --column ag_name:200 \
    --column ag_status:150 \
    --column ag_priority:100 \
    --filter-active \
    --solution itworx_dev

{ "ok": true, "dry_run": true, "body": {
    "name": "Open Tickets",
    "returnedtypecode": "ag_ticket",
    "querytype": 0,
    "fetchxml": "<fetch version=\\"1.0\\" output-format=\\"xml-platform\\" mapping=\\"logical\\" no-lock=\\"false\\"><entity name=\\"ag_ticket\\"><attribute name=\\"ag_name\\"/><attribute name=\\"ag_status\\"/><attribute name=\\"ag_priority\\"/><filter type=\\"and\\"><condition attribute=\\"statecode\\" operator=\\"eq\\" value=\\"0\\"/></filter></entity></fetch>",
    "layoutxml": "<grid name=\\"resultset\\" object=\\"10828\\" jump=\\"ag_name\\" select=\\"1\\" icon=\\"1\\" preview=\\"1\\"><row name=\\"result\\" id=\\"ag_ticketid\\"><cell name=\\"ag_name\\" width=\\"200\\"/><cell name=\\"ag_status\\" width=\\"150\\"/><cell name=\\"ag_priority\\" width=\\"100\\"/></row></grid>"
  } }

The fetchxml carries the statecode=0 filter from --filter-active, and the layoutxml has the three column cells in the exact order specified. The object="10828" attribute in the grid is your OTC.

Live create

Remove --dry-run to write to the org:

agent-cloud · exit 0crm --json view create ag_ticket \
    --name "Open Tickets" \
    --otc 10828 \
    --column ag_name:200 \
    --column ag_status:150 \
    --column ag_priority:100 \
    --filter-active \
    --solution itworx_dev

{ "ok": true,
  "created":      true,
  "name":         "Open Tickets",
  "entity":       "ag_ticket",
  "savedqueryid": "a1b2c3d4-0038-0000-0000-000000000001",
  "solution":     "itworx_dev",
  "published":    true
}

The CLI publishes the view automatically (published: true). It is immediately visible in the view picker. Save the savedqueryid — you'll need it to reference or delete the view.

✓
Column order is preserved exactly as specified. The platform does not re-sort. If you want "Name" first, put ag_name first in --column. Width is in pixels and is advisory — users can resize in the browser.

Deleting a view

There is no crm view delete command. Views are savedquery records — delete them through entity delete:

agent-cloud · exit 0crm --json entity delete savedqueries a1b2c3d4-0038-0000-0000-000000000001 --yes

{ "ok": true,
  "deleted":        true,
  "_entity_id":     "a1b2c3d4-0038-0000-0000-000000000001",
  "_entity_id_url": "https://orgexample.crm.dynamics.com/api/data/v9.2/savedqueries(a1b2c3d4-0038-0000-0000-000000000001)"
}

05 Check yourself

Three questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You run crm --json view list ag_ticket on a freshly created entity. Which views do you see?
Correct. create-entity auto-provisions two public views (querytype 0): the default "Active <Table>" (statecode=0 filter) and "Inactive <Table>" (statecode=1 filter).
D365 auto-provisions two views when you create an entity: "Active <Table>" (default) and "Inactive <Table>". You only see this entity's views, not all entities.
2 · You run view create and get an error about a wrong object attribute in layoutxml. What's the likely cause?
Correct. --otc requires the integer ObjectTypeCode, not the MetadataId GUID. Get it from crm metadata entity ag_ticket → ObjectTypeCode field. It varies per org for custom entities.
The --otc flag must be the integer ObjectTypeCode (e.g. 10828), NOT the MetadataId GUID. Get it from crm --json metadata entity ag_ticket and look for ObjectTypeCode.
3 · You want to delete a view you created. What is the correct command?
Correct. There is no crm view delete command. Views are savedquery records — delete them via entity delete savedqueries <id> --yes.
There is no crm view delete. Views are saved as savedquery entity records. Delete them with crm --json entity delete savedqueries <savedqueryid> --yes.
