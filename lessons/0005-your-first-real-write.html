Your First
Real Write

Drop --dry-run, create one record for real, read it back, then clean it up — the whole safe-write loop, start to finish.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. Four lessons of groundwork converge here. You can connect (L02), read any result (L03), preview a write (L04), and find records (L06's coming). This is the moment the CLI stops being read-only and actually changes the org — done the disciplined way, so it's never scary.

Today's win: run the full loop — preview → commit → verify → clean up — and create (then remove) a real contact in your live cloud org, reading the server's confirmation at every step.

Beats 1, 3 and 4 you've already met or will in L06 — the only new action is beat 2, and it's just beat 1 with --dry-run removed. That's the deliberate design: the command you commit is identical to the one you previewed, minus the safety flag. No surprises between plan and execution.

02 Commit — the real POST

Same command as your L04 preview, with --dry-run gone:

agent-cloud · exit 0 · a row now existscrm --json entity create contacts --data '{"firstname":"Ada","lastname":"Lovelace","jobtitle":"Mathematician"}'

{
  "ok": true,
  "data": {
    "@odata.context": "https://orgexample.crm.dynamics.com/api/data/v9.2/$metadata#contacts/$entity",
    "@odata.etag": "W/\"3053428\"",
    "contactid": "1f6c71bf-3467-f111-a825-6045bd05c06e",   ← the server-assigned GUID
    "firstname": "Ada", "lastname": "Lovelace",
    "fullname": "Ada Lovelace",                  ← computed by the server
    "jobtitle": "Mathematician",
    "statecode": 0, "statuscode": 1,          ← default state: Active
    "createdon": "2026-06-13T14:32:48Z",
    "_ownerid_value": "bf40e4d2-…",           ← your application user (L02)
    "telephone1": null, "emailaddress1": null, … 120+ more columns, mostly null
  }
}

Two things the server did that you didn't ask for:

  • It assigned the contactid GUID and computed derived fields (fullname from first + last), stamped createdon, set the default statecode/statuscode, and made your application user (from L02's whoami) the owner.
  • It returned the whole new row. Remember the Prefer: return=representation header in L04's dry-run plan? That's why — the create echoes the full created record back (all ~130 columns). To skip the firehose and get just the GUID, add --no-return.create
✓
Grab that contactid. It's the handle for everything next — the GUID you'll feed to entity get, update, or delete. The create response is the one place the server hands it to you.

03 Verify — don't trust, confirm

ok:true on the create is strong evidence, but the disciplined habit is to read the row back from the server with entity get (L06) — independent confirmation it's really persisted:

agent-cloud · exit 0crm --json entity get contacts 1f6c71bf-3467-f111-a825-6045bd05c06e --select fullname,jobtitle,statuscode

{ "ok": true, "data": {
    "fullname": "Ada Lovelace",
    "jobtitle": "Mathematician",
    "statuscode@…FormattedValue": "Active",  ← human label for the code
    "statuscode": 1
  } }

Bonus you'll see often: the server attaches @…FormattedValue twins to coded fields — statuscode: 1 comes with "Active". Those are the human-readable labels behind every option-set number; you'll rely on them once you work with choice columns.query

04 Clean up — delete needs a seatbelt of its own

Because this was a throwaway, remove it. delete is your first destructive verb, and it's guarded: run non-interactively without confirming and it refuses (you saw "aborted by user" in L03). Pass --yes to confirm intent:

agent-cloud · exit 0crm --json entity delete contacts 1f6c71bf-3467-f111-a825-6045bd05c06e --yes

{ "ok": true, "data": { "deleted": true, "id": "1f6c71bf-…" } }
!
--yes is the destructive-verb confirmation. It says "yes, I mean it" so the command doesn't hang on a prompt — but it removes the safety question, so only attach it when you're certain. delete here is permanent; there's no --dry-run undo after the fact (preview before, not after).

05 Do it yourself ~4 min

Run the whole loop on agent-cloud. This one does write — but you preview first and delete at the end, so the org is left exactly as you found it:

# 1 · PREVIEW (validate names + see the plan — writes nothing)
crm --json --dry-run entity create contacts --data '{"firstname":"Grace","lastname":"Hopper"}' --validate

# 2 · COMMIT (same line, --dry-run removed) — copy the contactid from the output
crm --json entity create contacts --data '{"firstname":"Grace","lastname":"Hopper"}'

# 3 · VERIFY — read it back
crm --json entity get contacts <paste-contactid> --select fullname,createdon

# 4 · CLEAN UP
crm --json entity delete contacts <paste-contactid> --yes

Confirm, from your own output:

  1. In step 2, which field is the server-assigned GUID, and what fullname did the server compute that you never typed?
  2. Step 3 returns ok:true — what does that prove that step 2's ok:true didn't?
  3. Step 4: what does data contain on a successful delete? Re-run step 3 now — what exit code and category do you get?
✓
You just did the thing. Every customization later in the mission — a column, a relationship, a plug-in step — is this same loop at a larger scale: preview, commit, verify. You now own the core move.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You previewed a create with --dry-run and it looked right. What's the exact command to commit it for real?
Correct. That's the design: the committed command is the previewed one minus --dry-run. No --commit flag, no magic second run — just drop the safety flag. (--yes is for destructive verbs like delete, not create.)
Not quite. There's no --commit flag and re-running changes nothing. You commit by running the identical command with --dry-run taken off. --yes is a different safety, for destructive verbs.
2 · The create returns "fullname": "Ada Lovelace" and a contactid you never put in --data. Where did they come from?
Correct. The server assigns the GUID and computes derived fields (fullname from first+last), then — thanks to Prefer: return=representation — returns the whole created row. The response is the persisted record.
No. The CLI doesn't invent IDs or names. Dataverse assigns the contactid and computes fullname server-side, then returns the full created row (that's the return=representation behavior from L04's plan).
3 · After a create returns ok:true, why still run entity get on the new GUID?
Correct. The create's ok:true is strong, but reading the row back is the disciplined confirmation that closes the loop — the same verify habit you'll want before trusting any write in production work.
Not quite. The create already persisted and assigned the GUID; nothing rolls back. The reason to get is verification — independent proof from the server that the row is really there.
4 · You run entity delete contacts <guid> non-interactively (under --json) without --yes. What happens?
Correct. Destructive verbs need explicit confirmation. Non-interactively without --yes, the CLI aborts safely ("aborted by user", exit 1) rather than guessing or hanging — so an unattended script can't delete by accident.
Not quite. It doesn't delete without confirmation and it doesn't hang. Non-interactively it aborts safely — {ok:false, "aborted by user"}, exit 1 — protecting you from an accidental destructive write. --yes is what authorizes it.
