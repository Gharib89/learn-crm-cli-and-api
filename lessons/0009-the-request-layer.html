The Request
Layer

One session, one auth adapter, four headers — how D365Backend turns a verb call into an authenticated HTTP request.

source  github.com/Gharib89/crm  ·  @ main
Why now. L01 placed the transport box at the bottom of the map. This is the box where M01's biggest split lives: cloud (OAuth, v9.2) vs on-prem (NTLM, v9.1). In L02 of M01 you saw that crm infers the auth scheme from the URL — now you'll read the code that acts on it. Get this layer, and "why does the same command behave differently on two orgs?" becomes a question you answer by reading one method.

Today's win: read _make_auth and name, for any profile, exactly which credentials go on the wire and where the secret lives.

01 One session, built once per run

Everything in this layer hangs off a single object: D365Backend. Its constructor builds one requests.Session, attaches the auth adapter and TLS setting, and reuses it for every call that run:init

class D365Backend:
    def __init__(self, profile, password, ...):
        # requests imported here, not at module top (fast startup, issue #247)
        import requests
        self._session = requests.Session()
        self._session.auth   = self._make_auth(password)   # ← the auth branch
        self._session.verify = profile.verify_ssl            # TLS verification

The base URL is built from the profile in the api_base property — the version segment comes straight from the profile, which is why cloud reports v9.2 and on-prem caps at v9.1 (the M01 L02 tell):init

return f"{self.url.rstrip('/')}/api/data/{self.api_version}/"
# cloud  → https://orgexample.crm.dynamics.com/api/data/v9.2/
# on-prem→ http://crm.contoso.local/Contoso/api/data/v9.1/

02 The auth branch — one switch, three worlds

_make_auth is a single switch on profile.auth_scheme. That one field — inferred from your URL back in M01 L02 — picks the entire credential mechanism:auth

def _make_auth(self, password):
    scheme = self.profile.auth_scheme
    if scheme == "ntlm":                 # on-prem, Windows Integrated
        user = f"{domain}\\{username}" if domain else username
        return HttpNtlmAuth(user, password)
    if scheme in ("kerberos", "negotiate"):  # on-prem AD via SSPI
        return HttpNegotiateAuth()       # no creds in the profile
    if scheme == "oauth":                # cloud / Dataverse online
        return self._make_oauth_auth(password)
    raise D365Error("Unknown auth_scheme ...")

Each branch returns a requests auth adapter — an object that knows how to sign every outgoing request. The session calls it automatically; the rest of crm never thinks about auth again.

✓
The whole cloud-vs-on-prem split is this one method. No other layer branches on org type. So if auth misbehaves, this is the only file you read — and a profile's auth_scheme is the only field that steers it.

03 NTLM vs OAuth, side by side

The two you'll actually meet (your agent-on-prem and agent-cloud profiles):

Secret usedthe user's passwordapp-registration client secret
Profile fieldsusername + domaintenant_id + client_id
What's sentNTLM handshake each connection; DOMAIN\user + hashed challengeAuthorization: Bearer <token> header

The OAuth side is the more involved one. _make_oauth_auth derives the scope from your org host and points at the public-cloud authority, then hands msal a cache:oauth

host      = urlparse(profile.url).netloc
scope     = f"https://{host}/.default"                 # e.g. orgexample.crm.dynamics.com/.default
authority = f"https://login.microsoftonline.com/{tenant}"

Then, on every request, the bearer adapter asks msal for a token and stamps the header — msal serves it from cache and only round-trips to Entra ID (AAD) when none is valid:oauth

def __call__(self, request):                 # runs per request
    result = app.acquire_token_for_client(scopes=[self._scope])
    request.headers["Authorization"] = f"Bearer {token}"
    self._persist_cache()                      return request
!
A rejected credential is operational, not transient. The OAuth path does no automatic refresh-retry on a bad secret — it raises a 401 D365Error pointing you at the app registration. (That feeds the M01 L03 envelope's category: auth_failed.)

04 The headers every request carries

Auth signs the request; a fixed set of headers makes it a Web API request. Each call starts from DEFAULT_HEADERS — the OData v4 contract:headers

OData-Version: 4.0
The protocol the response must speak.
OData-MaxVersion: 4.0
The highest version the client will accept.
Accept: application/json
Give me JSON, not XML/Atom.
Content-Type: application/json; charset=utf-8
My request body (on POST/PATCH) is JSON.

Per-request headers layer on top of these — the Prefer header (annotations, page size), impersonation (MSCRMCallerID), and concurrency (If-Match). Those belong to the next two lessons; here, just see the base. Once auth + headers are set, the call fires through self._session.request(...) — and the retry/paging wrapper around that line is L04's story.

✓
Where annotations come from (preview): the Prefer: odata.include-annotations header set here is exactly what makes Dataverse return human-readable values — the enrichment story is L03.

05 Do it yourself ~6 min

A source-reading drill plus one read-only check. Open crm/utils/d365_backend.py on GitHub (or your clone):

  1. Find _make_auth. Trace all three branches. For your agent-cloud profile (auth_scheme: oauth), which branch runs — and which secret does it consume?
  2. Find DEFAULT_HEADERS and the api_base property. Confirm with your own eyes why an on-prem org's URL ends in /v9.1/ while cloud ends in /v9.2/.
  3. Find where Authorization: Bearer … is set (_OAuthBearerAuth.__call__). Note that it runs per request, not once at startup.
  4. Now, read-only, inspect your own profiles and predict the branch each takes:
crm --json profile list   # read each profile's auth_scheme → which _make_auth branch?
✓
The habit this builds: "is this an auth problem?" stops being a guess. You check the profile's auth_scheme, jump to that one branch in _make_auth, and know exactly what's on the wire.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · Your profile points at orgexample.crm.dynamics.com. Which _make_auth branch runs?
Correct. A *.dynamics.com URL infers auth_scheme: oauth (M01 L02), so the OAuth branch builds a bearer-token adapter via msal.
Not quite. Cloud (*.dynamics.com) infers oauth; _make_auth takes the oauth branch and builds a bearer-token adapter via msal. NTLM/Negotiate are the on-prem branches.
2 · Where does the cloud (OAuth) access token live between crm invocations?
Correct. msal persists a serialized token cache (mode 0600) so the token survives across runs; AAD is only contacted when no valid token is cached.
Not quite. msal serves the token from a persistent cache at ~/.crm/msal_token_cache.json (0600). It isn't in the profile, and it isn't re-fetched every request when a valid one is cached.
3 · What do DEFAULT_HEADERS mainly encode?
Correct. OData-Version/OData-MaxVersion: 4.0 + Accept/Content-Type: application/json declare "I speak OData v4 in JSON." Auth, query params, and retry live elsewhere.
Not quite. The defaults are the OData v4 contract (version headers + JSON accept/content-type). Credentials come from the auth adapter; query params are built in the core layer.
4 · On-prem NTLM and cloud OAuth differ in what they put on the wire. Which is true?
Correct. NTLM negotiates a challenge/response per connection (no token stored); OAuth stamps an Authorization: Bearer <token> header sourced from the msal cache.
Re-check §3: NTLM uses a Windows challenge/response handshake (DOMAIN\user + hashed challenge, nothing cached); OAuth sends an Authorization: Bearer token header.
