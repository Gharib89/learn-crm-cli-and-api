Milestone 12 · Capstone · Lesson 04

The logic layer,
and the walls

Register a plug-in step on Opportunity Update — assembly first, then the step bound to it. And learn the harder skill: spotting the four things a Deal Desk will ask for that the CLI cannot build, and handing them back cleanly.

live on agent-cloud  ·  writes detect & escalate
Why this matters: Real requirements are never 100% automatable. The mission asks you to judge what is automatable and what is not — and a senior agent's value is as much in the clean "this part is GUI-only, here's why" as in the build. This lesson does both: the plug-in step that is automatable, and the four walls you must detect instead of fake.

A plug-in is two things, not one: code (a signed .NET assembly you compile elsewhere) and registration (telling Dataverse when to run it). The CLI owns the second. It can't write your C# — so the assembly is a prerequisite, not a step.

01 The runbook

Prerequisite: a built, signed DealDesk.Plugins.dll (authored in Visual Studio / dotnet — outside crm's scope). With that in hand, registration is two ordered writes.

GOAL A plug-in step firing on opportunity Update, registered inside DealDesk.

target: agent-cloud · prerequisite: a signed assembly DLL on disk

  1. 1. Register the assembly. Upload the DLL, sandboxed. The path is validated — a missing file fails fast, before any write.

    step 1 · upload assemblycrm plugin register-assembly ./DealDesk.Plugins.dll \
  --isolation-mode sandbox --solution DealDesk

    ✓ verify: crm plugin list-types --assembly DealDesk.Plugins lists your plug-in type(s).

  2. 2. Register the step. Bind a plug-in type to the Update message on opportunity. postoperation + async suits "after the deal is saved, create a review" without blocking the user.

    step 2 · register stepcrm plugin register-step --message Update --entity opportunity \
  --plugin-type "DealDesk.Plugins.DealReviewCreator" \
  --stage postoperation --mode async \
  --filtering-attributes estimatedvalue,statuscode \
  --solution DealDesk

    ✓ verify: the result returns a sdkmessageprocessingstepid; the step shows under the plug-in type.

  3. 3. Confirm it's enabled. A registered step defaults to enabled; confirm, and know how to pause it without unregistering.

    step 3 · state checkcrm plugin set-step-state <step-id> --enable # or --disable to pause

    ✓ verify: updating an Opportunity's estimatedvalue now triggers the step (watch crm async for the async job).

▸ 0/3 steps checked

WHY ASSEMBLY FIRST
register-step --plugin-type resolves the type out of a registered assembly. Run step 2 before step 1 and there's no type to bind to — the same dependency-ordering rule as the schema chain, one layer up. --filtering-attributes keeps the step from firing on every unrelated field change.

02 The four walls

Now the harder skill. A stakeholder describing a Deal Desk will casually name four things the crm CLI / Web API cannot create in v1.0.0. Your job is to recognize them on sight and not waste commands trying:

The askThe wallHow you know
Rollup column
"sum of open deal value on the account"		No rollup/calculated column typemetadata add-attribute --kind offers string…lookup,customer,image,file — no rollup, no calculated.
Approval BPF
"a guided stage bar"		Business process flow = workflow category 4The workflow group only lists / activates / imports — there is no workflow create.
Power Automate flow
"email the manager on submit"		Modern flow = category 5Same: authored in the Power Automate maker portal, not creatable through the CLI.
Dashboard / chart
"a deals overview screen"		No verb exists at allThere is no dashboard or chart command group in crm describe.
PROVE IT, DON'T GUESS IT
The agent doesn't assume a wall — it confirms one. crm describe is the source of truth for what verbs exist; crm metadata add-attribute --help shows the exact --kind choices. When a kind or a verb isn't there, that's your evidence the ask is GUI-only — not a memory, a check.

03 How to hand a wall back

Detecting the wall is half the skill; escalating it well is the other half. A good hand-back has three parts — and never a silent skip or a fake substitute:

  1. Name it precisely. "The 'total deal value' field is a rollup column — not creatable via the CLI in v1.0.0."
  2. Say why. "add-attribute has no rollup kind; rollups recalc via async system jobs, capped 10 per table / 100 per org." 2
  3. Give the manual path. "Create it in the maker portal → table → New → Rollup column, then I'll fold it into the DealDesk solution and it promotes with everything else."
NEVER FAKE
The worst move is approximating a rollup with a plug-in that writes a number, or quietly dropping the BPF and reporting "done". Both hide a gap the stakeholder is counting on. A capstone is "done" only when the automatable part is built and the manual remainder is named — that honesty is the deliverable.

Lesson check

4 questions · click to reveal

1. Why must register-assembly run before register-step?

Correct. The step binds to a plug-in type, and the type only exists once its assembly is registered. No assembly → nothing to bind → no step. Same ordering discipline as the schema chain.
The step binds to a type living inside a registered assembly, so the assembly must come first. It's not about publish speed, a reverse dependency, or separate solutions.

2. A stakeholder asks for a rollup column. The honest, correct response is to…

Correct. There's no rollup kind, so detect-and-escalate: name it, explain (async recalc, 10/table·100/org), hand over the maker-portal step, then fold it into the solution. Faking or silently dropping it both hide the gap.
Name it, justify it, hand it back. There's no --kind rollup; a plug-in substitute or a silent omission misrepresents what was delivered.

3. How does the agent confirm the BPF is a wall rather than assume it?

Correct. The verb surface is the evidence: the workflow group lists/activates/imports only — no create. Proving the wall from describe beats memory and beats trial-and-error against the live org.
Check the verb surface — describe shows the workflow group has no create. That's confirmation, not memory, and you don't need to crash a live attempt to learn it.

4. Which stage/mode fits "after the deal is saved, create a review, without blocking the user"?

Correct. You act after the save (postoperation) and off the user's thread (async) — the review is created in the background. Pre-stages run before the write; sync would make the user wait.
Post-save work that shouldn't block is postoperation + async. Pre-stages fire before the record is written, and sync holds up the user.
