How tables
relate

Before you wire two tables together, get the model: there are only two real relationship types — and a lookup column is one of them.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. M04 built standalone tables. Real data isn't standalone — a project belongs to an account, a skill applies to many projects. In Dataverse those connections aren't foreign keys you bolt on after; they're relationships, first-class metadata with their own create verbs. M03 L06 had you read the wiring; this milestone has you build it. Start with the mental model — get "a lookup is a 1:N" and everything else this milestone clicks.

Today's win: explain 1:N, N:1 and N:N to a colleague, say which side the lookup lives on — and read any table's relationships with one command.

01 Two types, three names

Power Apps shows you three relationship choices, but there are really only two kinds. N:1 is not a third thing — it's a 1:N looked at from the other end.types

TypeMeansSchool example
1:N one-to-manyOne parent row → many child rowsOne account has many projects
N:1 many-to-oneThe same relationship, seen from the childEach project belongs to one account
N:N many-to-manyMany ↔ many, both directionsA project uses many skills; a skill serves many projects
✓
1:N and N:1 are one relationship, two viewpoints. The relationship exists between the tables. Microsoft's own docs say it plainly: "A N:1 relationship is just a 1:N relationship viewed from the related table."types So there are two real types to create — 1:N and N:N — which is exactly why the CLI has two create verbs.

02 A lookup column is a 1:N relationship

This is the one idea to walk away with. When you put a lookup column on a table, Dataverse creates a 1:N relationship under it automatically — the column is just the visible half. The lookup always lives on the N side (the child), and it stores the GUID of one parent row.lookup

account — 1:N —▶ project
  (referenced / "1" / parent)      (referencing / "N" / child)

the lookup column ag_accountid lives on project → points at one account

Two pieces of vocabulary that the CLI uses everywhere — learn them now:

referenced entity
the "1" / parent side — e.g. account. Holds nothing extra.
referencing entity
the "N" / child side — e.g. ag_project. Gets the lookup column.
referencing attribute
the lookup column itself — e.g. ag_accountid — storing the parent's GUID.
!
The lookup goes on the child, not the parent. Beginners expect "add accounts to a project" to put something on account. It doesn't — the ag_accountid column lands on project, the many-side. One account, referenced by many project rows. (Microsoft's maker UI even warns: in a one-to-many, the lookup is created on the related table.)lookup

03 N:N needs a hidden third table

A many-to-many can't be stored as a single column — where would you put "many" GUIDs? Dataverse solves it with an intersect table (a "relationship table"): a hidden join table whose rows are just pairs of GUIDs, one per link.types

project ◀— ag_project_skill —▶ skill
               the intersect table — pairs of (projectid, skillid), no other data

You never put business columns on the intersect table — it's pure plumbing. You don't even create rows in it directly; you associate two records and Dataverse writes the pair for you (L05). The takeaway: 1:N = a lookup column; N:N = an intersect table.

04 Read the wiring: metadata relationships

You met this in M03 L06. Now it's the verb you'll lean on all milestone — to confirm a relationship landed and to find the exact names you need. It returns three arrays, mirroring the model above:relationships

agent-cloud · exit 0crm --json metadata relationships ag_project

{ "ok": true, "data": {
    "OneToMany":  [ # where ag_project is the "1"/parent (system rels: notes, tasks…) ],
    "ManyToOne":  [ # where ag_project is the "N"/child — your account lookup shows up here
      { "SchemaName": "ag_account_ag_project",
        "ReferencedEntity":  "account",        # the "1" side
        "ReferencingEntity": "ag_project",     # the "N" side
        "ReferencingAttribute": "ag_accountid", # the lookup column
        "ReferencingEntityNavigationPropertyName": "ag_AccountId" }
    ],
    "ManyToMany": [ # intersect-table relationships, both directions equal ]
  } }

Each entry hands you everything later lessons need: SchemaName (to update or delete the relationship), ReferencingAttribute (the lookup column you'll set), and the NavigationPropertyName (the name you bind to when creating a child row — L04). One read, all the names.

✓
Why both OneToMany and ManyToOne arrays? Because the same relationship appears under different headings depending on which side this table plays. Ask account and your project relationship is in its OneToMany; ask ag_project and it's in ManyToOne. Same relationship — two viewpoints, exactly as §01 said.

05 Do it yourself ~5 min

All read-only — nothing is created. You're training your eye to spot the model on a real table:

# 1 · read the account's relationships — it's the hub of the org
crm --json metadata relationships account

# 2 · pick one OneToMany entry and read its three key fields
#     ReferencedEntity / ReferencingEntity / ReferencingAttribute

# 3 · a built-in N:N: read the lead's relationships, find a ManyToMany entry
crm --json metadata relationships lead
  1. In query 1, find contact in the account's OneToMany array. Which table holds the lookup column — account or contact? (The ReferencingEntity answers it.)
  2. For that same entry, what is the ReferencingAttribute? That's the actual lookup column name on the child.
  3. In query 3, an entry in ManyToMany has no single "referencing attribute" — why not? (Where do the links live instead?)
  4. Out loud: a customer says "let me tag each project with several skills." 1:N or N:N? Which physical structure does Dataverse build?

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You add a lookup column ag_accountid so each project points to one account. On which table does that column physically land?
Correct. A lookup lives on the referencing (N/child) entity and stores one parent GUID. The "1" side (account) gets nothing new — it's referenced, not modified.
The lookup always sits on the N/child side — here ag_project. It holds the parent account's GUID. The "1" side stays untouched.
2 · True or false: N:1 (many-to-one) is a distinct relationship type you create separately.
Correct. There are only two real types: 1:N and N:N. "N:1" is the same 1:N relationship seen from the related table — which is why create-one-to-many covers both viewpoints.
N:1 isn't a separate thing — it's a 1:N looked at from the child. Two real types exist (1:N, N:N), hence two create verbs.
3 · A project can use many skills and a skill applies to many projects. What does Dataverse physically build?
Correct. N:N can't fit in one column, so Dataverse creates an intersect table — each row is a (projectid, skillid) pair, with no other data.
Many-to-many needs an intersect table: a hidden join table of GUID pairs. A single lookup column can only hold one parent, so it can't express N:N.
4 · You run metadata relationships ag_project and your account lookup appears under ManyToOne. Run it on account — where does the same relationship appear?
Correct. The relationship is shared by both tables. From account (the parent) it's OneToMany; from project (the child) it's ManyToOne. One relationship, two viewpoints.
It's the same relationship seen from the other end. Account is the "1"/parent, so on account it lists under OneToMany.
