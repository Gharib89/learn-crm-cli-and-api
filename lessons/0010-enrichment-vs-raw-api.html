Enrichment vs
the Raw API

Which values come straight from Dataverse — and which (almost none) does crm compute itself?

source  github.com/Gharib89/crm  ·  @ main
Why now. A natural assumption: when crm shows "Active" instead of 1, the tool must have looked it up. It didn't. L02 set the Prefer header; this lesson follows the response back up and draws the line — Dataverse decorates the data, crm mostly just passes it through. Knowing that line is the difference between "crm is hiding the real value" (wrong) and "I didn't ask for annotations" (right).

Today's win: look at any crm row and say, field by field, whether the value came from the API or from crm.

01 The annotation switch

Remember the Prefer header from L02? The --annotations flag is what sets it. In the core query builder:prefer

core/query.pyif include_annotations:
    headers["Prefer"] = 'odata.include-annotations="*"'

That single header asks Dataverse to decorate each row with extra, @-keyed sibling fields. For query odata the default is --no-annotations (you opt in); saved & user queries turn it on by default.flag Without it, a row is just raw values and lookup GUIDs. crm only asks — Dataverse does the decorating. (Note: crm v4.7.0 also stops echoing the OData control annotations @odata.context/@odata.etag in its output, so the only @-keys you'll now see are enrichment ones.)

02 Anatomy of an annotated row

One contact row, fetched --annotations. Watch the colours: amber = raw field, green = the server added it:

crm --json query odata contacts --annotations{
  "fullname": "Dana Khan",                       ← raw value
  "statuscode": 1,                              ← raw option-set int
  "statuscode@OData...FormattedValue": "Active",  ← API label
  "_parentcustomerid_value": "a1b2…-guid",         ← raw lookup GUID
  "_parentcustomerid_value@...lookuplogicalname": "account"  ← API: lookup's table
}

The label "Active" and the lookup's table "account" are sibling fields the server attached — not values crm derived. Strip the Prefer header (--no-annotations) and the green lines simply vanish; you're left with statuscode: 1 and a bare GUID.

✓
Naming it: @OData.Community.Display.V1.FormattedValue = the human label for a typed value (option set, money, date). @Microsoft.Dynamics.CRM.lookuplogicalname = which table a lookup points at. Both are Dataverse annotations, defined by the Web API spec.

03 What crm does not do (the surprising part)

Three things people expect crm to do for records — and it doesn't:

  • It doesn't translate option sets itself. No local metadata lookup turns 1 → "Active" for a record. That label is only the API's FormattedValue — so no --annotations, no label.
  • It doesn't fetch a lookup's name. You get the GUID and the target table (account) — never "Contoso Ltd." To see the related record's name, you --expand it or run a second query. crm makes no hidden secondary call.
  • It doesn't format types. Dates, money, booleans render as whatever Dataverse returned — no locale, currency symbol, or true/false prettifying.
!
So a bare statuscode: 1 and a raw GUID are not bugs. They mean "no annotations requested" — exactly what you'd file wrongly as a defect if you didn't know this line. The fix is a flag, not an issue.

04 What crm does add — and it's mostly subtractive

The tool's own contribution to record output is small. --minimal removes the annotations for a clean JSON shape — one dict-comprehension:prune

def _prune_annotations(record):
    # drop every key containing '@'; keep business fields, _*_value GUIDs, the id
    return {k: v for k, v in record.items() if "@" not in k}

And in human (table) mode, crm picks columns and truncates — it does not add data:cols

  • Columns: _infer_columns scans the first 5 rows, skips every @-key, takes the first 8 fields. Headers are the raw logical names — not display names.
  • Cells: each value is str()-ed and cut to 40 chars. No type awareness.

The one-line model: crm is a faithful pass-through. The human-readable magic is Dataverse's annotations (you toggle them with --annotations); crm's own work is selecting, pruning, and truncating — never inventing values.

05 Do it yourself ~6 min

A contrast drill — run the same query three ways and watch the row change. All read-only:

# 1 · raw — statuscode is a bare int, lookups are bare GUIDs
crm --json query odata contacts --select fullname,statuscode --top 1 --no-annotations

# 2 · annotated — the @FormattedValue / @lookuplogicalname siblings appear
crm --json query odata contacts --select fullname,statuscode --top 1 --annotations

# 3 · minimal — annotations stripped back out for a clean shape
crm --json query odata contacts --select fullname,statuscode --top 1 --annotations --minimal

Then, to prove where the magic lives:

  1. Diff run 1 vs run 2. Every new key contains an @ — and it was added by Dataverse, not crm.
  2. In run 2, where did "Active" come from — a crm metadata lookup, or a field on the response? (Trace it back to §2.)
  3. Read _prune_annotations and _infer_columns in commands/_helpers.py. Confirm crm only drops and selects keys — it never writes a label.
✓
The habit this builds: when an output looks "wrong," your first question is now "raw or enriched — did I ask for annotations?" — not "is crm broken?" That alone filters out a whole class of false bug reports.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · A row shows statuscode: 1 and statuscode@…FormattedValue: "Active". Where did "Active" come from?
Correct. The label is a sibling field the Web API attached because --annotations set the Prefer header. crm passes it through unchanged.
Not quite. crm doesn't resolve option-set labels itself. "Active" is the API's FormattedValue annotation, returned because annotations were requested.
2 · You query with --annotations but no --expand. For a lookup field, what do you get?
Correct. You get _x_value (GUID) and its lookuplogicalname (the table). The display name needs --expand or a second query — crm makes no hidden call.
Not quite. Without --expand you get the GUID plus the lookup's target table (lookuplogicalname), not the related record's name.
3 · What does the --minimal flag do to JSON output?
Correct. _prune_annotations keeps business fields, _*_value GUIDs, and the id, dropping every @-key — crm's contribution here is purely subtractive.
Not quite. --minimal removes the @-keyed annotations for a clean shape. It never adds or formats data.
4 · True or false: in table mode, crm converts a money field into a currency-formatted string.
Correct. The table renderer infers columns and truncates to 40 chars; there's no type-aware date/money/boolean formatting. A formatted money value would have to come from the API's FormattedValue.
It's false. crm does no type formatting — cells are stringified and truncated. Any formatted money string would be the API's FormattedValue, not crm's doing.
