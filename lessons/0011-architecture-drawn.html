01 What's in the box

Install crm and you get one Python package, crm/, with a handful of sub-packages — each a single, separable job. This is the whole map at the directory level:layout

The pairing is the thing to memorise: for any group — entity, query, solution — there's a matched pair of files. commands/entity.py (talks to your terminal) sits opposite core/entity.py (talks to Dataverse). Same name, opposite jobs.

02 The one rule: arrows point one way

The reason you can predict where a bug lives is a single architectural rule: dependencies flow in one direction only. Each layer calls the one below it and never reaches back up or skips ahead:dispatch

Two invariants enforce that shape, and you can prove both with a grep (you'll do it in §4):

  • Core never imports Click. The business logic has no idea it's being driven by a CLI — which is why it's unit-testable without spawning a terminal. grep "import click" crm/core/ returns nothing.
  • All Dataverse traffic goes through the transport. Only utils/d365_backend.py builds the authenticated requests session that talks to your org. A command file never opens a socket.
!
The one honest exception. core/update.py does call requests.get() directly — but to crm's own release host to check for and download a new version, not to Dataverse. It can't route that through a Dataverse backend (different host, no auth profile), so it's deliberately separate. The layering rule is about Dataverse I/O; self-update is a different axis.update

04 Prove the invariant yourself ~5 min

Diagrams are claims. Here's how to check the load-bearing one — that core never depends on the CLI framework — against the real tree. Clone or open the repo and run:

verify the invariant# 1 · core must NOT import Click — expect zero hits
grep -rl "import click" crm/core/

# 2 · commands DO import Click — expect many files
grep -rl "import click" crm/commands/*.py | wc -l

# 3 · who builds the requests session? — the transport
grep -rn "requests.Session" crm/utils/d365_backend.py

Expected: (1) prints nothing — the invariant holds; (2) prints a count near 29 — every group is a Click command; (3) finds the one session that all Dataverse traffic shares. You just confirmed Figure 2 is true, not decorative.

✓
The habit this builds: an architecture diagram is only worth trusting if you can falsify it with a command. When you read any codebase's docs, ask "what one grep would prove this?" — then run it.

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · In Figure 1, what is the relationship between commands/entity.py and core/entity.py?
Correct. Each group is a pair with the same name and opposite jobs: the command file parses flags and renders; the core file builds the OData request. The arrow always goes command → core.
Look again at §1: they're a matched pair with opposite jobs — commands/entity.py handles the terminal, core/entity.py builds the Dataverse request. Command calls core, never the reverse.
2 · Which grep would prove the load-bearing invariant from Figure 2, and what result confirms it?
Correct. Core having no Click import is what keeps logic independent of the CLI. (Note A is a trap: core/update.py imports requests for the self-updater, so that grep is not empty.)
It's C. The invariant is "core never imports Click," so an empty result for grep "import click" crm/core/ confirms it. A is a trap — core/update.py imports requests for self-update, so that one isn't empty.
3 · In Figure 3, at which numbered step does any output first reach you?
Correct. Everything before the last hop is internal; the single place output is produced — envelope and exit code — is CLIContext.emit(). One request, one exit.
Re-read Figure 3: data climbs back up the chain silently. The only place it becomes your output is the last step, emit() — the single chokepoint for the envelope and exit code.
4 · core/update.py calls requests.get() directly. Does that break the layering rule?
Correct. The invariant is specifically about Dataverse traffic. The self-updater fetches crm's own new version from the release host — a separate concern that can't use a Dataverse backend, so it's intentionally outside the layering.
It's D. The layering rule is about Dataverse traffic, all of which goes through d365_backend.py. The self-updater hits crm's release host — a different concern, deliberately separate.
