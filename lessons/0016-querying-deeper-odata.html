Deeper

Filter whole sets on the values you now know are really there — option-set codes and lookup GUIDs — then order, count, and page.

Why now. L06 gave you query odata with a simple text filter. L01–L02 taught you what records actually store: gendercode: 1, a lookup as _ownerid_value. Put them together and you can ask the org real questions — "all male contacts," "everything I own" — by filtering on the stored codes and GUIDs, not the labels. This is the query half of the milestone, done properly.

Today's win: filter contacts on an option-set code and on a lookup GUID, then shape the result with --orderby, --count, and --page-size.

01 You filter on stored values

One shape change to note first: in this CLI version, query odata returns the rows as data itself — a bare array, with meta carrying entity_set and (on request) count. No value wrapper:

agent-cloud · exit 0crm --json query odata contacts --top 2 --select fullname,jobtitle

{ "ok": true,
  "data": [
    { "fullname": "Sara Mitchell", "jobtitle": "Sales Manager", "contactid": "a1b2…b91" },
    { "fullname": "James Okafor", "jobtitle": "IT Director", "contactid": "b2c3…b92" }
  ],
  "meta": { "entity_set": "contacts" } }

The big idea: a --filter is written in the stored vocabulary from L01 — codes and _x_value GUIDs — never the friendly labels the form shows you.query

02 Filter on codes and lookups

An option-set, by its code

"All male contacts." Gender is an option-set, so you filter on the code 1 — exactly what L01 showed is stored — not 'Male':

agent-cloud · exit 0crm --json query odata contacts --filter "gendercode eq 1" --select fullname,gendercode --orderby "fullname asc" --count

{ "ok": true, "data": [
    { "fullname": "James Okafor", "gendercode": 1, "contactid": "b2c3…b92" },
    { "fullname": "Robert Chen", "gendercode": 1, "contactid": "d4e5…b94" }
  ], "meta": { "entity_set": "contacts", "count": 2 } }

--count adds the matched total to meta.count (exact and filtered — unlike query count's cached whole-table estimate from L06). --orderby "fullname asc" sorts; desc for the other direction.

A lookup, by its GUID

"Everything owned by this user." A lookup is filtered on its _x_value GUID — again, the read-only field from L01. Combine with whoami's UserId for the classic "my records":

agent-cloud · exit 0crm --json query odata contacts --filter "_ownerid_value eq 99887766-aaaa-bbbb-cccc-1234567890ab" --select fullname,jobtitle --top 3

The operator kit

NeedOData --filter
equals / notgendercode eq 1 · statecode ne 1
comparecreatedon gt 2026-01-01T00:00:00Z · lt, ge, le
text contains / startscontains(fullname,'Mitch') · startswith(jobtitle,'Sales')
combinegendercode eq 1 and statecode eq 0 · … or …
empty / setjobtitle eq null · _parentcustomerid_value ne null

String literals use single quotes ('Sales'); codes, GUIDs, dates, and null are bare. Wrap the whole expression in shell double-quotes.filter

03 Big result sets: --page-size

Dataverse never returns everything at once — a single page maxes out at 5,000 rows. Ask for a page with --page-size; if more rows exist, meta.next_link hands you the URL for the next page (an @odata.nextLink carrying a paging cookie):page

agent-cloud · exit 0crm --json query odata contacts --page-size 2 --select fullname

{ "ok": true, "data": [ /* 2 rows */ ],
  "meta": { "entity_set": "contacts",
    "next_link": "https://orgexample.crm.dynamics.com/api/data/v9.2/contacts?$select=fullname&$skiptoken=…" } }
✓
--top vs --page-size. --top N = "I want at most N rows, full stop." --page-size N = "give me N at a time, and tell me if there are more" (next_link). Use --top for a sample; use --page-size when you intend to walk the whole set. No next_link in meta = you're on the last page.

04 Do it yourself ~4 min

All read-only. Build each query from a real question:

# 1 · all FEMALE contacts (gendercode 2), alphabetical, with an exact count
crm --json query odata contacts --filter "gendercode eq 2" --select fullname,gendercode --orderby "fullname asc" --count

# 2 · who do YOU own? grab your UserId, then filter the lookup on it
crm --json connection whoami            # copy UserId
crm --json query odata contacts --filter "_ownerid_value eq <your-UserId>" --select fullname

# 3 · page two at a time — look for meta.next_link
crm --json query odata contacts --page-size 2 --select fullname
  1. In query 1, what's meta.count? Now drop --filter and re-run — how does the count change, and what does that tell you about --count (filtered vs whole-table)?
  2. Query 2: would --filter "ownerid eq 'Your Name'" work instead? Why not? (Recall L01: what's stored.)
  3. Query 3: is there a meta.next_link? If the org has only a handful of contacts, why might it be absent?

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You want every active contact. statecode shows "Active" on the form but is an option-set. The filter?
Correct. You filter on the stored code (active = statecode 0), never the label. "Active" only ever exists as a FormattedValue annotation — it's not in the column, so it can't be matched.
Re-read L01. The column stores the code, and "Active" is just a label. Filter on statecode eq 0 — quoting the label matches nothing (or errors).
2 · A query odata succeeds with matches. In this CLI, where are the returned rows?
Correct. This CLI version unwraps the rows straight into data as an array — no value wrapper. Count and entity-set live in meta.
Not here. query odata returns data as the array of rows directly; meta carries entity_set and (with --count) count.
3 · Show only the contacts owned by the user with id 99887766-…ab. The filter?
Correct. A lookup is filtered on its _x_value GUID — the read-only key you read in L01. No quotes (a GUID is bare), and never by the owner's name.
A lookup filters on the _<name>_value GUID: _ownerid_value eq <guid>. Not the display name, not a bare owner, and lookups absolutely can be filtered.
4 · You run --page-size 2, get 2 rows, and suspect more exist. What signals another page, and what's the per-page ceiling?
Correct. When more rows remain, meta.next_link carries the URL (with a paging cookie) for the next page. A single page returns at most 5,000 rows; no next_link means you're on the last page.
It's meta.next_link — the @odata.nextLink with a skiptoken cookie — that tells you more pages exist, and Dataverse caps a page at 5,000 rows (not 50/100).
