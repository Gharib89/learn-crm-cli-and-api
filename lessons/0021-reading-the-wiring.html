Reading
the wiring

Tables don't stand alone — they're wired together by relationships. Read that wiring and you'll find the exact navigation property you bound to back in M02.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. In M02 L02 you set a contact's parent account by binding parentcustomerid_account@odata.bind — a string you were handed. Where did it come from? The relationship metadata. This lesson is where that name lives, and reading it is the difference between guessing a bind and looking it up. It's also half of this milestone's "done when": describe a table's relationships.

Today's win: read a table's three kinds of relationship, and tell the "one" side from the "many" side — including the nav property a lookup write needs.

01 Three kinds of link

Dataverse wires tables three ways. metadata relationships <logical> returns all of them, grouped:relmeta

OneToMany (1:N)
This table is the parent ("one") — other rows point at it. One account → many contacts.
ManyToOne (N:1)
This table is the child ("many") — it holds a lookup pointing out. Many contacts → one account. This is where lookup columns live.
ManyToMany (N:N)
A symmetric peer link through a hidden intersect table — no lookup column on either side. Many contacts ↔ many marketing lists.

1:N and N:1 are the same physical link seen from opposite ends — the child stores one foreign key; the parent sees the reverse. N:N is genuinely different: neither side stores anything, a join table in the middle does.

02 Referenced vs Referencing — read the direction

Every 1:N / N:1 entry names two roles. Get these two words right and the rest follows:

ReferencedEntity
The "one" side — the parent being pointed at.
ReferencingEntity
The "many" side — the child that physically stores the foreign key.
ReferencingAttribute
The actual lookup column on the child (e.g. parentcustomerid).
ReferencingEntity­NavigationPropertyName
The bind target — the name you put before @odata.bind when writing the lookup.

Here's contact's N:1 link to account — the one behind M02 L02's write:

agent-cloud · exit 0crm --json metadata relationships contact

# ... inside "ManyToOne": [ ... ]
{ "SchemaName": "contact_customer_accounts",
  "ReferencedEntity":  "account",        # the "one" / parent
  "ReferencingEntity": "contact",        # the "many" / child (holds the FK)
  "ReferencingAttribute": "parentcustomerid",            # the lookup column
  "ReferencingEntityNavigationPropertyName": "parentcustomerid_account" }
                                       # ↑ exactly the bind you used in M02 L02
✓
This is the bind lookup, completed. To set the lookup you write "parentcustomerid_account@odata.bind": "/accounts(<id>)". The nav-property prefix is ReferencingEntityNavigationPropertyName; the /accounts set comes from the ReferencedEntity's EntitySetName (L01). Two metadata facts, one valid write.

03 The polymorphic twist & N:N

Scan contact's ManyToOne and you'll spot parentcustomerid twice — once referencing account, once referencing contact:

ReferencingAttributeReferencedEntityNav property
parentcustomeridaccountparentcustomerid_account
parentcustomeridcontactparentcustomerid_contact

That's a polymorphic (customer) lookup — one column that can point at either an account or a contact. That's why the nav property is suffixed with the target table: it disambiguates which kind you're binding. Pick the suffix that matches the record you're linking.

N:N is its own group, and it reads differently — no Referenced/Referencing, because neither side owns a key:

agent-cloud · exit 0# ... inside "ManyToMany": [ ... ]
{ "SchemaName": "contact_subscription_association",
  "Entity1LogicalName": "subscription",
  "Entity2LogicalName": "contact",
  "IntersectEntityName": "subscriptionmanuallytrackedobject" }

Two symmetric peers (Entity1 / Entity2) joined by an intersect table that stores the pairs. You don't set an N:N with a column write — you associate / disassociate the two records (a verb you saw listed in M02). Reading it here is enough for now.

04 Do it yourself ~4 min

All read-only. Trace the wiring both ways:

# 1 · every relationship on contact — three groups
crm --json metadata relationships contact

# 2 · the SAME account↔contact link from the account's end
crm --json metadata relationships account
  1. In query 1's ManyToOne, find the link to account. What's its ReferencingEntityNavigationPropertyName? Compare it to the bind you wrote in M02 L02.
  2. In query 2's OneToMany, find the contacts link. Now account is the ReferencedEntity and contact is the ReferencingEntity — same physical link, opposite roles. Confirm it.
  3. For the account→contact 1:N, which side stores the foreign key — account or contact? (Whoever is Referencing.)
  4. In query 1's ManyToMany, pick any entry. What's its IntersectEntityName, and why is there no ReferencingAttribute?

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · In a 1:N / N:1 relationship, which entity physically stores the foreign key?
Correct. The Referencing (many/child) entity holds the lookup column. The Referenced (one/parent) only sees the reverse navigation. An intersect table is the N:N case, not 1:N.
The child — the ReferencingEntity — owns the foreign-key column. The parent (Referenced) is just pointed at. Intersect tables belong to N:N relationships.
2 · You want to set a contact's parent account. Which field gives you the @odata.bind name?
Correct. The nav-property name is what precedes @odata.bind (parentcustomerid_account@odata.bind). The ReferencingAttribute is the underlying column; the bind uses the navigation property.
It's ReferencingEntityNavigationPropertyName — the bind target. ReferencingAttribute (parentcustomerid) is the raw column; you bind via the navigation property, not the column name.
3 · parentcustomerid appears twice in ManyToOne — once → account, once → contact. Why?
Correct. A customer lookup is polymorphic: the single parentcustomerid can reference an account or a contact. Each target gets its own nav property (…_account / …_contact) so the bind is unambiguous.
It's a polymorphic lookup — parentcustomerid can point at an account or a contact. The two entries (and their …_account / …_contact nav properties) just spell out the two possible targets.
4 · How does an N:N relationship differ from 1:N in storage?
Correct. N:N uses an intersect (join) table — IntersectEntityName — to record each pair. That's why the entry has symmetric Entity1/Entity2 instead of Referenced/Referencing, and why you link rows with associate, not a column write.
N:N has no FK column on either side. A separate intersect table stores the pairings, which is why the metadata shows symmetric Entity1/Entity2 and you use associate/disassociate to link.
