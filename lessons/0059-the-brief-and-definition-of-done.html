The brief,
and what "done" means

Everything you've learned now meets one real requirement. This lesson is the contract: the business ask, the exact definition of done, the ordered build chain — and the line between what the agent can build and what it must hand back to a human.

capstone runs on agent-cloud  ·  read & plan your turn next
Why this matters: The mission's whole point is this moment — take a real customization requirement and drive it end-to-end with Claude Code + the crm CLI, across schema → solution → UI → logic, then close the feedback loop. Before you touch a single verb, you need a brief precise enough to verify against and a build order that respects the platform's dependencies. Get the brief wrong and every command after it is wasted.

A capstone is not a new skill — it's the proof you can sequence the skills you already have. So we start where every real engagement starts: pin the requirement, define "done", and lay the chain in dependency order before building anything.

01 The requirement: a Deal Desk

Your stakeholder is a sales manager. The ask, in their words:

BRIEF
"When a sizable Opportunity is worked, I want a lightweight deal-review record that tracks an approval status — Pending, Approved, or Rejected. Reviews should hang off the Account. The whole thing has to ship as one package I can move from our dev org into test and then production."

Translated into the D365 platform pieces — the mental model you built across M03–M09 — that brief becomes:

  • A custom table Deal Review (itworx_dealreview).
  • A global option set Approval Status (Pending / Approved / Rejected) and a status column on Deal Review that uses it.
  • A 1:N relationship Account → Deal Review, so each review points at its account via a lookup.
  • A plug-in step on Opportunity Update — the business logic that reacts to the deal changing.
  • A model-driven app (with a sitemap subarea) and a form field so a human can actually see and set the status.
  • All of it inside a solution that exports managed and promotes dev → test → prod.

02 Definition of done

This is the milestone's tracker issue, made concrete. You are done — and not before — when all three hold1:

VERIFY ≠ TRUST
"Delivered" means verified, not "the command exited 0". Every build step in this milestone ends with a read-back that proves the component exists — the same --json envelope discipline from M01, now applied to a whole feature.

03 The chain — in dependency order

The platform forces an order: you can't put a lookup on a table that doesn't exist, can't use an option set before you've created it, can't register a step before its assembly is uploaded. The capstone's spine is this ordered chain — and it maps one lesson per layer:

  1. Publisher + solution — the container everything else is created into. L02
  2. Deal Review table → Approval Status global option set → status column (column needs the option set first). L02
  3. 1:N Account → Deal Review + the lookup column (needs both tables to exist). L03
  4. Plug-in assembly → register step on Opportunity Update (step needs the assembly first). L04
  5. Model-driven app + sitemap subarea → form field → publish once (table is invisible until a subarea points at it). L05
  6. Export managed → promote → learning record → upstream issue. L06
LIFECYCLE
Notice the macro-order: schema → solution-bound → logic → UI → ship. This is the customization lifecycle from M10 L03, now executed for real instead of planned on paper.

04 What the agent can't build

Part of being done is knowing what to not attempt. A stakeholder will casually ask for things the crm CLI / Web API simply cannot create — and the agent's job is to detect and escalate, not fake it. Four traps live inside a Deal Desk:

Tempting askWhy it's GUI-only in crm v1.0.0
Rollup column ("total deal value on the account")metadata add-attribute --kind has no rollup/calculated kind — and rollups recalc via async system jobs, capped 10/entity.2
Approval business process flowA BPF is a workflow category 4; the workflow group only lists/activates/imports — no create verb.
Power Automate flowA modern flow is category 5 — same story, no create path through the CLI.
Dashboard / chartNo CLI verb exists for these components at all.
ESCALATE
When the agent hits one of these, the correct move is a one-line flag: "This part is GUI-only — here's the manual step, and here's why the CLI can't do it." Pretending otherwise is worse than stopping. You separate the automatable chain from the manual remainder first, before you build.

05 How this milestone works

M12 is a runbook, not a reading. From L02 on, each lesson hands you the verbs and you drive Claude Code against agent-cloud. The loop, every step:

  1. Preview the write with --dry-run (read the method/url/body) before it lands.
  2. Commit it, then immediately verify with a read-back — the component must show up.
  3. Paste me the envelope. I check it against the expected shape and we move to the next link in the chain.
SAFE ORG
agent-cloud is a mutate-safe dev org — destructive practice is fine, that's what it's for. Confirm the target with whoami before the first write (M10 L05's rule), and we'll tear the whole Deal Desk down cleanly at the end of L06.

Lesson check

4 questions · click to reveal

1. Why must the Approval Status option set be created before the status column?

Correct. The status column is a picklist that binds to the global option set, so the set has to exist first. That dependency — not a CLI quirk — is what fixes the order.
It's a real data dependency: the picklist column references the option set, so the set must exist first. It's not a blanket rule about tables, publish speed, or alphabetical ordering.

2. A stakeholder asks for a rollup column summing deal value on the account. The agent should…

Correct. There is no rollup kind in add-attribute, so the CLI can't create it. Detect and escalate — name the manual step and the reason. Silently skipping or faking it with a plug-in is worse than stopping.
There's no --kind rollup, so it can't be created via the CLI. The right move is to flag it as GUI-only and explain — not fake it with a plug-in, and never skip silently.

3. Which of these means the capstone is done?

Correct. Those three criteria are the definition of done. A clean exit code or a single visible component is necessary but nowhere near sufficient — done is the whole feature, verified, plus the feedback loop closed.
Done is all three acceptance criteria together: delivered-in-a-solution-and-verified, a learning record, and an upstream issue if one is warranted. A single exit code or component doesn't get you there.

4. In the M12 runbook loop, what comes between committing a write and moving on?

Correct. Preview → commit → verify with a read-back → paste the envelope. Verification is the gate on every step; you never trust a clean exit alone.
You verify with a read-back before moving on. Publishing, switching orgs, and exporting are later steps in the chain — not the per-step gate.
