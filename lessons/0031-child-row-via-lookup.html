at its parent

The relationship exists — now create a project that actually belongs to an account, using the binding syntax Dataverse expects.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. A lookup column is useless until a row fills it in. This is where M05 meets M02: you write records (M02) and set a relationship at the same time. But you can't just put a GUID in the lookup field — the Web API wants a special @odata.bind form. Learn that one syntax and you can connect records on create or update, and read the links back.

Today's win: create a project row bound to a real account, then read the link back two ways — expanded from the child, and counted from the parent.

01 A lookup has three names at runtime

From M03 you know the lookup column's logical name (ag_accountid). But when you read and write records, two more names show up. Pin them down — mixing them up is the #1 lookup mistake:nav props

ag_accountid
the logical name — what metadata shows. You don't set this directly on a record.
ag_AccountId
the navigation property (= the lookup schema name) — what you bind to when writing.
_ag_accountid_value
the read form — how the GUID comes back in query results (underscored, _value suffix).

So: you write through the navigation property with @odata.bind, and you read the GUID out of _ag_accountid_value. The plain logical name is what metadata calls it. Find the navigation property any time with metadata relationships (the ReferencingEntityNavigationPropertyName field, L01).

02 Set the lookup on create: @odata.bind

To make a project that belongs to an account, you don't pass the account GUID as a value — you bind the navigation property to the account's Web API path. The key is <nav>@odata.bind, the value is /<entityset>(<guid>):associate

agent-cloud · exit 0crm --json entity create ag_projects --data '{
    "ag_name": "Apollo Migration",
    "ag_AccountId@odata.bind": "/accounts(e5f6a7b8-5555-…)"
  }'

{ "ok": true, "data": {
    "ag_projectid": "44b3e97b-926a-f111-…",
    "ag_name": "Apollo Migration",
    "_ag_accountid_value": "e5f6a7b8-5555-…"   # ← the link, read form
  } }

The new project came back with _ag_accountid_value populated — it's bound to the account. Three rules that trip people up:

RuleWhy
Use the nav property (ag_AccountId), not ag_accountidThe bind targets the relationship, not the raw column
Value is a path: /accounts(<guid>) — entity set name, in parenthesesIt's a reference to a record, not a string
Names are case-sensitiveag_AccountId@odata.bind ≠ ag_accountid@odata.bind
✓
Dry-run shows the body before you commit. crm --dry-run entity create … --validate prints the exact JSON and runs a field-name check — confirm the @odata.bind key is spelled right before the write lands. (You met --validate in M02.)

03 Set or clear it on an existing row

Already have a project and want to (re)point it? You don't need to hand-write @odata.bind — two dedicated verbs do it. set-lookup binds, clear-lookup unbinds:set-lookup

agent-cloud · exit 0# point project → account (PATCH @odata.bind under the hood)
crm --json entity set-lookup ag_projects 44b3e97b-… \
    ag_AccountId accounts e5f6a7b8-…
#   args: ENTITY_SET  RECORD_ID  NAV  RELATED_SET  RELATED_ID

# later: detach it (DELETE /$ref) — leaves the project, blanks the lookup
crm --json entity clear-lookup ag_projects 44b3e97b-… ag_AccountId

Same navigation property, same idea as the bind — just as standalone operations on a row that already exists. set-lookup is the update twin of the create-time bind in §02.

04 Read the link back — both directions

Prove the relationship works by traversing it. From the child, $expand the navigation property to pull parent fields inline (you met $expand in M02 L03):associate

agent-cloud · exit 0crm --json query odata ag_projects --select ag_name \
    --expand "ag_AccountId(\$select=name)" --top 3

[ { "ag_name": "Apollo Migration",
    "ag_AccountId": { "name": "ITWorx" } } ]   # child → parent, resolved

From the parent, count children per relationship without writing a query — entity children walks every 1:N where the account is the "1" side:children

agent-cloud · exit 0crm --json entity children accounts e5f6a7b8-… \
    --filter-entities "ag_" --non-empty

[ { "entity": "ag_project", "attribute": "ag_accountid",
    "set": "ag_projects", "count": 1 } ]   # parent sees 1 child via that lookup
✓
Two viewpoints again. The same link reads as a parent object when you expand from the child, and as a count when you ask the parent for its children — exactly the "one relationship, two sides" idea from L01, now at the data level instead of the metadata level.

05 Do it yourself ~8 min writes

Needs the L02 relationship (account → ag_project) in place. Creates one linked row, reads it, cleans up.

# 0 · grab a real account GUID to bind to
crm --json query odata accounts --select name --top 1

# 1 · PREVIEW the create + validate the field name (no write)
crm --json --dry-run entity create ag_projects --validate --data \
  '{"ag_name":"Apollo Migration","ag_AccountId@odata.bind":"/accounts(PASTE-GUID)"}'

# 2 · create it for real (drop --dry-run --validate)

# 3 · read the link from the child (expand) and the parent (children)
crm --json query odata ag_projects --select ag_name --expand "ag_AccountId(\$select=name)"
crm --json entity children accounts PASTE-GUID --filter-entities "ag_" --non-empty

# 4 · clean up the row
crm --json entity delete ag_projects --key ag_projectid=PASTE-PROJECT-GUID --yes
  1. In step 2's envelope, find _ag_accountid_value. Why does it come back with that underscored name and not ag_AccountId?
  2. Deliberately break step 1: change the key to ag_accountid@odata.bind (wrong case). What does --validate say?
  3. In step 3's expand, you got the account's name inline. Try adding another field to $select inside the parentheses.
  4. One sentence: when would you use set-lookup instead of binding at create time?

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · To set the account lookup when creating a project, which JSON key do you use?
Correct. You bind the navigation property to the account's Web API path: <nav>@odata.bind = /<entityset>(<guid>). Not the raw GUID, not the read-form name.
Lookups are set by binding the nav property: ag_AccountId@odata.bind with value /accounts(<guid>). The plain logical name and the _value read form don't work for writing.
2 · After creating the linked project, how does the account GUID appear in query results?
Correct. The raw GUID reads back in the underscored _ag_accountid_value form. (To get the account's fields, you'd $expand the nav property instead.)
The lookup's GUID comes back as _ag_accountid_value. You write through the nav property but read the value from the underscored field.
3 · You want the account's name alongside each project in one query. Which clause?
Correct. $expand on the navigation property pulls related fields inline — child → parent in a single round trip. $select alone only gives you the GUID.
Use $expand on the nav property to resolve parent fields inline. --select _value gives only the GUID; you'd otherwise need a second query.
4 · You have an existing project and want to attach it to an account — without rewriting the whole record. Best verb?
Correct. set-lookup PATCHes the single lookup via @odata.bind — the update twin of the create-time bind. clear-lookup is its inverse.
set-lookup sets one lookup on an existing row (PATCH @odata.bind). update-relationship edits metadata, not data; children only reads counts.
