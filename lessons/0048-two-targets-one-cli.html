Milestone 09 · On-prem Differences · Lesson 01

Two targets,
one CLI

On-premises and Dataverse online are the same Web API behind two different front doors. The profile is the switch — and it decides how you authenticate.

live on agent-on-prem  ·  crm.contoso.local/Contoso  ·  reads
Why this matters: The mission needs you to handle the on-prem (NTLM, v9.1) vs cloud (OAuth, v9.2) split correctly. Every command you learned in M01–M08 ran against the cloud org. They run against on-prem unchanged — but the connection underneath is wired differently. This milestone starts where the difference actually lives: not in the commands, but in the profile.

D365 CE ships in two deployment models. On-premises: you (or your IT dept) run the server on a Windows box inside a corporate network. Dataverse online: Microsoft runs it in the cloud. Same OData v4 Web API — but you knock on the door differently.

01 Two deployment models, one Web API

Both targets expose the same Dataverse Web API at <url>/api/data/v9.x/, speaking OData v4.1 That is why the crm CLI is "write once, run on both" — entity create, query odata, metadata entities all issue the identical HTTP request shape to either server.

What differs is everything around the request — how you prove who you are, the transport, and which API version the server will answer:

 On-premisesDataverse online
Who runs itYour org, on a Windows ServerMicrosoft, in the cloud
Identity storeActive Directory (Windows accounts)Microsoft Entra ID
Auth schementlm (Windows Integrated)2oauth (client-credentials)3
Credentialsdomain + username + passwordtenant-id + client-id + secret (app registration)
Transportoften plain http:// (inside VPN)always https://
URL shapehttp://host/OrgNamehttps://org*.crm.dynamics.com
API versionv9.1 maxv9.2
ReachabilityVPN / corporate network onlypublic internet

The CLI infers the scheme from the URL: a *.dynamics.com host → OAuth, anything else → NTLM.2 You don't pick "on-prem mode" — you point a profile at a host, and the rest follows.

KEY
The commands never change. There is no --on-prem flag and no on-prem command set. The same verbs run against both targets; the profile carries the target-specific wiring. Everything in this milestone is about reading and reasoning about that wiring.

02 The profile is the switch

connection status makes **no network call** — it just prints the active profile's resolved config. Run it against each profile and the split is right there in the fields. Here are the two side by side (secrets and IDs masked):

crm --json --profile agent-on-prem connection status{ "ok": true, "data": { "active_profile": "agent-on-prem",
  "profile": {
    "url":          "http://crm.contoso.local/Contoso",   # plain http
    "auth_scheme":  "ntlm",
    "username":     "contoso\\alice",                     # domain-qualified
    "tenant_id":    null,
    "client_id":    null,
    "api_version":  "v9.1"
  } } }
crm --json --profile agent-cloud connection status{ "ok": true, "data": { "active_profile": "agent-cloud",
  "profile": {
    "url":          "https://orgexample.crm.dynamics.com", # https
    "auth_scheme":  "oauth",
    "username":     "",                                  # empty — no user/pass
    "tenant_id":    "727f34ab-0000-0000-0000-5ed673dd203b",
    "client_id":    "4e156fdd-0000-0000-0000-c6844dcaf9ed",
    "api_version":  "v9.2"
  } } }

Five fields tell the whole story:

FieldOn-premCloudWhat it means
auth_schementlmoauthhow the CLI proves identity
urlhttp://…https://…transport + which world
usernamecontoso\aliceemptyNTLM uses a Windows user; OAuth uses an app, not a person
tenant_id / client_idnullpopulatedthe Entra app registration — OAuth-only
api_versionv9.1v9.2the negotiated API ceiling (next lesson)
NOTE
NTLM authenticates a person; OAuth authenticates an app. On-prem, the CLI logs in as a domain user (contoso\alice) — exactly the account a human would use. Cloud uses an application user: a non-human Entra app registration granted a security role. That is why username is empty for the cloud profile — there is no person behind it.

03 Confirm the live target before you mutate

connection status is offline — it reports what the profile claims. To confirm a live server actually answers as that identity, you need a round-trip. Two verbs do it:

shell · WhoAmI round-trip (on-prem)crm --json --profile agent-on-prem connection whoami
envelope · live-verified 2026-06-20 · crm v1.0.0{ "ok": true, "data": {
    "BusinessUnitId": "078a426a-0000-0000-0000-00155d467b90",
    "UserId":         "f890426a-0000-0000-0000-00155d467b90",
    "OrganizationId": "b948cd5f-0000-0000-0000-00155d467b90"
  } }

Notice the result shape is identical to the cloud WhoAmI you saw in M08 — three GUIDs, no scheme-specific fields. The auth difference is entirely below the envelope; once you're connected, on-prem and cloud look the same to your commands. That is the whole promise of "one CLI."

The fuller check is connection test (or the deeper connection doctor, covered in L03), which also reports api_base and api_version — so you can see which host + which version answered, not just who you are:

shell · confirm host + versioncrm --json --profile agent-on-prem connection test
# → api_base: http://crm.contoso.local/Contoso/api/data/v9.1/
# → api_version: v9.1
HABIT
Confirm the target before any write. When you switch between orgs, one stray --profile (or a forgotten profile use) can point a mutation at the wrong world. A one-line connection whoami / connection test first is the cheap insurance — the same discipline you used on cloud, now doubly important when two live orgs are in play.

Lesson check

3 questions · click to reveal

1. You run connection status for a profile and see "auth_scheme": "oauth" with a populated tenant_id and an empty username. Which target is this?

Correct. OAuth client-credentials + a tenant/client id + no username is the cloud signature: the CLI authenticates as an app registration (an application user), not a person. On-prem uses NTLM with a domain-qualified username and null tenant/client ids.
OAuth + populated tenant_id/client_id + empty username is the Dataverse online signature. The CLI authenticates as an application user (an Entra app registration), which is why there is no person's username. On-prem uses ntlm with a domain-qualified username.

2. How does the CLI decide whether to use NTLM or OAuth for a profile?

Correct. The scheme is inferred from the profile's URL: a *.dynamics.com host means OAuth, anything else means NTLM. You can override per-run with --auth-scheme, but you rarely need to — pointing a profile at a host wires the rest.
The CLI infers the scheme from the URL — *.dynamics.com → OAuth, any other host → NTLM. There is no required per-command flag and no try-both fallback (a per-run --auth-scheme override exists but is rarely needed).

3. Why do the same crm commands (e.g. entity create, query odata) work unchanged against both targets?

Correct. On-prem and online are the same Dataverse Web API (OData v4) at /api/data/v9.x/. The request shape is identical; the profile supplies the auth scheme and version. That shared surface is what makes the CLI portable across both.
Both targets expose the identical OData v4 Web API; the request shape is the same. There is no separate on-prem command set and no SOAP translation — the profile supplies auth + version, and the command surface is shared.
