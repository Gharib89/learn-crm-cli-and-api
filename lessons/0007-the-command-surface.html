01 Every command has the same shape

The CLI is a tree. Almost every invocation reads the same way:

crm --json entity get contacts <guid> --select fullname
└─ global ─┘ └ group ┘ └verb┘ └─ arguments ─┘ └─ command flags ─┘ the noun (a domain) the action

A group is a noun — a domain of the platform (entity, query, solution, metadata). A verb is the action within it (get, create, list). Global flags like --json and --dry-run sit before the group and apply to everything; command flags like --select come after and are specific to that verb.docs Once you see the tree, you stop memorizing commands and start locating them.

02 Tool 1 — --help, at any depth

--help works at every level of the tree. Three levels, three questions:

Top level — "what groups exist?"

crm --helpCommands:
  connection  Diagnose the active connection (WhoAmI, reachability, ...).
  entity      Record CRUD against entity sets (accounts, contacts, ...).
  query       Run OData and FetchXML queries.
  metadata    Browse entity / attribute / relationship metadata.
  solution    Solution lifecycle (create-publisher / create / list ...).
  scaffold    Generate D365 metadata from shorthand specs.
  plugin      Register and manage plug-in assemblies, ...
  profile     Create, switch, and manage connection profiles.
  … ~30 groups total — solution, metadata, plugin, view, webresource, workflow, security …

Group level — "what verbs does this group have?"

crm entity --helpCommands:
  create        POST a new record.
  get           GET <entity-set> <guid>.
  update        PATCH an existing record.
  delete        DELETE a record.
  upsert        PATCH with create-if-missing semantics.
  clone         Clone a single record, optionally its children.
  set-lookup    Set a single-valued lookup via @odata.bind PATCH.
  … also: associate, disassociate, children, clear-lookup

And the third level — crm entity create --help — lists that verb's exact arguments and flags (--data, --validate, --no-return…). This is how you answer "how do I do X" without guessing. Notice you just learned, from the help alone, that updating is entity update (a PATCH) and deleting is entity delete — verbs we never covered in a lesson.

03 Tool 2 — crm describe, the machine-readable map

--help is for human eyes. crm describe emits the same tree as structured data — every command, option, and choice — built for an agent (or a script) to consume:describe

crm describe --helpUsage: crm describe [OPTIONS] [GROUP]

  Emit a machine-readable catalogue of all commands, options, and choices.

  With no argument, walks the whole tree. With GROUP, walks only that
  command's subtree — importing just that one module (a lazy win over
  the full walk).

So crm describe entity hands back the full, structured spec of the entity group — every verb, every flag, every allowed choice — in one call. That's the difference from --help: --help is prose for you to read; describe is a catalogue an agent can parse to decide what to call. Pass a GROUP to scope it (faster — it loads only that module).

✓
Which to reach for: you, at the terminal, exploring → --help. An agent (or you, scripting) needing the complete option set to plan a command → crm describe [group]. Same map, two readers.

04 The global flags worth knowing now

From crm --help, a few globals (before the group) you'll lean on across the whole mission — several you've already met:

--json
The machine-readable envelope. Your default. (L03)
--dry-run
Preview writes; reads run normally. (L04)
--profile
Run against a named profile for this call, without switching the active one. (L01)
--auth-scheme
Override ntlm/oauth for one run — the cloud-vs-on-prem switch. (L02; M09)
--stage-only
Stage metadata changes without publishing — a write-safety net for customizations. (later)

You don't need to learn these now — just know they live at crm --help, and that being global means they go before the group, not after.

05 Do it yourself ~3 min

A discovery drill — answer each by asking the CLI, not by memory. All read-only:

# 1 · list every group
crm --help

# 2 · which group manages solutions? then list its verbs
crm solution --help

# 3 · find the flags for retrieving a record
crm entity get --help

# 4 · the machine-readable spec of one group
crm describe query

Then, without me telling you:

  1. Which group + verb would you use to delete a record? To update one? (You can find both from §2's help output.)
  2. In step 3, what flag limits which columns come back? (You've used it — now confirm it from the help.)
  3. Which group would you explore first to create a custom table? Name your guess, then check its --help to test it.
✓
The habit that outlasts this course: when you don't know a command, you no longer ask "what's the syntax?" — you ask crm <group> --help (or crm describe) and read it off. The tool documents itself.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You want to know every verb available under the solution group. What do you run?
Correct. --help at the group level lists that group's verbs. crm --help (top level) only lists groups; crm describe with no arg walks the whole tree (more than you need, slower).
Not quite. crm --help lists groups, not one group's verbs; bare crm describe walks everything. To see a single group's verbs, run crm <group> --help — here, crm solution --help.
2 · Where does a global flag like --json or --dry-run go in the command?
Correct. Global flags apply to the whole invocation, so they sit before the group. Command-specific flags (like --select) come after the verb. Keeping the two straight avoids a class of "unknown option" usage errors.
Not quite. Globals (--json, --dry-run, --profile) go before the group; verb-specific flags (--select, --data) go after. Putting a global after the verb can trip a usage error.
3 · You're writing an agent that must programmatically learn every option of the entity group before composing a call. Best source?
Correct. crm describe exists precisely for this: a machine-readable catalogue of commands, options, and choices. Passing the GROUP scopes it to entity and loads only that module — faster than walking the whole tree.
Not the robust choice. describe is built for exactly this — crm describe entity returns the group's full option set as structured data, no fragile text-scraping, no guessing, no trial writes.
4 · From the help output in §2, which group + verb updates an existing record — even though no lesson drilled it?
Correct — and notice how you know: the crm entity --help output listed update — PATCH an existing record. That's the whole point of this lesson: the answer was discoverable, not memorized.
Re-read §2's entity --help output: record CRUD lives under entity, and the verb is update ("PATCH an existing record"). You found it by reading the help — exactly the skill here.
