Milestone 12 · Capstone · Lesson 02

Build the
schema layer

The first runbook. Five writes, in the one order the platform allows: publisher → solution → table → option set → status column. Preview each, commit it, read it back — then tick it off.

live on agent-cloud  ·  writes your turn
Why this matters: Schema is the foundation the whole Deal Desk stands on — relationships, logic and UI all attach to the table and column you build here. Build them into the solution from the first write (so they travel to test and prod), in dependency order (so each command has something to attach to), and verify each one — because a wrong schema name found three lessons later is expensive to unwind.

You've created tables, option sets and columns before — in M04. What's new is doing it as a chain: each write depends on the one before it, and each lands inside the same solution. So the order isn't a preference. It's the platform's.

01 The runbook

Work top to bottom. Each step is preview → commit → verify: run it with --dry-run first and read the method/url/body, then drop --dry-run to commit, then run the read-back. Paste me each envelope and tick the box.

GOAL A DealDesk solution holding a Deal Review table with an Approval-Status column.

target: agent-cloud · whoami first · all names use the itworx prefix

  1. 1. Create the publisher. The prefix (itworx) brands every component's logical name; the option-value prefix seeds option-set integers.

    step 1 · POST /publisherscrm solution create-publisher --name DealDeskPub \
  --display "Deal Desk Publisher" --prefix itworx --option-value-prefix 30000

    ✓ verify: crm query odata publishers --filter "uniquename eq 'DealDeskPub'" --select uniquename,customizationprefix returns one row.

  2. 2. Create the solution, bound to that publisher. Everything after this carries --solution DealDesk so it lands inside this container.

    step 2 · POST /solutionscrm solution create --name DealDesk --display "Deal Desk" \
  --version 1.0.0.0 --publisher DealDeskPub

    ✓ verify: crm solution list shows DealDesk at 1.0.0.0, unmanaged.

  3. 3. Create the Deal Review table. --primary-attr is the required primary-name column; --publish makes the change live.

    step 3 · POST /EntityDefinitionscrm metadata create-entity --schema-name itworx_dealreview \
  --display "Deal Review" --display-collection "Deal Reviews" \
  --primary-attr itworx_name --primary-label "Name" \
  --ownership UserOwned --solution DealDesk --publish

    ✓ verify: crm metadata entity itworx_dealreview returns the definition (note its EntitySetName).

  4. 4. Create the global Approval Status option set. :Pending = auto-assigned value, label "Pending". Global, so other components could reuse it.

    step 4 · POST /GlobalOptionSetDefinitionscrm metadata create-optionset --name itworx_approvalstatus \
  --display "Approval Status" \
  --option ":Pending" --option ":Approved" --option ":Rejected" \
  --solution DealDesk --publish

    ✓ verify: crm metadata get-optionset itworx_approvalstatus lists the three options + their values.

  5. 5. Add the status column, bound to that set. A picklist column with --optionset-name binds to the global set from step 4 — which is exactly why step 4 came first.

    step 5 · POST …/Attributescrm metadata add-attribute itworx_dealreview --kind picklist \
  --schema-name itworx_status --display "Status" \
  --optionset-name itworx_approvalstatus --solution DealDesk --publish

    ✓ verify: crm metadata entity itworx_dealreview now lists itworx_status among its attributes.

02 Read the preview before you commit

Step 1 with --dry-run returns the exact HTTP request the commit would send — no write, but a real round-trip. Read it like a contract:

live · crm --dry-run solution create-publisher …{ "ok": true, "data": {
  "_dry_run": true,
  "method": "POST",
  "url": "https://orgexample.crm.dynamics.com/api/data/v9.2/publishers",
  "body": {
    "uniquename": "DealDeskPub",
    "customizationprefix": "itworx",
    "customizationoptionvalueprefix": 30000 },
  "_exists": false } }
DRY-RUN IS LIVE
--dry-run means "no write", not "no traffic" (M01 L04). Proof: try --dry-run solution create before the publisher exists and it fails PublisherNotFound — it really looked. That live lookup is your ordering guard: the dry-run won't let you bind a solution to a publisher that isn't there yet.

03 Two flags that make it travel

Two flags appear on nearly every write in this milestone, and both are about getting the change downstream intact:

FlagWhat it does · why it matters
--solution DealDeskAdds the new component to the solution as it's created. Omit it and the component lands in the default solution — orphaned from your package, and it won't export in L06. Bind from the first write.
--publishRuns PublishAllXml after the change so it's live. Unpublished customizations don't appear in forms and don't travel in an export (M10 L05). Publish-per-step keeps your read-backs honest.
VERIFY EACH
Don't batch all five then check once. A typo in --schema-name at step 3 makes step 5's add-attribute target the wrong table — and you won't know until the read-back. The chain only stays cheap if every link is proven before the next.

Lesson check

4 questions · click to reveal

1. Why does the option set get created before the status column?

Correct. add-attribute --kind picklist --optionset-name … binds the column to the global option set via GlobalOptionSet@odata.bind — the set has to exist first. That's the dependency fixing the order.
The picklist column references the global set (GlobalOptionSet@odata.bind), so the set must exist first. It's not about publish speed, table publish state, or component sorting.

2. You forget --solution DealDesk on create-entity. What's the consequence?

Correct. Without --solution the component goes to the default solution — it exists, but it's orphaned from your package and silently absent from the L06 managed export. Bind from the first write.
It doesn't fail or duplicate — the component just lands in the default solution, orphaned from DealDesk, and won't be in the export. Always pass --solution.

3. --dry-run solution create returns PublisherNotFound. What does that tell you?

Correct. Dry-run means no write, not no traffic — it really resolved the publisher and found nothing. The fix is the chain order: publisher (step 1) before solution (step 2).
Dry-run isn't broken — it performed a live lookup and the publisher genuinely doesn't exist yet. Create it first; that's the ordering guard working as intended.

4. Best practice for verifying the five writes?

Correct. Verify each link before forging the next. A bad --schema-name at step 3 silently misaims step 5; per-step read-backs catch it immediately instead of three lessons later.
Per-step read-backs are the point. Batching the check, trusting exit codes, or checking only the end lets an early typo propagate undetected down the chain.
