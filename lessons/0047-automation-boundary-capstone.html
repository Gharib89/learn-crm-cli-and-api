Why this matters: The automation boundary is the most important thing an agentic D365 workflow needs to know. When a requirement lands — "send an email on record create", "validate a field before save", "route a ticket by SLA" — the right answer depends on knowing which tier owns it, whether the CLI can reach it, and what needs a designer, a .NET project, or another tool entirely. This lesson closes M08 with that map.

D365 automation spans five surfaces: business rules, workflows, cloud flows, plug-ins/webhooks, and SLAs. The crm CLI reaches all five — but in very different ways and to very different depths.

01 crm action — call OData functions and actions

The action group is the CLI's direct line to the Dataverse Web API's OData function and action layer. These are not the same as classic workflow actions (category 3) — they are HTTP GET/POST calls to named endpoints exposed by the platform.

Functions (GET) — action function

OData functions are read-only (GET). The most useful unbound function is WhoAmI — it returns the calling user's GUIDs, and double-checks that your profile and OAuth token are wired to the right org:

shellcrm --json action function WhoAmI
envelope · live-verified 2026-06-18{
  "ok": true,
  "data": {
    "BusinessUnitId": "4a617aa7-0000-0000-0000-000d3a3b279b",
    "UserId":         "bf40e4d2-0000-0000-0000-6045bd001b10",
    "OrganizationId":  "9102ced1-0000-0000-0000-000d3a302819"
  }
}

Functions that take parameters use --params as a JSON dict:

shell · RetrieveCurrentOrganizationcrm --json action function RetrieveCurrentOrganization \
    --params '{"AccessType":"Default"}'

Returns the org's metadata: OrganizationVersion, FriendlyName, Geo, EnvironmentId, and more. Useful for verifying which environment an agent is targeting.

Bound functions — --bind-set and --bind-id

Pass --bind-set alone for collection-bound; add --bind-id for record-bound:

shell · record-bound function# Bound to a single record (set + id)
crm --json action function MyNamespace.SomeFunction \
    --bind-set contacts \
    --bind-id a1b2c3d4-1111-2222-3333-00155d467b91

Actions (POST) — action invoke

OData actions have side effects (POST). The workflow run command you used in L02 is just a convenience wrapper around one: it POSTs to workflows(<id>)/Microsoft.Dynamics.CRM.ExecuteWorkflow. You can call any action directly:

shell · unbound actioncrm --json action invoke PublishAllXml \
    --body '{"ParameterXml":"<importexportxml />"}'
shell · bound action (record-level)crm --json action invoke Microsoft.Dynamics.CRM.ExecuteWorkflow \
    --bind-set workflows \
    --bind-id 79fcb86b-0000-0000-0000-8d5fbb78b84e \
    --body '{"EntityId":"a1b2c3d4-1111-2222-3333-00155d467b91"}'
TIP
Function vs action: if the operation has side effects (creates, updates, triggers something), it's an action → use invoke. If it's read-only (returns data, no mutation), it's a function → use function. The Microsoft.Dynamics.CRM namespace in the Web API reference lists all built-in bound/unbound actions and functions.

02 crm sla activate — unlock SLA enforcement

An SLA (Service Level Agreement) in D365 enforces response-time KPIs — for example, "a case must be resolved within 4 hours." Each SLA has one or more KPI items, and each KPI item has a backing classic workflow that fires the warning/failure actions. The full SLA must be activated before it can enforce anything.

The complication: backing workflows must be activated before the SLA record itself. If you import an SLA as part of a solution (M06), the workflows land in Draft state and the SLA is inactive. The CLI activates the whole stack in one call:

shellcrm --json sla activate <sla-id>

What happens internally:

  1. CLI reads the SLA's backing workflow IDs from slaitem records
  2. Activates each backing workflow that is currently in Draft (already-active workflows are skipped — re-running is safe)
  3. Sets the SLA record's statecode to Activated

If any backing workflow fails to activate (compile errors after a solution import), the SLA is left untouched and the per-workflow error report tells you which workflow needs attention in the GUI.

WARN
SLA creation and KPI definition are GUI-only. You cannot create an SLA record, define KPI items, or set warning/failure times via the CLI. sla activate manages the activation step only — the rest lives in the Customer Service Hub app. This is one of the clearest automation-boundary cases in D365.

Finding SLA IDs

shell · read SLAscrm --json query odata slas --select name,slaid,statecode,statuscode

Returns a list of SLA records. Filter to inactive ones (statecode eq 0) to find candidates for activation.

04 The automation boundary — full decision matrix capstone

Given any automation requirement, use this matrix to decide which tier handles it and whether the crm CLI can reach it:

Requirement Tier crm CLI Other tool needed
Show/hide a field based on another field's value (client-side) Business rule (cat 2) workflow activate/deactivate only Power Apps form designer to create/edit
Send an email when a record is created (no code) Classic workflow or cloud flow workflow activate/run/export/import Power Automate designer to create/edit logic
Validate a field before save and block if invalid Plug-in (pre-operation, sync) plugin register-assembly/step (full CLI) Visual Studio + .NET to build the .dll
Notify an external system on every contact create (no .NET) Webhook (event pipeline step) plugin register-webhook + register-step (full CLI) Nothing — webhook needs no code
Enforce a 4-hour response SLA on cases SLA sla activate only Customer Service Hub UI to define KPI items
Migrate classic workflows to Power Automate Cloud flow (cat 5) workflow migration-assess (inventory only) Power Automate designer to build the flow
Call a custom OData action or platform function OData API action invoke / action function (full CLI) Nothing — fully CLI-callable
Design a Business Process Flow (multi-stage form wizard) BPF (cat 4) None (not exposed) Power Apps BPF designer (GUI-only)

The automation-boundary rule

A useful short rule for agentic workflows:

  • Register and manage existing automation → CLI
  • Create or edit logic (workflow steps, BPF stages, SLA KPIs) → designer / IDE
  • Register server-side event handlers (plug-ins, webhooks) → CLI (except the .NET code)
  • Call the platform API (actions, functions) → CLI

05 M08 capstone — full round-trip writes cleanup

Done-when: register a workflow (trigger it) OR register a plug-in step, and articulate what needs other tools.

capstone · M08 full round-trip · crm v1.0.0 · live-verified 2026-06-18# ── PART A: Classic workflow ─────────────────────────────────────────────────

# 1. Inventory classic workflows on the contact entity
crm --json workflow list --category workflow --entity contact
#   → find an on-demand workflow (ondemand: true, statecode: 1)
#   → note its workflowid

# 2. Trigger it on-demand against a contact   ← done-when ①
crm --json workflow run <workflowid> --target <contactid>
#   async_operation_id: null is normal — system job queued

# 3. Assess for cloud migration blockers
crm --json workflow migration-assess --entity contact
#   → check verdict + blockers[]

# ── PART B: Webhook step ─────────────────────────────────────────────────────

# 4. Register a webhook service endpoint
crm --json plugin register-webhook \
    --name "itworx-m08-hook" \
    --url "https://webhook.site/<your-token>" \
    --auth httpheader \
    --auth-value "X-Hook-Secret: m08-demo" \
    --solution itworx_dev
#   → note serviceendpointid

# 5. Register a step on contact Create   ← done-when ②
crm --json plugin register-step \
    --message Create \
    --service-endpoint "itworx-m08-hook" \
    --entity contact \
    --stage postoperation \
    --mode async \
    --name "M08 notify on contact create" \
    --solution itworx_dev
#   → note stepid

# 6. Fire the step — create a test contact
crm --json entity create contacts \
    --data '{"firstname":"M08","lastname":"TestContact"}'
#   → platform queues the webhook POST to webhook.site
#   → note contactid for cleanup

# 7. Confirm identity and org (sanity check)
crm --json action function WhoAmI
#   → UserId, BusinessUnitId, OrganizationId

# ── CLEANUP ──────────────────────────────────────────────────────────────────

# 8. Unregister the step (before the webhook endpoint)
crm --json plugin unregister-step <stepid> --yes

# 9. Delete the webhook service endpoint (no unregister-webhook; use entity delete)
crm --json entity delete serviceendpoints <serviceendpointid> --yes

# 10. Delete the test contact
crm --json entity delete contacts <contactid> --yes

4 questions · click to reveal

1. A requirement arrives: "Validate a contact's phone number before the record is saved, and block the save if the format is wrong." Which automation tier and CLI coverage applies?

Correct. Pre-save validation that can block the operation requires a synchronous plug-in at pre-operation or pre-validation stage. The CLI registers the assembly and step. The .NET code (the blocking logic) must be built with Visual Studio — outside the CLI's scope. A webhook is async postoperation — it can't block the save.
Pre-save blocking requires a synchronous plug-in at pre-operation (stage 20) or pre-validation (stage 10). Classic workflows and webhooks are async (or late-stage) — they can't veto the save. Business rules are client-side show/hide logic, not server-side validators. The CLI registers the plug-in step; building the .NET assembly requires Visual Studio.

2. What is the correct cleanup order after registering a webhook and attaching a step?

Correct. Steps reference the service endpoint — if you delete the endpoint while steps still point to it, you leave orphaned references. Unregister steps first, then delete the endpoint. Also: there is no unregister-webhook command; delete the service endpoint via entity delete serviceendpoints <id> --yes.
Steps reference the service endpoint. If you delete the endpoint first, steps are left with a broken reference and can cause errors when the platform tries to fire them. Always unregister steps before deleting their service endpoint.

3. Which command verifies your current user identity and the target org from the CLI?

Correct. action function WhoAmI is the OData WhoAmI function — returns UserId, BusinessUnitId, and OrganizationId for the calling user. It's the lightweight org-identity check (distinct from connection test which verifies host/version).
action function WhoAmI returns the calling user's GUIDs and org ID. workflow list reads the workflow table; plugin list-types reads registered assemblies; sla activate is a write operation — none of those confirm identity.

4. migration-assess returns a workflow with "verdict": "ready", "blockers": []. What does this mean?

Correct. Verdict "ready" with empty blockers means the assess tool found no migration-blocking patterns (no custom activities, no real-time mode, no wait conditions). The workflow can be rebuilt in Power Automate without structural redesign — though you still need the Power Automate designer to do the actual migration.
Verdict "ready" means the assess tool found no migration blockers — no custom activities, real-time mode, or wait conditions. The workflow can be migrated to a cloud flow without redesign. It does not mean the workflow is currently running or already migrated.
