The automation
landscape

D365's four automation tiers — business rules, classic workflows/Power Automate, plug-ins/webhooks, and SLA — and which tier the crm CLI reaches.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads only
Why this matters: Before touching crm workflow or crm plugin, you need a mental map. D365 has four automation tiers that differ by when they fire, who writes them, and whether the CLI can reach them. Without this map, you won't know which tool to reach for when a requirement lands.

D365 automation is not one thing — it is four distinct tiers, each with a different scope, runtime, and authoring model. The crm CLI has full coverage on some tiers and read-only access on others. Knowing the map upfront saves you from reaching for the wrong tool.

01 Four automation tiers

Every automated behaviour in D365 belongs to one of these tiers. The tier determines where the logic runs, who can author it, and what the CLI can do.

Tier Category # When fires Who writes crm CLI coverage
Business rule 2 Client-side, on form load / field change. Scoped to a single form; no server round-trip. Maker (GUI — no .NET) workflow list --category businessrule, activate, deactivate. Creation/logic = GUI-only.
Classic workflow 0 Background or real-time; on record event or on-demand trigger. Maker (XAML-based designer — GUI) Full management: list, activate, deactivate, run, export, import, migration-assess. Creation = GUI-only.
Power Automate cloud flow 5 Cloud; triggered by events, schedules, or HTTP. Modern replacement for classic workflows. Power Automate designer (GUI — cloud-only) Read-only: workflow list --category flow. No create/edit via CLI.
Plug-in / Webhook N/A — registered in event pipeline Synchronous or asynchronous; fired when an SDK message (Create/Update/Delete/etc.) matches a registered step. Developer (.NET assembly for plug-in; any HTTP endpoint for webhook) Full lifecycle: crm plugin register-assembly, register-step, register-image, register-webhook, set-step-state, unregister-*.
SLA Separate entity slas Background; monitors response-time KPIs via backing workflows. Fires when KPI windows open/close. Maker (GUI — KPI definition) Partial: crm sla activate only. KPI definition and creation = GUI-only.
TIP
The dividing line. The CLI can manage the lifecycle of existing automation (activate, deactivate, export, register) across all tiers. The only tier where it also handles the event-pipeline registration end-to-end is plug-ins and webhooks — because those don't have a visual designer.

02 Workflow categories deep-dive

The workflow entity in D365 is overloaded — it stores business rules, classic workflows, dialogs, actions, BPFs, and cloud flows. The category field is the discriminator:

category valueNameNotes
0WorkflowClassic background/real-time workflow (XAML). The primary migration target for Power Automate.
1DialogDeprecated interactive dialogs. Not covered in this course.
2Business RuleClient-side form logic. No .NET; fires in browser.
3ActionCustom OData actions — reusable server-side logic callable from anywhere.
4Business Process Flow (BPF)Guided stage-based process on a form.
5FlowPower Automate cloud flow. Read-only via CLI.

03 The event pipeline preview (plug-ins)

Classic workflows run outside the synchronous request cycle. Plug-ins are different: they are registered inside the event pipeline — the D365 mechanism that routes every SDK message (Create, Update, Delete, Associate, etc.) through a chain of registered handlers before and after the database write.

The pipeline has three stages:

StageNumberWhen it runsCan abort?
Pre-Validation10Before security checks; outside the transactionYes — throws pre-transaction
Pre-Operation20Inside the transaction, before the database writeYes — rolls back
Post-Operation40Inside the transaction, after the database writeYes — rolls back the write

Plug-ins hook into one of these stages on a specific message + entity combination. Webhooks hook into post-operation asynchronously. The crm CLI can register, update, and unregister assemblies, steps, images, and webhooks — but building the .NET assembly is outside its scope (that's Visual Studio / dotnet SDK territory).

shellcrm plugin --help
help output · trimmedUsage: crm plugin [OPTIONS] COMMAND [ARGS]...

  Manage plug-in assemblies, steps, images, and webhooks.

Commands:
  register-assembly   Register a .NET plug-in assembly (.dll)
  register-step       Register an execution step on a pipeline stage
  register-image      Register a pre/post entity image on a step
  register-webhook    Register an HTTP webhook endpoint
  set-step-state      Enable or disable an individual step
  unregister-assembly Remove assembly and all its steps
  unregister-step     Remove a single step registration
  unregister-webhook  Remove a webhook registration
  list-assemblies     List registered assemblies
  list-steps          List registered steps (filterable by entity/message)
TIP
Webhooks need no .dll. register-webhook only needs an HTTP endpoint URL. This makes webhooks the fastest path to server-side automation with the CLI — no .NET toolchain required.

04 CLI coverage map

Full reference of what the CLI can and cannot do in each automation area:

Command group Verbs Automatable via CLI Notes
workflow list activate deactivate run export import clone delete migration-assess Yes — manage existing Create/edit workflow logic = GUI only. --category accepts: workflow(0), businessrule(2), action(3), bpf(4), flow(5).
plugin register-assembly register-step register-image register-webhook set-step-state unregister-* list-assemblies list-steps Yes — full lifecycle Requires pre-built .dll for plug-ins. Webhooks require only an HTTP endpoint — no .dll.
sla activate Partial KPI definition and SLA record creation = GUI only. CLI activates an already-configured SLA.
action invoke function Yes — invoke Call any OData bound/unbound action or function. Does not create custom API definitions (that's workflow list --category action + GUI).
WARN
Power Automate create/edit = GUI-only, always. workflow list --category flow is read-only. There is no CLI path to create or modify a cloud flow's trigger, actions, or connections — only the Power Automate designer can do that.

Lesson check

4 questions · click to reveal

1. Which workflow category number represents classic background processes (the ones that migrate to Power Automate)?

Correct. Category 0 is the classic workflow (XAML-based background/real-time process). Category 2 = business rule, 4 = BPF, 5 = Power Automate cloud flow.
Category 0 is the classic workflow. Category 2 = business rule (client-side form logic), 4 = Business Process Flow, 5 = Power Automate cloud flow.

2. An on-demand classic workflow has statecode: 0. Can you run it with workflow run?

Correct. statecode: 0 = Draft (inactive). A workflow must be activated (statecode: 1) before it can fire — on-demand or otherwise.
A workflow must be activated (statecode: 1 / statuscode: 2) before it can be triggered by any means. Draft workflows are blocked. Use workflow activate <id> first.

3. Which crm CLI group can fully manage the event pipeline for server-side code?

Correct. crm plugin covers register-assembly, register-step, register-image, register-webhook, set-step-state, and all unregister verbs — the full pipeline lifecycle.
crm plugin manages the event pipeline. crm workflow manages classic workflows/flows. crm sla is activate-only. crm action invokes existing actions — it doesn't register pipeline steps.

4. You need to create a new classic workflow that sends an email when a contact is created. Which tool do you use?

Correct. Classic workflow creation and logic authoring is GUI-only — Power Apps maker or the legacy workflow designer. The CLI can only manage workflows that already exist: activate, export, run, etc.
Workflow creation is GUI-only. crm workflow import can import an existing exported workflow definition, but you cannot craft a new XAML definition by hand via CLI. Use Power Apps maker or Power Automate designer.
