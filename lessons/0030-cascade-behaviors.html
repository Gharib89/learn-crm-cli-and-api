Milestone 05 / Relationships / Lesson 03

When the parent
moves

Delete an account — what happens to its projects? Cascade behaviors are the answer, and they're the most D365-specific thing in this milestone.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. Your L02 dry-run printed a CascadeConfiguration block you never typed. That block is doing real work: it decides whether deleting an account also deletes its projects, whether reassigning it reassigns them, whether sharing it shares them. This is genuinely Dataverse-specific — no plain SQL foreign key carries ownership and sharing rules like this. Get it wrong and you orphan data or delete too much.

Today's win: read a relationship's six cascade actions, name the difference between parental and referential, and flip one with update-relationship.

01 Six actions on the parent, each cascades (or not)

A 1:N relationship answers six "what should happen to the children when…?" questions about the parent row. Each is one cascade action:behavior

Action"When the parent is…"CLI flag
Deletedeleted — delete / unlink / block?--cascade-delete
Assignreassigned to a new owner — follow?--cascade-assign
Reparentits lookup is changed — re-inherit access?--cascade-reparent
Shareshared with a user — share children too?--cascade-share
Unshareunshared — remove children's share?--cascade-unshare
Mergemerged with another row — bring children?--cascade-merge

Each action takes one of these cascade types — the values the CLI accepts:cascade types

NoCascade
Do nothing to the children.
Cascade
Apply to all related rows ("Cascade All").
Active
Apply only to active related rows.
UserOwned
Apply only to related rows owned by the same user.
RemoveLink
Delete only. Clear the children's lookup, leave the rows.
Restrict
Delete only. Block the parent delete while children exist.
!
Not every type is legal for every action. Delete only accepts Cascade / RemoveLink / Restrict. Merge only accepts Cascade / NoCascade. Assign / Reparent / Share / Unshare accept Active / Cascade / NoCascade / UserOwned.cascade types

02 Two named bundles: parental vs referential

You rarely set six actions by hand. The platform names a few standard bundles, and these are the words a D365 consultant will actually say to you:behavior

BehaviorWhat it meansDeleteAssign/Share/Reparent
ParentalChildren belong to the parent — actions flow downCascadeCascade
ReferentialChildren are independent; just linkedRemoveLinkNoCascade
Referential,
Restrict Delete		Independent, but can't orphanRestrictNoCascade

The mental test: "If I delete the account, should its projects vanish too?" If yes, that's parental (a project can't exist without its account). If the projects should survive with a blank account, that's referential / RemoveLink. If you'd rather forbid deleting an account that still has projects, that's Restrict Delete.

✓
The CLI default is referential. Your L02 dry-run showed Delete: RemoveLink and every other action NoCascade — that's the safe, independent default. Deleting the account would just blank the ag_accountid on its projects, not delete them. You opt into parental on purpose.

03 Set cascades at create time

Every --cascade-* flag is available on create-one-to-many (L02). To make the account → project relationship parental from the start — projects die with their account:create-one-to-many

agent-cloud · exit 0crm --json metadata create-one-to-many \
    --schema-name ag_account_ag_project --referenced-entity account \
    --referencing-entity ag_project \
    --lookup-schema ag_AccountId --lookup-display "Account" \
    --cascade-delete Cascade --cascade-assign Cascade \
    --cascade-share Cascade --cascade-reparent Cascade
!
You can't always choose parental. A custom table can't be the primary (parent) side of a cascading relationship to a system table, and each table pair usually allows only one parental relationship — to prevent a row having two cascade-parents.behavior Account → your custom project is fine; custom → account cascading is not.

04 Change cascades later: update-relationship

Decisions change. update-relationship does a retrieve-merge-write: it reads the current definition, applies only the cascade flags you pass, and writes it back — no need to restate the whole thing:update-relationship

agent-cloud · exit 0# promote an existing referential rel to parental on delete + assign
crm --json metadata update-relationship ag_account_ag_project \
    --cascade-delete Cascade --cascade-assign Cascade

{ "ok": true, "data": {
    "updated": true,
    "path": "RelationshipDefinitions(2c99a64a-926a-f111-…)",
    "schema_name": "ag_account_ag_project",
    "published": true } }

To audit what's currently set, read it back — the cascade block is part of the relationship metadata, and you saw its shape in the L02 dry-run body. update-relationship publishes by default; pass --no-publish to stage.

✓
This is reversible. Set --cascade-delete RemoveLink to demote back to referential. Because it's retrieve-merge-write, flags you don't pass keep their current value — you only change what you name.

05 Do it yourself ~8 min writes

Creates a relationship, changes its cascade, then cleans up. (Assumes ag_project exists — create it first if needed, M04.)

# 1 · create referential (the default) and read the cascade block in the dry-run
crm --json --dry-run metadata create-one-to-many \
    --schema-name ag_account_ag_project --referenced-entity account \
    --referencing-entity ag_project \
    --lookup-schema ag_AccountId --lookup-display "Account"
#   → note CascadeConfiguration: Delete=RemoveLink, rest NoCascade

# 2 · create it for real (drop --dry-run)

# 3 · promote to parental-on-delete
crm --json metadata update-relationship ag_account_ag_project \
    --cascade-delete Cascade

# 4 · try an illegal value — read the error
crm --json metadata update-relationship ag_account_ag_project \
    --cascade-merge RemoveLink

# 5 · clean up
crm --json metadata delete-relationship ag_account_ag_project --yes
  1. In step 1, write down all six cascade defaults. Which single one is not NoCascade, and what is it?
  2. After step 3, decide: if you now delete an account that has projects, what happens to the projects? (vs. before step 3?)
  3. Step 4 passes RemoveLink to --cascade-merge. Does it succeed? Merge only allows Cascade/NoCascade — read what the CLI/server says.
  4. One sentence: give a real example where you'd choose Restrict Delete over RemoveLink.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · Deleting an account should also delete all its projects. Which cascade behavior?
Correct. "Children die with the parent" is the parental pattern — Cascade on delete. The project can't exist without its account.
To delete children with the parent you need --cascade-delete Cascade (parental). RemoveLink just blanks the lookup; Restrict blocks the delete; NoCascade isn't valid for delete.
2 · The CLI's default cascade for a new 1:N is…
Correct. The L02 dry-run showed it: Delete: RemoveLink, Assign/Reparent/Share/Unshare/Merge all NoCascade. Safe and independent — you opt into parental.
The default is referential: Delete: RemoveLink with the rest NoCascade. Parental is something you choose deliberately, not the default.
3 · You want to forbid deleting an account while it still has projects. Set --cascade-delete to…
Correct. Restrict blocks the parent delete while related rows exist — the user must remove the children first. That's "Referential, Restrict Delete."
Restrict prevents deleting the parent while children exist. Cascade would delete them; RemoveLink would orphan them; NoCascade isn't valid for Delete.
4 · You set the relationship referential, then later want parental delete. Best tool?
Correct. update-relationship is a retrieve-merge-write — it changes only the cascade flags you pass and keeps the rest. No need to drop the relationship.
Cascades are editable in place via update-relationship; it merges your flag onto the existing definition. No delete-and-recreate, and it's not on the attribute.
