What it can —
and can't

The CLI wraps the Dataverse Web API. If a customization has a Web-API write path, the agent can do it. If it lives only in a designer or the solution-zip pipeline, it's GUI-only — and the agent's job is to say so, not fake it.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads boundary
Why this matters: The mission asks you to judge, for any requirement, what's automatable via the crm CLI / Dataverse Web API and what isn't. Get this line wrong and the agent either refuses work it could do, or — worse — confidently "completes" a step that has no API and leaves a silent hole in your customization. Knowing the boundary up front is what makes the agentic workflow trustworthy.

There's one clean test behind the whole boundary: is there a Web-API write path? The CLI is a faithful wrapper over the Dataverse Web API — it can do what the API can do, and it refuses rather than fakes what the API can't.

01 The line has two questions, not one

Every crm verb issues a real HTTP request to /api/data/v9.x/, so it's tempting to say "automatable = there's a Web-API path." That's too coarse. Two questions actually decide what the agent does: (1) is there a supported Dataverse API path? and (2) does the crm CLI expose a verb for it?

TEST
The two answers give three outcomes: API + CLI verb → do it now (most of M01–M09). Supported API but no verb yet → a CLI gap: doable via the raw API/SDK, and worth a feature request. No supported API path → genuinely GUI/pipeline-only. "GUI-only" in everyday talk usually means *"the CLI has no verb"* — which, as you'll see in §03, isn't always the same as *"the platform can't."*

03 What the CLI can't do — three different reasons

None of these have a crm verb in v1.0.0 (confirmed absent from crm describe). But why differs — and the right response differs with it:

ComponentBucketReality
Charts (savedqueryvisualization)CLI gapSupported Web-API/SDK create — the CLI just has no verb. Low risk.
Dashboards (systemform)CLI gapSupported SDK create (FormXml). No verb yet.
SLA + KPI itemsCLI gapEntities with a create path; CLI only does sla activate.
Rollup & calculated columnsunsupported via APIThe metadata field exists, but MS docs: "the only supported way to change FormulaDefinition is the formula editor."
BPF (cat 4) · business rules (cat 2)unsupported via APIAuthored in the designer; raw clientdata is unsupported, and enabling a table for BPF is irreversible.
Ribbon RibbonDiffXml · early-bound classesno API at allSolution-zip pipeline (ribbon) or external codegen (pac/CrmSvcUtil) only.
NUANCE
Two cautions. (1) "GUI-only" is per-component, not per-domain: the agent can add a ribbon button (`ribbon add-button`) but not hand-edit arbitrary `RibbonDiffXml`; it can register a plug-in step but its DLL is compiled outside. (2) "Modern flows (cat 5)" are not on this list: workflow export/import already round-trip and clone flow definitions — net-new authoring still needs designer-generated clientdata, but promotion is covered.
REAL
The three "CLI gap" rows above were filed as upstream feature requests during this lesson's authoring — #420 charts, #421 dashboards, #422 SLA create. A CLI gap isn't a dead end; it's a roadmap item (and exactly the feedback loop L06 teaches).

04 The move when you hit the wall

When a requirement contains a no-verb piece, the agent doesn't go quiet and it doesn't pretend. It detects it early (during planning), names it explicitly so it's a tracked task — and then the response depends on which bucket it fell in:

  1. CLI gap (supported API, no verb) → do it via the raw Web-API/SDK if it's on the critical path, and/or file a feature request so the verb exists next time. Low risk — the platform supports it.
  2. Unsupported via API (rollup, BPF, business rules) → hand off to the designer with a short manual checklist. Forcing the raw API here is unsupported and risky (invalid components, irreversible flags).
  3. No API at all (RibbonDiffXml, codegen) → it's the solution-zip pipeline or external tooling; flag it as out-of-band.
DEAL DESK
The requirement you'll plan next — a "Deal Desk" approval on Opportunity — is mostly automatable (table, option set, relationship, plug-in step, app). Its rollup of total approved value and approval BPF are the unsupported-via-API hand-offs (designer + checklist); its notification flow is designer-authored but, once built, promotes via workflow export/import. A trustworthy agent builds what it can and is precise about why each remainder is manual — which is what L03 and L04 teach you to orchestrate.

4 questions · click to reveal

1. A chart has a supported Web-API create path, but crm has no chart verb. What is this?

Correct. "No verb" ≠ "no path." Charts (savedqueryvisualization) have a documented, supported API, so this is a CLI gap — the agent can fall back to the raw API and file a feature request (we filed #420). It's not impossible, not unsupported, and a missing-feature isn't a defect.
It's a CLI gap: the platform supports charts via a documented API; the CLI just lacks the verb. Fall back to the raw API and/or file a feature request. It's not impossible, not unsupported-and-risky, and not a bug.

2. Which is unsupported via API — the field exists but Microsoft says use the designer?

Correct. The attribute metadata carries FormulaDefinition, but MS docs state the only supported way to set it is the formula editor — so authoring a rollup via raw API is unsupported/risky. Option sets and plug-in steps have real verbs; charts are a CLI gap (supported API, no verb).
The rollup's FormulaDefinition is the unsupported-via-API case (designer-only per MS). Option sets and steps are automatable; a chart is a CLI gap (supported API), not unsupported.

3. A requirement includes a Power Automate flow for email. What does a trustworthy agent do?

Correct. Cloud-flow logic has no Dataverse write path, so the agent names it as a manual task and proceeds with everything that is automatable. It doesn't fabricate an API call, abandon the whole job, or substitute an unrelated mechanism.
Flows are GUI-only — the agent surfaces that as a manual task and still does the automatable parts. Faking a POST, abandoning the requirement, or swapping in a plug-in step are all wrong responses.

4. "GUI-only is per-domain" — what's the correction?

Correct. The boundary runs through individual components. Within one domain, some pieces have a Web-API path and some don't — a ribbon button has a verb, arbitrary RibbonDiffXml doesn't; a plug-in step registers via the CLI, but its DLL is compiled outside. Always check the specific component.
It's per-component, not per-domain or per-target. The same domain mixes automatable and GUI-only pieces (ribbon button vs RibbonDiffXml; step registration vs DLL compilation), so judge each component on its own.
