Many ↔ many,

The capstone: build an N:N, link rows through it, choose the right type for a requirement — and tear it all down cleanly.

Why now. You've done 1:N end-to-end. The second real relationship type — N:N — needs a different create verb, a different way to link rows (you associate, you don't bind), and a clear sense of when to reach for it. Then comes the part every milestone owes you: removing what you made without leaving orphans. Get this and you can model any connection a requirement throws at you.

Today's win — the M05 done-when, all of it: a 1:N to account (✓ L02), an N:N, a child row through the lookup (✓ L04), and a clean teardown.

01 Create the N:N: create-many-to-many

A project uses many skills; a skill serves many projects. That's N:N — and Dataverse stores it in an intersect table (L01). The verb names both ends and the intersect table; there's no lookup, because neither side owns the other:create-many-to-many

agent-cloud · exit 0crm --json metadata create-many-to-many \
    --schema-name ag_ag_project_ag_skill \
    --entity1 ag_project --entity2 ag_skill \
    --intersect-entity ag_project_skill

{ "ok": true, "data": {
    "created": true, "kind": "ManyToMany",
    "schema_name": "ag_ag_project_ag_skill",
    "intersect_entity": "ag_project_skill",   # the hidden join table
    "relationship_id": "27175ea1-926a-…",
    "solution": "itworx_dev", "published": true } }

Note what's missing vs L02: no --lookup-schema, no CascadeConfiguration. N:N rows are peers — there's no parent to cascade from, and no single column could hold "many." entity1 and entity2 are interchangeable; the relationship is reciprocal.

02 Link two rows: entity associate

You don't insert into the intersect table by hand, and there's no @odata.bind on create (that's for single lookups). For N:N you associate two existing rows — Dataverse writes the GUID pair for you:associate

agent-cloud · exit 0crm --json entity associate \
    ag_projects 44b3e97b-… \
    ag_ag_project_ag_skill \              # NAV = the relationship schema name
    ag_skills af368db2-…
#   args: TARGET_SET TARGET_ID  NAV  RELATED_SET RELATED_ID

{ "ok": true, "data": { "associated": true,
    "target": "44b3e97b-…", "related": "af368db2-…" } }

Verify by expanding the collection-valued nav property — note it returns an array (many), unlike the single object a 1:N lookup expand returned in L04:

agent-cloud · exit 0crm --json query odata ag_projects --select ag_name \
    --expand "ag_ag_project_ag_skill(\$select=ag_name)"

[ { "ag_name": "Apollo Migration",
    "ag_ag_project_ag_skill": [ { "ag_name": "Data Migration" } ] } ]
✓
disassociate is the inverse. Same arguments — it deletes the one intersect row (the link), never the records themselves. Associating and disassociating only ever touch the join table.

03 Choose: 1:N or N:N?

This is the judgment the whole milestone was for. The deciding question is simple: can a child belong to more than one parent at once?

Ask1:N (lookup)N:N (intersect)
Child belongs to…one parent at a timemany peers at once
Physical structurea lookup column on the childa hidden intersect table
Set the link by@odata.bind / set-lookupentity associate
Cascade / ownershipyes — parental/referential (L03)none — rows are peers
Can carry extra data on the link?n/ano — pure join (use a manual intersect table if you need data)
Exampleproject → accountproject ↔ skills
!
If the link itself needs data, neither stock type fits. "Project ↔ employee with a role and an allocation %" can't live on a peer-only N:N intersect table. The pattern is a manual intersect: a real custom table (e.g. ag_assignment) with two 1:N lookups — back to L02. Plain N:N is for pure "which connects to which," nothing more.

04 Tear it down without orphans

Cleanup is a first-class skill, not an afterthought. delete-relationship removes a 1:N or N:N by schema name — and for a 1:N it also removes the lookup column. Preview first with --dry-run --check-dependencies to see whether anything blocks the delete:delete-relationship

agent-cloud · exit 0 · dry-runcrm --json --dry-run metadata delete-relationship \
    ag_account_ag_project --check-dependencies --yes

{ "ok": true, "data": {
    "would_delete": true, "schema_name": "ag_account_ag_project",
    "can_delete": true, "blockers": [] } }

can_delete: true and an empty blockers[] mean it's safe. A non-empty blockers list (e.g. the lookup is on a form, or a dependency exists) tells you what to remove first. Then the real teardown — order matters: relationships before the tables they connect:

agent-cloud · exit 0crm --json metadata delete-relationship ag_ag_project_ag_skill --yes   # N:N first
crm --json metadata delete-relationship ag_account_ag_project --yes  # then 1:N
crm --json metadata delete-entity ag_project --yes   # then the tables
crm --json metadata delete-entity ag_skill   --yes

# each returns {deleted: true, schema_name|logical_name, solution}
!
In JSON mode, deletes need --yes. Without it the CLI returns "aborted by user" — there's no interactive prompt to answer in an agent flow. (Same rule you met for delete-entity in M04 L05.)

05 Capstone — the full M05 round-trip ~12 min writes

Two tables, both relationship types, real links, then a clean org. This is the milestone's done-when:

# 1 · two custom tables to relate
crm --json metadata create-entity --display "Project"
crm --json metadata create-entity --display "Skill"

# 2 · 1:N account → project (L02)
crm --json metadata create-one-to-many \
    --schema-name ag_account_ag_project --referenced-entity account \
    --referencing-entity ag_project \
    --lookup-schema ag_AccountId --lookup-display "Account"

# 3 · N:N project ↔ skill (this lesson)
crm --json metadata create-many-to-many \
    --schema-name ag_ag_project_ag_skill \
    --entity1 ag_project --entity2 ag_skill --intersect-entity ag_project_skill

# 4 · a project bound to an account (L04) + a skill row
crm --json entity create ag_projects --data \
  '{"ag_name":"Apollo","ag_AccountId@odata.bind":"/accounts(ACCT-GUID)"}'
crm --json entity create ag_skills --data '{"ag_name":"Data Migration"}'

# 5 · link the project to the skill via N:N
crm --json entity associate ag_projects PROJ-GUID \
    ag_ag_project_ag_skill ag_skills SKILL-GUID

# 6 · verify both links
crm --json query odata ag_projects --select ag_name \
    --expand "ag_AccountId(\$select=name)"
crm --json query odata ag_projects --select ag_name \
    --expand "ag_ag_project_ag_skill(\$select=ag_name)"

# 7 · clean up: relationships, then tables (preview the 1:N delete first)
crm --json --dry-run metadata delete-relationship ag_account_ag_project --check-dependencies --yes
crm --json metadata delete-relationship ag_ag_project_ag_skill --yes
crm --json metadata delete-relationship ag_account_ag_project --yes
crm --json metadata delete-entity ag_project --yes
crm --json metadata delete-entity ag_skill --yes
  1. In step 3's envelope, find intersect_entity. Why is there no referencing_attribute like there was for the 1:N?
  2. Compare the two expands in step 6: one returns an object, one returns an array. Which is which, and why?
  3. In step 7's dry-run, is can_delete true even though a project row still uses the lookup? What does that tell you about the default cascade?
  4. After cleanup, run metadata entities and confirm ag_project/ag_skill are gone — the org is exactly as you found it.

06 Check yourself

Five questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · create-many-to-many needs which argument that create-one-to-many does not?
Correct. N:N is stored in an intersect (join) table, so you name it with --intersect-entity. There's no lookup column and no cascade — those are 1:N concepts.
N:N needs --intersect-entity (the join table). It has no lookup or cascade; --referenced-entity is a 1:N flag.
2 · To link an existing project and skill through the N:N, you…
Correct. N:N links are made with entity associate (collection-valued nav = the relationship schema name); Dataverse writes the intersect pair. Bind/set-lookup are for single 1:N lookups.
associate is the N:N tool — it writes the intersect pair for you. @odata.bind and set-lookup only set a single 1:N lookup.
3 · A project must belong to exactly one account; an account has many projects. Which type?
Correct. "Child belongs to exactly one parent" is the textbook 1:N — a single lookup column on the child. N:N is for many-on-both-sides.
One parent per child = 1:N (a lookup). N:N is only when both sides can have many.
4 · "Assign employees to projects, each assignment with a role and an allocation %." What do you build?
Correct. The link carries data (role, %), which a peer-only N:N intersect table can't hold. Make a real table with two lookups — a manual intersect.
A stock N:N intersect can't store role/%. When the link itself has attributes, build a custom table with two 1:N lookups (a manual intersect).
5 · Before deleting a 1:N relationship, what does --dry-run --check-dependencies give you?
Correct. The dry-run reports whether the delete can proceed and lists any blockers (forms, dependencies) — no write. Empty blockers + can_delete: true means go.
It previews the delete: would_delete, can_delete, and a blockers[] list — nothing is removed. You fix blockers, then delete for real.
