Managed layers,
layer conflicts, and GUI limits

The capstone: understand the managed/unmanaged layer stack, detect conflicts, and know where the CLI stops.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads cleanup
Why this matters: You've built, exported, and imported. The last piece of ALM you'll hit in real projects is the layer conflict: a managed solution ships a table, and someone adds an unmanaged customization on top of it. Now there are two layers — and moving the managed solution between orgs without resolving the conflict causes silent overwrites. layer-conflicts is the read-only diagnostic for this.

Every component in D365 can have two layers: a managed layer (from a managed solution) and an unmanaged layer (your customizations on top). Layer conflicts are where those two layers fight.

01 The layer model

Dataverse applies component definitions in stack order, bottom-up:

                         ┌─ unmanaged layer (your dev customizations)
Component: itworx_project ─┤
                         └─ managed layer (itworx_dev managed install)
  • A managed solution imports a table → that's the managed layer. It's read-only and version-controlled by the publisher.
  • You add a column or change a form on top of that table in your dev org → that's the unmanaged layer. It's your customization, living above the managed base.
  • Dataverse resolves the final component definition by merging the layers, with the unmanaged layer winning on fields it touches.
  • When the managed solution is upgraded, the managed layer changes. If your unmanaged layer references properties that the new managed version also changes, you have a layer conflict.
TIP
One-layer is cleaner. If you only ever have an unmanaged layer (all your own work, no ISV managed layer on top), you'll never see layer conflicts. They appear when you mix: install an ISV solution, then customize on top of it.

02 Detect conflicts: layer-conflicts

Pass the managed solution (the base layer) and your unmanaged solution (the customization layer). The command reads from the org — it doesn't change anything.

FlagPurpose
--solutionThe managed solution whose components form the base layer.
--unmanaged-solutionYour unmanaged solution whose components sit on top.
 
shellcrm --json solution layer-conflicts \
    --solution AccessTeam \
    --unmanaged-solution itworx_dev
envelope · live-verified 2026-06-18{
  "ok": true,
  "data": [],
  "meta": { "count": 0 }
}

Empty data[] with meta.count: 0 means no components overlap — itworx_dev's components don't touch anything that AccessTeam owns. Clean.

If there were conflicts, data[] would list each conflicting component by type and ID, like this (illustrative — not from live run):

// hypothetical conflict entry:
{
  "component_type": 1,
  "component_id": "246395a6-256b-f111-ab0c-70a8a5af9b7e",
  "schema_name": "itworx_project",
  "managed_solution": "AccessTeam",
  "unmanaged_solution": "itworx_dev"
}

On a real conflict: decide whether to keep your unmanaged customization (and live with the merge risk on upgrade) or remove it and let the managed layer be the source of truth.

03 Bump the version: set-version

Before shipping a solution to another org, bump the version so import history is unambiguous. The 4-part version (major.minor.build.revision) is what appears in the target org's Solutions UI and in solution.xml.

shellcrm --json solution set-version itworx_dev --version 1.0.1.0
envelope · extrapolated from pattern — not live-verified{
  "ok": true,
  "data": {
    "updated": true,
    "uniquename": "itworx_dev",
    "version": "1.0.1.0"
  }
}
TIP
ALM convention: bump revision for hot fixes, build for sprint patches, minor for feature drops, major for breaking changes. Managed solution upgrades on the target org require a strictly higher version than what's installed.

04 What the CLI does not automate

The crm CLI covers create → export → import. These tasks remain GUI-only or belong to a separate toolchain:

TaskWhy CLI can't do it
Solution Checker (quality scan) Requires the Power Apps Checker API — a separate REST service, not the Dataverse Web API. The crm CLI doesn't wrap it.
Solution history timeline Read from the SolutionHistoryData entity (undocumented/preview endpoint). Not exposed in the crm CLI.
Visual ZIP comparison Offline diff territory: extract customizations.xml from both ZIPs, then run git diff --no-index or SolutionPackager.
Solution upgrade UI (holding solution) solution stage-and-upgrade exists in the CLI but is advanced ALM; the guided UI walkthrough is only in the maker portal.
Patch management (clone-as-patch) solution clone-as-patch exists, but patch strategy is complex enough to need architectural planning before scripting.
Automated test coverage EasyRepro / Playwright against the Dataverse Test SDK — an entirely separate testing toolchain.
TIP
The CLI gets you through create → export → import. The remaining ALM surface — quality gates, history, upgrade strategy — lives in the Power Platform admin toolchain: Power Platform CLI (pac), Deployment Manager, and the maker portal.

05 Clean up the lesson state

The M06 done-when is satisfied. Now tear down the lesson solution so the org is clean for M07.

WARN
Order matters. Delete the entity before uninstalling the solution. solution uninstall removes the container, not the components. If you uninstall first and the entity still exists, the entity stays on the org as an orphan under no solution.
shell# Step 1: delete the entity (component inside the solution)
crm --json metadata delete-entity itworx_project --yes

# Step 2: uninstall the solution container (--solution is a named flag)
crm --json solution uninstall --solution itworx_dev --yes

delete-entity and solution uninstall envelopes:

Note on the publisher: The itworx publisher persists on the org — publishers are org-level resources not tied to a single solution. You'd only delete a publisher if you were certain nothing else uses that prefix.

06 M06 capstone: the full round-trip ~20 min writes

Run this script start-to-finish to lock in the muscle memory. Each step is annotated with what to verify.

capstone · full round-trip# 1 · Publisher
crm --json solution create-publisher \
    --name itworx --display "ITWorx Publisher" \
    --prefix itworx --option-value-prefix 72000 \
    --if-exists skip --no-set-default

# 2 · Solution
crm --json solution create \
    --name itworx_dev --display "ITWorx Dev Solution" \
    --version 1.0.0.0 --publisher itworx \
    --if-exists skip --no-set-default

# 3 · Table (the custom component)
crm --json metadata create-entity \
    --schema-name itworx_Project --display "ITWorx Project" \
    --solution itworx_dev

# 4 · Register as solution component
#     (grab MetadataId from step 3's metadata_id_url)
crm --json solution add-component \
    --solution itworx_dev --type 1 \
    --id <MetadataId from step 3>

# 5 · Inspect components
crm --json solution components itworx_dev

# 6 · Export
crm --json solution export itworx_dev -o itworx_dev.zip

# 7 · Import (re-import same org to prove the round-trip)
crm --json solution import itworx_dev.zip --overwrite --yes

# 8 · Check for layer conflicts
crm --json solution layer-conflicts \
    --solution AccessTeam --unmanaged-solution itworx_dev

# 9 · Clean up
crm --json metadata delete-entity itworx_project --yes
crm --json solution uninstall --solution itworx_dev --yes

Step-through questions

  1. In step 3's envelope, where is the MetadataId you need for step 4? (Hint: look at metadata_id_url.)
  2. After step 6, run unzip -l itworx_dev.zip. How many files are in the ZIP?
  3. In step 7's components[] array, which component type appears the most?
  4. After step 9, run crm --json solution list. Is itworx_dev still there?

Lesson check

4 questions · click to reveal

1. A managed solution installs itworx_project. Your dev org adds an extra column to it. What does layer-conflicts detect?

Correct. Layer conflicts are about the same component existing in two layers simultaneously: the managed base and your unmanaged customization on top. layer-conflicts reports exactly that overlap.
Not quite. layer-conflicts is about component-layer overlap — the same component present in both a managed solution (base layer) and an unmanaged solution (customization layer). Publisher or solution name duplicates are different errors.

2. You run solution uninstall --solution itworx_dev. What does this remove?

Correct. solution uninstall for an unmanaged solution removes the container — the bucket — but leaves the components (tables, columns, forms) on the org. You must delete them separately if you want a clean teardown. For managed solutions, uninstall does remove the managed-layer components cleanly.
Not quite. For an unmanaged solution, uninstall only removes the container. The components stay on the org. That's why the capstone script deletes the entity before uninstalling the solution.

3. You want to ship to production as a read-only upgradeable layer. Which export flag do you add?

Correct. --managed on export bakes the managed flag into the ZIP. When imported to prod, the components install as a managed layer — read-only in the customization UI, versioned, and cleanly upgradeable. There's no --managed-only flag.
Not quite. --managed is the flag. --publish triggers post-import publish, --overwrite is an import flag (not export), and --managed-only doesn't exist.

4. Solution Checker quality scans are NOT in the crm CLI. Which service provides them?

Correct. Solution Checker is backed by the Power Apps Checker API — a distinct REST service separate from the Dataverse Web API. It runs a rule set against the solution ZIP and reports issues by component. The crm CLI doesn't wrap it.
Not quite. Solution Checker lives in the Power Apps Checker API — a completely separate service from the Dataverse Web API. The $metadata endpoint describes the schema but doesn't run quality rules. The crm CLI has no validate command for ZIPs, and solution dependencies is about dependency graphs, not quality analysis.
