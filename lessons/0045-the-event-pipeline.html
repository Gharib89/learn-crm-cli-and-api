The event
pipeline

Pre-validation, pre-operation, post-operation — the three stages D365 routes SDK messages through. Register assemblies, steps, and images via the crm CLI.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads only
Why this matters: Plug-ins are the most powerful server-side automation tool in D365. They run inside the platform, execute synchronously or asynchronously, and can read/modify every field on the record before or after it's saved. Understanding the pipeline stages determines where your logic fires and what data it can see. The crm CLI registers and manages plug-in registrations — the .NET code itself lives outside.

The event pipeline is D365's backbone for server-side logic. Every SDK message — Create, Update, Delete, Retrieve — flows through three ordered stages, each with different transaction guarantees and capabilities. Knowing which stage to use is the first design decision in any plug-in project.

01 The three pipeline stages

When D365 receives an SDK message, it routes it through a chain of registered handlers. The chain has three fixed stages. The stage determines transaction membership, what data is available, and whether async execution is allowed.

pipeline diagramSDK message arrives (e.g. Create contact)
      ↓
┌─────────────────────────────────────────────────────────┐
│ Pre-validation (stage 10)  · outside the DB transaction │
│   · Sync only · can abort before any DB work begins     │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│ Pre-operation (stage 20)   · inside the transaction     │
│   · Sync only · can modify InputParameters.Target       │
│   · DB not written yet                                  │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│  Platform writes record to database                     │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│ Post-operation (stage 40)  · inside or after tx         │
│   · Sync OR async · can read saved data                 │
│   · Async queues a system job (caller not blocked)      │
└─────────────────────────────────────────────────────────┘
StageNumberTransactionModeDB stateCan modify Target?
Pre-validation 10 Outside — aborts cheaply, no rollback needed Sync only Not yet touched Yes
Pre-operation 20 Inside — exception rolls back Sync only Not yet written Yes — primary use case
Post-operation 40 Inside (sync) or after (async) Sync or async Written (readable) No — record already saved

Key rules:

  • Sync plug-ins block the caller until they complete — use for validation and field modification.
  • Async plug-ins queue a system job (asynchronous operation); the caller gets a response immediately without waiting.
  • Async requires stage = postoperation — this is a platform constraint. Attempting --stage preoperation --mode async is rejected.
  • Pre-image is available on pre-operation and post-operation steps; post-image requires post-operation.
TIP
Choose your stage by what you need to do: Abort before any DB work? Use pre-validation. Modify the incoming record before it's saved? Use pre-operation. React to a saved record or notify an external system? Use post-operation (async recommended for external calls).

03 register-assembly — upload your .dll

The registration lifecycle is: register assembly → register step(s) → register image(s). The assembly upload is the first step — the CLI reads your .dll from disk and stores it in the D365 database.

Command shape (concept walkthrough — requires a compiled .dll):

shellcrm --json plugin register-assembly /path/to/MyPlugin.dll \
    --name "MyPlugin" \
    --isolation-mode sandbox \
    --solution itworx_dev
conceptual envelope{ "ok": true, "data": {
    "pluginassemblyid": "a1b2c3d4-0045-0000-0000-000000000003",
    "name": "MyPlugin",
    "version": "1.0.0.0",
    "isolation_mode": 2,
    "solution": "itworx_dev"
}}
FlagDescription
--nameAssembly name — defaults to the file stem if omitted
--versionVersion string (default 1.0.0.0)
--isolation-mode sandbox|nonesandbox=2 (restricted, default, use in production); none=1 (no restrictions, on-prem debugging only)
--updatePATCH an existing assembly by name — uploads a new .dll without re-registering steps
--solutionTarget solution to include this assembly component
WARN
Sandbox is the default and the right choice for cloud. isolation-mode none removes all platform restrictions and is only for on-premises debugging. Always use sandbox in production — cloud orgs enforce it.

04 register-step — bind to a message and stage

A step connects an assembly type to a specific SDK message + entity + stage. This is the declaration that tells the platform "when this message fires on this entity, call my code at this stage."

FlagRequired?Description
--message TEXTYesSDK message: Create, Update, Delete, Retrieve, RetrieveMultiple, etc.
--plugin-type TEXTOne of these twoFully qualified class name (exclusive with --service-endpoint)
--service-endpoint TEXTOne of these twoWebhook name (exclusive with --plugin-type) — covered in L04
--entity TEXTNoPrimary entity logical name; omit = message-level step (all entities)
--stage prevalidation|preoperation|postoperationNoPipeline stage (default postoperation)
--mode sync|asyncNoExecution mode (default sync). Async requires --stage postoperation.
--filtering-attributes TEXTNoComma-separated attribute names; step fires ONLY when these attrs are in the Update payload. Omit = always fires.
--async-auto-deleteNoAuto-delete system job on success (async steps only)
--name TEXTNoExplicit step name
--rank INTEGERNoExecution order within the same stage (default 1)

Example 1 — pre-operation sync step (field validation)

shellcrm --json plugin register-step \
    --message Create \
    --plugin-type "MyNamespace.MyPlugin" \
    --entity contact \
    --stage preoperation \
    --mode sync \
    --solution itworx_dev

Example 2 — post-operation async step with filtering

shellcrm --json plugin register-step \
    --message Update \
    --plugin-type "MyNamespace.MyPlugin" \
    --entity contact \
    --stage postoperation \
    --mode async \
    --filtering-attributes emailaddress1 \
    --async-auto-delete \
    --solution itworx_dev
WARN
Always use --filtering-attributes on Update steps. The most common performance mistake is registering an Update step without filtering — the step fires on every single update to that entity, regardless of which fields changed. Add --filtering-attributes to scope it to the columns that actually matter for your logic.

05 register-image — snapshot the record

Images capture the record state at a fixed point in the pipeline and make it available inside your plug-in's execution context. Without an image, you only see the fields in the current operation's payload — not the full record.

Image typeWhen availableContents
prePre-operation and post-operation stepsRecord state BEFORE the operation — the "before" snapshot
postPost-operation steps onlyRecord state AFTER the operation — the "after" snapshot
bothPost-operation steps onlyBoth pre and post snapshots on a single registration
 
shellcrm --json plugin register-image \
    --step "a1b2c3d4-0045-0000-0000-000000000005" \
    --type pre \
    --alias "PreImage" \
    --attributes firstname,lastname,emailaddress1 \
    --solution itworx_dev

The --attributes flag restricts the image to only the listed columns (lighter payload). Omit it to capture all attributes — avoid this in production as it significantly increases message size.

Enable, disable, and teardown

shell# Enable/disable a step (non-destructive)
crm --json plugin set-step-state <stepid> enable
crm --json plugin set-step-state <stepid> disable

# Teardown — order matters: image → step → assembly
crm --json plugin unregister-image <imageid> --yes
crm --json plugin unregister-step <stepid> --yes
crm --json plugin unregister-assembly MyPlugin --yes
TIP
Teardown order matters. Always remove images before steps and steps before the assembly. The platform enforces referential integrity — unregistering an assembly with active steps will fail.

Lesson check

4 questions · click to reveal

1. Which pipeline stage runs INSIDE the database transaction, BEFORE the record is written to the DB?

Correct. Pre-operation (stage 20) is inside the DB transaction and fires before the record is written. This makes it the right stage for modifying InputParameters.Target — any changes you make will be persisted when the transaction commits.
Pre-operation (stage 20) is the one inside the transaction before the DB write. Pre-validation (10) is outside the transaction. Post-operation (40) fires after the write. Stage 10 is the cheapest abort; stage 20 rolls back if it throws.

2. You want a step to fire ONLY when the emailaddress1 field is updated, not on every update to the contact. Which flag?

Correct. --filtering-attributes scopes an Update step to only fire when the listed attributes appear in the Update payload. Without it, the step fires on every update to the entity — a common and costly performance mistake.
--filtering-attributes emailaddress1 is the correct flag. --entity sets the entity type, not a field filter. --configuration passes unsecure config to the plug-in class. --message sets the SDK message (e.g. Update).

3. You try --stage preoperation --mode async. What happens?

Correct. Async mode is a platform constraint: it requires --stage postoperation. The reason is architectural — async steps queue a system job and return immediately, which only makes sense after the record is saved (postoperation).
This combination is rejected. Async requires --stage postoperation — it's a platform constraint, not a CLI restriction. Async steps queue a system job after the record is saved; pre-operation and pre-validation are synchronous-only stages.

4. After registering an assembly, what MUST you register before the platform can call your code?

Correct. A step is the declaration that binds your assembly type to a specific message + entity + stage. Without a step, the platform has no instruction to call your code — the assembly is registered but inert.
A step is required. The step is the binding that tells the event pipeline "when this message fires, call this type at this stage." An assembly alone does nothing. Images are optional — they add record snapshots to an existing step.
