Update a
Record

Change a row in place with PATCH — a scalar, an option-set, and a lookup — and watch the preview show exactly which keys move.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. In L01 you learned to read a record's true shape — codes, GUIDs, the _x_value lookups. Updating is the mirror image: you now write back in that same vocabulary. An update speaks exactly the anatomy you just decoded — set an option-set by its code, point a lookup at another row by GUID and table. This is where reading pays off.

Today's win: run the full safe-write loop on a real contact — preview → commit → verify → revert — changing a text field, a choice, and a relationship, leaving the org exactly as you found it.

01 Update is a surgical PATCH

Create was a POST — you hand the server a whole new row. Update is a PATCH — you hand it only the keys that change, against a record that already exists. Everything you don't mention is left untouched. Preview it first, exactly like any write (M01 L04):update

agent-cloud · dry-run · exit 0crm --json --dry-run entity update contacts a1b2c3d4-1111-2222-3333-00155d467b91 \
    --data '{"jobtitle":"Senior Sales Manager","gendercode":1}'

{ "ok": true, "data": {
  "_dry_run": true,
  "method": "PATCH",
  "url": "https://orgexample.crm.dynamics.com/api/data/v9.2/contacts(a1b2c3d4-…b91)",
  "headers": { … "If-Match": "*" },
  "body": { "jobtitle": "Senior Sales Manager", "gendercode": 1 }
} }, "meta": { "dry_run": true } }

Three things to read off that plan, all of which separate update from create:

  • PATCH, not POST — the verb that says "modify this existing GUID," not "make a new one." (Sending a POST when you meant a PATCH was L04's duplicate footgun.)
  • The body is only your two keys. A PATCH is a partial update — the other ~130 columns on the contact are not sent and not changed.
  • If-Match: "*" appears for free — that's the safety net, next section.
✓
Update is silent. Where create echoes the whole new row back by default, update returns just the record id — no row body. Add --return-record to get the updated row, or just entity get it afterward (you'll do that to verify).

02 It won't create by accident

That If-Match: "*" header is doing real work. In the raw Web API, a PATCH to a record that doesn't exist will happily create it (an "upsert") — so a typo'd GUID could silently make a new row instead of editing the one you meant. crm blocks that by default: If-Match: "*" means "only proceed if a row with this id already exists."upsert

default
Update-only. If-Match: "*" sent → a missing id fails (no accidental create). This is what you want 99% of the time.
--allow-create
Opt into upsert. Drops the header → PATCH-creates if the id is absent. (Or use the dedicated entity upsert verb.)
--if-match 'W/"…"'
Optimistic concurrency. Pass a row's version etag → the update only lands if it hasn't changed since you read it. (crm v4.7.0 no longer prints the etag in --json output, so in practice you use --if-match "*" — "any current version" — and let the server enforce the check.)

03 Writing each field shape

This is the L01 payoff. Each of the three field shapes is written a particular way — and the lookup is the one that trips everyone:

Plain column
Put it straight in --data: "jobtitle": "Senior Sales Manager". The value is the value.
Option-set
Send the code, never the label: "gendercode": 1 — not "Male". (That's why L01 had you read the stored code.) The wrong shape is a common 400.
Lookup
You cannot write _parentcustomerid_value — it's read-only. You set the relationship through the navigation property with @odata.bind, naming the related set + GUID:
"parentcustomerid_account@odata.bind": "/accounts(<guid>)". GUID + table, never a name.bind

The lookup form is fiddly, so crm gives you two purpose-built verbs that build the @odata.bind PATCH (and the clear) for you — no hand-written JSON:skill

agent-cloud · exit 0# set the lookup: contact → parent account, by nav-property + related set + id
crm --json entity set-lookup contacts <contact-id> parentcustomerid_account accounts <account-id>

# clear it again (DELETE on the /$ref)
crm --json entity clear-lookup contacts <contact-id> parentcustomerid_account
{ "ok": true, "data": { "cleared": true, "nav": "parentcustomerid_account" } }
!
Where does parentcustomerid_account come from? It's the navigation property for that lookup — and you already saw it in L01: it's the associatednavigationproperty annotation the server returned when you read the lookup. parentcustomerid is a customer lookup (it can point at an account or a contact), so the nav property carries the target table as a suffix: parentcustomerid_account vs parentcustomerid_contact. Read the annotation to learn the name; don't guess it.

04 The loop, for real

Here's the whole safe-write loop run live on agent-cloud — one contact, all three shapes at once, then put back. Same four beats as L05, now for an update:

1 · preview
--dry-run
read the PATCH plan (§01)
2 · commit
drop --dry-run
silent — id only
3 · verify
entity get
confirm it landed
4 · revert
restore
leave org clean
commit · exit 0crm --json entity update contacts a1b2c3d4-…b91 \
    --data '{"jobtitle":"Senior Sales Manager","gendercode":1,
            "parentcustomerid_account@odata.bind":"/accounts(0c1d2e3f-…cf0)"}'

{ "ok": true, "data": { "_entity_id": "a1b2c3d4-…b91" } }   # silent: id only
verify · exit 0crm --json entity get contacts a1b2c3d4-…b91 --annotations \
    --select jobtitle,gendercode,_parentcustomerid_value

{ "jobtitle": "Senior Sales Manager",                          # ✓ text changed
  "gendercode": 1, "gendercode@…FormattedValue": "Male",             # ✓ choice flipped 2→1
  "_parentcustomerid_value": "0c1d2e3f-…cf0",                       # ✓ lookup now set
  "_parentcustomerid_value@…FormattedValue": "Contoso Ltd",
  "_parentcustomerid_value@…lookuplogicalname": "account" }

The verify read is the proof: the option-set code moved and its label tracked, and the lookup resolved to the right row in the right table (lookuplogicalname: "account"). Then revert — restore the two scalars and clear the lookup — so the org ends exactly where it started:

revert · exit 0crm --json entity clear-lookup contacts a1b2c3d4-…b91 parentcustomerid_account
crm --json entity update contacts a1b2c3d4-…b91 --data '{"jobtitle":"Sales Manager","gendercode":2}'
✓
Revert is just another update. There's no "undo" in Dataverse — you put a record back by writing the old values over the new ones (and clearing what you set). Which is exactly why step 1 reads the originals before you touch anything.

05 Do it yourself ~5 min

A real, fully reversible round-trip on a seed contact. Read the originals first so you can restore them:

# 0 · pick a contact + READ ORIGINALS (write these down)
crm --json query odata contacts --top 3 --select fullname,jobtitle,gendercode
crm --json entity get contacts <id> --select jobtitle,gendercode

# 1 · PREVIEW the change (no write yet)
crm --json --dry-run entity update contacts <id> --data '{"jobtitle":"Lead Analyst","gendercode":1}'

# 2 · COMMIT (same command, --dry-run removed)
crm --json entity update contacts <id> --data '{"jobtitle":"Lead Analyst","gendercode":1}'

# 3 · VERIFY it landed
crm --json entity get contacts <id> --annotations --select jobtitle,gendercode

# 4 · REVERT to the values from step 0
crm --json entity update contacts <id> --data '{"jobtitle":"<original>","gendercode":<original>}'

Then answer, from your own output:

  1. In step 1's preview, what method and what If-Match header? What's in the body — all the contact's columns, or just your two?
  2. Step 2's response — does it contain the updated row, or only the id? Why? (What flag would return the row?)
  3. Bonus: set a lookup with entity set-lookup contacts <id> parentcustomerid_account accounts <account-id>, verify with entity get … --annotations, then entity clear-lookup it. What does lookuplogicalname read?

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You entity update a contact that has 40+ columns, sending --data '{"jobtitle":"Lead Analyst"}'. What's in the PATCH body?
Correct. A PATCH sends only the keys you name; the other columns are neither sent nor changed. That's why update is "surgical" — you can't accidentally blank a field you didn't mention.
Not quite. PATCH is a partial update — the body is exactly the keys in your --data (here, just jobtitle). Unmentioned columns are untouched, not resent and not wiped.
2 · crm entity update sends If-Match: "*" by default. What does that buy you, and what removes it?
Correct. A raw PATCH will create a row if the id is missing (upsert). If-Match: "*" says "only if it already exists," so a typo'd GUID errors instead of spawning a stray record. Opt into create with --allow-create or the upsert verb.
Re-read §02. If-Match: "*" makes the update update-only — it blocks the accidental create-if-missing (upsert) that a bare PATCH would do. You opt back into create with --allow-create or entity upsert.
3 · You want to set a contact's parent account (a lookup). Which payload is correct?
Correct. Lookups are written through the navigation property with @odata.bind, pointing at the related set + GUID. The _x_value field is read-only; you never set it directly, and never by display name. (Or skip the JSON entirely: entity set-lookup.)
No — _parentcustomerid_value is read-only, and a lookup is never set by name or by a bare GUID. The correct shape is the nav property + @odata.bind → "/accounts(<guid>)" (or use entity set-lookup).
4 · After a successful entity update you get { "ok": true, "data": { "_entity_id": "…" } } — no jobtitle, no row. Something wrong?
Correct. ok:true means it landed; update just doesn't echo the row back the way create does. Ask for it with --return-record, or read it with entity get — which is your verify step anyway.
It worked — ok:true. Unlike create, update is silent: it returns only the id, no row body. Add --return-record to get the row, or entity get to verify. --select is a read option, not a write one.
