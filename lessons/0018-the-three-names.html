Milestone 03 / Metadata Model / Lesson 01

The three
names

Every table in Dataverse answers to three internal names plus a display label. Knowing which is which ends the guesswork behind every command you've typed.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. All through M01–M02 you typed contacts to query but contact to count — and never learned why. The answer is that a table has more than one name, each for a different job. This milestone is about reading the schema instead of memorising it; the naming triad is the first thing the metadata API hands you, and it unlocks every command that follows.

Today's win: ask the org to list its tables, and read the three names each one carries — so you always know which to type where.

01 One table, three names (plus a label)

A Dataverse table isn't identified by a single string. Behind the friendly word "Account" sit three machine names, each owning a different context:

LogicalName
All-lowercase internal id — account, contact. The canonical name. Used by metadata, FetchXML <entity name>, query count, and inside $filter conditions.
SchemaName
The same name in PascalCase — Account, Contact. A design-time identity: it's how the table appears in solution XML and how relationship/column schema names are built (new_ prefixes, casing). You rarely type it, but it's what the maker portal shows as "Name".
EntitySetName
The plural form — accounts, contacts. This is the URL path segment in the Web API, so it's what query odata and entity get/create use.
DisplayName
The localized human label — "Account", "Microsoft Entra ID". Lives per language (1033 = English). For people and forms, never for code — it can be renamed or translated out from under you.

The first three are stable contracts; the fourth is cosmetic. Mixing them up is the single most common beginner error — and the cause of the 400s you'd hit guessing contact in a URL.

Two flags shape the list:

FlagEffect
--top NCap the rows (the org has hundreds of system tables — start small).
--custom-onlyHide the stock Microsoft tables; show only what's been added. IsCustomEntity:true marks each — though note "custom" includes managed add-ins like aaduser, not just your tables.

The DisplayName arrives as a fat localized-label object (every language it's translated into). The build trims it above — the live JSON nests the English string under DisplayName.UserLocalizedLabel.Label.

03 Which name goes where

This is the table to burn in. Every command you've run already obeys it — now you can see the rule behind them:

ContextWantsExample
query odata · entity get/createEntitySetNamequery odata contacts
query count · metadata * · $filter fieldsLogicalNamequery count contact
FetchXML <entity name>LogicalName<entity name="contact">
Solution XML · schema designSchemaNameAccount
A human reading a formDisplayName"Account"
!
The 400 trap, explained. Back in M01 L06 you learned query count wants contact, not contacts — and a URL with the singular fails. Now you know it's not arbitrary: count and metadata speak logical, the URL path speaks entity-set. Same table, two names, two jobs.

04 Do it yourself ~3 min

All read-only on agent-cloud. Find the names, then prove the rule:

# 1 · list a few tables — read the three names of each
crm --json metadata entities --top 5

# 2 · what's been ADDED to this org? (custom + managed add-ins)
crm --json metadata entities --custom-only

# 3 · prove the naming rule with the SAME table, two ways:
crm --json query count contact          # logical name → works
crm --json query odata contacts --top 1  # entity-set → works
  1. In query 1, pick any table and say out loud its logical, schema, and set names. Which one is plural? Which is PascalCase?
  2. Run query 3 — both succeed. Now swap them (query count contacts, query odata contact) and watch both fail. Which name did each command actually want?
  3. From query 2, how many tables carry IsCustomEntity:true? Is every one of them something you'd recognise as custom?

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You're writing an OData URL to read accounts. Which name belongs in the path?
Correct. The Web API URL path is the entity-set — the plural form. That's why query odata accounts works and query odata account 404s.
The URL path is always the EntitySetName (plural) — accounts. The lowercase singular account is the logical name (metadata/count/filter), not the URL.
2 · crm metadata entities and query count both want which name?
Correct. Metadata, query count, FetchXML's <entity name>, and $filter fields all speak the canonical logical name.
Both speak the LogicalName — the lowercase singular (contact). The plural contacts is only for the URL path.
3 · Why should you never key code off the DisplayName?
Correct. DisplayName is cosmetic and per-language (1033 = English). A label change or a new translation would silently break anything that matched on it. Code keys off the stable logical/set names.
DisplayName is a localized label — translatable and renamable. It's for humans on forms; the three machine names are the stable contracts code relies on.
4 · A table shows SchemaName: "Account" but LogicalName: "account". What's the relationship?
Correct. They're the same name in two casings: SchemaName carries the design-time PascalCase (and is what solution XML uses), LogicalName is the lowercased canonical form the platform normalises to.
They're one name in two casings — SchemaName (PascalCase, design-time) and LogicalName (lowercase, canonical). The platform lowercases the schema name to get the logical name.
