FetchXML:
the other query

An XML query language for the things OData can't do — counts-by-group, deep joins, and the saved views your org already has.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. OData (L03) handles everyday filtering beautifully — but it cannot aggregate. The moment a requirement says "how many contacts per gender?" or "total revenue by city," OData runs out of road. FetchXML is Dynamics' own query language, the one Advanced Find and every saved view are built on. Knowing both — and when to reach for which — closes the Records & Queries milestone.

Today's win: run a FetchXML query, read its XML shape, and use its one superpower over OData — aggregation with group-by.

01 Two languages, one API

Dataverse speaks two query languages over the very same Web API door (M01 L02):

OData
Query options on a URL — --filter, --select, --orderby. Compact, composable, what you've used all milestone.
FetchXML
A query written as an XML document. More verbose, but more powerful — and it's the native format Dynamics stores saved views and Advanced Find in.

Both are GETs, both come back in the same envelope with rows in the data array (L03). You pass FetchXML inline with --xml, or from a file with --file:fetchxml

agent-cloud · exit 0crm --json query fetchxml --xml '<fetch top="5">
  <entity name="contact">
    <attribute name="fullname"/>
    <attribute name="jobtitle"/>
    <filter><condition attribute="gendercode" operator="eq" value="1"/></filter>
    <order attribute="fullname"/>
  </entity>
</fetch>'

{ "ok": true, "data": [
    { "fullname": "James Okafor", "jobtitle": "IT Director", "gendercode": 1 },
    { "fullname": "Robert Chen", "jobtitle": "VP of Finance", "gendercode": 1 }
  ], "meta": { "entity_set": "contacts" } }

That query is the exact twin of --filter "gendercode eq 1" --orderby "fullname" from L03 — same result, more typing. Note the anatomy:

  • <entity name="contact"> uses the logical name (singular — L01), not the entity-set. crm resolves it to contacts for you (that's meta.entity_set); pass the set as a positional arg to skip the lookup.
  • <attribute> = your --select. <filter>/<condition> = your --filter (still on the code value="1"). <order> = your --orderby.

For plain filtering, this is strictly more work than OData. So why learn it?

02 The superpower: aggregate & group

Here's what OData's Web API simply cannot do: "count the contacts in each gender." That's an aggregation with a group-by — FetchXML's home turf:aggregate

agent-cloud · exit 0crm --json query fetchxml --xml '<fetch aggregate="true">
  <entity name="contact">
    <attribute name="gendercode" alias="gender" groupby="true"/>
    <attribute name="contactid" alias="num" aggregate="count"/>
  </entity>
</fetch>'

{ "ok": true, "data": [
    { "num": 3, "gender": 2 },     # 3 contacts with gendercode 2 (Female)
    { "num": 2, "gender": 1 }      # 2 contacts with gendercode 1 (Male)
  ], "meta": { "entity_set": "contacts" } }

Three new attributes carry the whole feature:

AttributeMeaning
aggregate="true"On <fetch> — switches the whole query into aggregate mode.
aggregate="count"On an attribute — the rollup function. Also sum, avg, min, max, countcolumn.
groupby="true"Bucket the rows by this attribute (here, by gender). One result row per bucket.
alias="…"Required in aggregate mode — names each output key (num, gender above).

The result is grouped by the stored code (gender: 2/1) — to label them, map each code back as you learned in L01. This single capability — counts, sums, and averages bucketed by a field — is the most common reason to choose FetchXML.

03 When to reach for which

You now have two tools. The decision is usually quick:

Reach for…When the requirement is…
OData query odataEveryday reads: filter, select, sort, page a set of rows. Simpler to type, composable flags, easy to script. Your default.
FetchXML query fetchxml① Aggregation / group-by (count, sum, avg by bucket). ② Deep multi-table joins (<link-entity>, including left-outer). ③ Running an existing saved view — they're stored as FetchXML.
✓
You rarely hand-write FetchXML. The usual flow: build the query visually in Advanced Find (or a model-driven app's view editor), click Download FetchXML, and feed that file to crm query fetchxml --file q.xml. And a view already saved in the org runs by GUID without any XML at all — crm query saved <view-id> — because the platform stored its FetchXML for you.saved

04 Do it yourself ~4 min

All read-only. Run the aggregate, then bend it:

# 1 · count contacts per gender (the superpower)
crm --json query fetchxml --xml '<fetch aggregate="true"><entity name="contact">
  <attribute name="gendercode" alias="gender" groupby="true"/>
  <attribute name="contactid" alias="num" aggregate="count"/>
</entity></fetch>'

# 2 · the SAME question, but only ACTIVE contacts — add a filter inside <entity>
crm --json query fetchxml --xml '<fetch aggregate="true"><entity name="contact">
  <attribute name="gendercode" alias="gender" groupby="true"/>
  <attribute name="contactid" alias="num" aggregate="count"/>
  <filter><condition attribute="statecode" operator="eq" value="0"/></filter>
</entity></fetch>'

# 3 · a plain (non-aggregate) fetch — prove it equals an OData --filter
crm --json query fetchxml --xml '<fetch><entity name="contact">
  <attribute name="fullname"/><order attribute="fullname"/>
  <filter><condition attribute="gendercode" operator="eq" value="2"/></filter>
</entity></fetch>'
  1. Query 1: what keys come back, and where do num / gender come from? (Hint: the aliases.)
  2. Query 3: write the OData query odata that returns the same rows. Which was less typing?
  3. From query 1's output, which gender code has the most contacts — and what label does that code map to (L01)?

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · OData and FetchXML — how do they relate at the transport level?
Correct. Both are read queries over the same Dataverse Web API, returning the same envelope with rows in data. FetchXML is just a different (XML) way to express the query.
Both go to the same Web API as GET reads and return the same array envelope. FetchXML isn't SOAP, isn't write-only, and isn't org-type-specific — it's just an alternative query syntax.
2 · In <entity name="contact">, what is "contact"?
Correct. FetchXML names the table by its logical name (contact) — the L01/L02 singular. crm resolves it to the entity-set contacts via one metadata lookup (or you pass the set as a positional arg).
It's the logical name (singular contact) — the same internal name from L01/L02. The plural contacts is the entity-set; crm maps one to the other for you.
3 · A requirement: "how many open opportunities per owner, and their total value?" Best tool?
Correct. Counts and sums bucketed by a field = aggregation with group-by, which the OData Web API can't do. FetchXML's aggregate/groupby/sum is exactly the tool.
That's an aggregation (count + sum) grouped by owner — OData's Web API can't aggregate, and query count only gives a single cached total. FetchXML with aggregate="true" + groupby is the fit.
4 · A colleague built a view in Advanced Find that you want to run exactly as-is from the CLI. Fastest path?
Correct. System views (savedquery) are stored as FetchXML, so query saved <view-id> runs one directly — no XML to write or re-derive. (User-owned views: query user.)
No need to rebuild it. The view is already stored as FetchXML in the org; crm query saved <view-id> executes it by GUID. (For personal views, query user.)
