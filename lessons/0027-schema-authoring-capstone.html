Three ways to build a table — when to use each, how to turn a live table back into a spec, and how to delete without leaving a mess.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. You've built the same Project table three ways — imperative (L02), scaffold (L03), declarative (L04). The capstone ties them together: a decision rule for picking one, the export-spec verb that reverses a live table back into apply-ready YAML, and a disciplined delete so every practice run leaves the org exactly as you found it. That closes M04 and the milestone's "done when."

Today's win: state which authoring style fits a given task, capture a live table to a spec with export-spec, and delete it after a dependency check — org left clean.

01 Three roads — pick by the job

Use…WhenPublishes
Imperative
create-entity + add-attribute		One-off tweak; a column needing precision/format/min/max or an inline local picklist; exploring interactivelyPer call (stage with --no-publish)
Scaffold
scaffold table		A new table of plain typed columns, fast, from one command; quick demosOnce
Declarative
apply -f		Anything you want reviewed, versioned, or deployed to more than one org; multiple entities + shared option sets; the agentic defaultOnce
✓
Rule of thumb. Reach for apply -f by default — it's reviewable and idempotent. Drop to imperative only when you need a knob the shorthand/spec can't express on that column, or you're poking around. Scaffold is the quick one-liner in between. All three converge on the identical metadata — they differ in ergonomics, not result.

02 export-spec — a live table → a spec

The reverse of apply: read an existing table over pure GETs and emit an apply-consumable spec. Great for capturing something built in the GUI, or snapshotting your own work into git:export-spec

agent-cloud · exit 0crm metadata export-spec itworx_project -o project.yaml
# without -o, the spec comes back under the {ok,data} envelope instead
# --with-views and --with-relationships pull those in too

⚠ dropped column 'itworx_stage': referenced option set is unreadable (local picklist)
⚠ dropped column 'itworx_stagename': attribute type 'VirtualType' is not one apply can create

Two things to internalise from the output:

  • It warns about what it can't round-trip. A local option set isn't readable through this API path, so a local picklist column is dropped; and the read-only *name shadow columns (the VirtualType mirrors you met in M03 L02) are dropped too. Export is faithful for what apply can recreate — and honest about the rest.
  • A money column round-trips as two. Every money column auto-spawns a _Base companion (e.g. itworx_cost_Base, "Cost (Base)") holding the value in the org's base currency — the currency analogue of M03's lookup shadow columns. It shows up in the exported spec.
project.yaml (excerpt)entities:
- schema_name: itworx_Project
  attributes:
  - { kind: money, schema_name: itworx_Cost,      display_name: Cost, precision: 2 }
  - { kind: money, schema_name: itworx_cost_Base, display_name: "Cost (Base)", precision: 2 }

03 Delete cleanly — look before you destroy

delete-entity permanently removes a custom table and all its rows. Before you pull the trigger on anything real, preview blockers with --dry-run --check-dependencies:delete-entity

agent-cloud · exit 0 · dry-runcrm --json --dry-run metadata delete-entity itworx_project --check-dependencies --yes

{ "ok": true, "data": {
    "_dry_run": true, "would_delete": true,
    "can_delete": true, "blockers": [] } }    # empty = safe to delete
agent-cloud · exit 0crm --json metadata delete-entity itworx_project --yes
{ "ok": true, "data": { "deleted": true, "logical_name": "itworx_project" } }

# a GLOBAL option set is independent — it survives the table. Delete it on its own:
crm --json metadata delete-optionset itworx_priority --yes
{ "ok": true, "data": { "deleted": true, "name": "itworx_priority" } }
!
--yes is required and the delete is irreversible. Both delete-entity and delete-optionset prompt for confirmation; in --json/agent use you must pass --yes or the call aborts with "aborted by user". There's no undo — the dependency dry-run is your seatbelt. And remember: deleting the table does not delete the global option sets it used.

04 Do it yourself ~10 min writes

The full lifecycle — build declaratively, capture, then tear down cleanly:

# 1 · build from the L04 spec
crm --json apply -f project.yaml --solution itworx_dev

# 2 · capture the live table back to a fresh spec — read the warnings
crm metadata export-spec itworx_project -o roundtrip.yaml

# 3 · check dependencies before deleting
crm --json --dry-run metadata delete-entity itworx_project --check-dependencies --yes

# 4 · delete table, then the global option set
crm --json metadata delete-entity   itworx_project  --yes
crm --json metadata delete-optionset itworx_priority --yes

# 5 · confirm clean — no itworx_* custom tables remain from this run
crm --json metadata entities --custom-only
  1. Diff roundtrip.yaml (step 2) against the project.yaml you applied. What's added (hint: a _Base column) and what's missing (the local picklist, the *name shadows)? Read the export warnings to explain each.
  2. In step 3, blockers is [] and can_delete is true. In your words: when might blockers be non-empty? (Think relationships — M05.)
  3. Run step 4's delete-entity without --yes. What happens, and why does agent/JSON use require the flag?
  4. After deleting the table, run metadata get-optionset itworx_priority before deleting it. It's still there — that's why step 4 has two deletes.

05 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You need a decimal column with precision 3 and a value range, on a table you'll deploy to three orgs. Best approach?
Correct. Multi-org + reviewable points to declarative; and the spec can carry precision and ranges that scaffold's shorthand can't. (Imperative could too, but isn't reviewable/re-deployable as a file.)
Deploy-to-many + needs-precision/range = the declarative spec. Scaffold's shorthand can't express precision; the spec can, and it's the reviewable, re-deployable artifact.
2 · export-spec warns it dropped a local picklist column and two *name columns. What does that tell you?
Correct. The warnings are a feature: the spec round-trips everything apply can rebuild, and flags the rest (local picklist, read-only *name mirrors) rather than emitting something that won't apply.
Nothing was deleted and nothing failed. Export drops only what apply couldn't recreate — the unreadable local option set and the read-only VirtualType shadow columns — and tells you so.
3 · You delete a custom table that used a global option set. What happens to the option set?
Correct. A global option set is a standalone component shared across tables, so deleting one table leaves it intact. Clean it up with metadata delete-optionset --yes if nothing else uses it.
Globals are independent and reusable — they outlive any single table. Remove them explicitly with delete-optionset once nothing references them.
4 · Before deleting a real table, which call shows you whether it's safe?
Correct. The dependency dry-run reports would_delete, can_delete, and a blockers list (e.g. relationships, components depending on it) — your seatbelt before an irreversible delete.
It's delete-entity --dry-run --check-dependencies: it returns can_delete and a blockers array without deleting anything. Empty blockers = safe.
