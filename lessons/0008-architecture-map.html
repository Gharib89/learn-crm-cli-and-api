01 Four boxes, one direction

crm is built on Click (the Python CLI framework), and its code is layered. A request flows down through four responsibilities and the answer flows back up — each layer talks only to its neighbour:entry

spine Entry / spine · crm/cli.py
The Click root command. Parses global flags (--json, --dry-run, --profile), holds the per-run CLIContext, builds the backend from your active profile, and emits every envelope.
▼   dispatches to the matching group
command Command layer · crm/commands/<group>.py
One file per group (entity, query, solution…). Parses verb args/flags, calls the core function, renders the result, catches errors. Speaks terminal.
▼   calls a plain function, passing the backend
core Core layer · crm/core/<group>.py
The business logic. Builds the OData request — paths, $select/$filter, Prefer headers — validates input, then calls one backend verb. Speaks Dataverse, not terminal.
▼   backend.get / post / patch / delete
transport Transport · crm/utils/d365_backend.py
The only place that does HTTP. Auth (NTLM / OAuth), headers, the requests session, retry, paging, and turning non-2xx into a typed D365Error.
▼   HTTPS · OData v4
Dataverse Web API  /api/data/v9.x/
✓
The seam that matters: the core layer never imports Click, and the command layer never builds an HTTP request. Logic and I/O are kept apart — which is exactly why you can predict where a given bug lives.

02 Follow one command all the way down

Take a command you already know from M01:

your shellcrm --json query odata contacts --filter "lastname eq 'Khan'"

Here is the round-trip, layer by layer:query

  1. argv → spine. Click's root group cli in crm/cli.py parses the globals: --json sets json_mode on the CLIContext; no --profile, so the active profile from the session is used.
  2. Lazy dispatch. query isn't imported until now — a registry (_LazyJsonAwareGroup._lazy_commands) maps the name "query" → crm.commands.query:query_group and imports just that module.
  3. Command layer. query_odata() in crm/commands/query.py receives the parsed flags (entity_set="contacts", filter_="lastname eq 'Khan'") and calls the core function — handing it ctx.backend().
  4. Core layer. odata_query() in crm/core/query.py turns those into OData query params ($filter=…), adds any Prefer header, validates, and calls backend.get("contacts", params=…).
  5. Transport. D365Backend.get() in crm/utils/d365_backend.py attaches auth + headers, makes the HTTPS GET, follows paging, and returns parsed JSON — or raises D365Error on a non-2xx.
  6. Back up to the spine. The result returns to query_odata(), which hands it to ctx.emit(...) — the one method that prints the {ok,data,meta} envelope and sets the exit code.

Every group works this way. Swap query odata for entity create and the only things that change are which command file (commands/entity.py), which core function (core/entity.py → create()), and which backend verb (post instead of get). Same four boxes, same direction.

03 The spine: CLIContext

The single object threaded through every command is CLIContext (in crm/cli.py). It's the bridge between the M01 behavior you know and the code that produces it:entry

.json_mode / .dry_run
The global flags from L03 / L04, captured once and read everywhere downstream.
.backend()
Lazily builds a D365Backend from your active profile — resolving the profile + secret via core/connection.py (the L01 profile system). Cached per run.
.emit(ok, data, meta)
The one chokepoint that renders output: in --json mode it assembles the {ok,data,meta} envelope and, on failure, exits 1; usage errors exit 2. This is the L03 contract, in code.

So the envelope shape, the exit codes, and the profile resolution you learned as rules in M01 all live in one file. When an envelope looks wrong, emit() is where you look — not the command that called it.

✓
Error as control flow: the core/transport layers raise a typed D365Error; the command catches it and routes it through _handle_d365_error(), which classifies it (status, code, category, retryable) into meta. That's why every failure you saw in M01 had the same structured shape.

04 Why this shape — and where to look when it breaks

Two design choices are worth naming, because they're the deep-dive payoff:

  • Lazy command loading. The _lazy_commands registry imports a group's module only when you invoke it — so crm --version or crm --help stays fast and doesn't drag in requests, auth libraries, or 30 command modules it won't use.
  • Layer separation = predictable blame. Because logic (core) and I/O (command + transport) never bleed into each other, a symptom points almost uniquely at a layer. That turns "crm did something weird" into "the bug is in core/query.py" — the difference between a vague and a fileable issue.

Keep this table handy — it's the mission skill (precise upstream bug reports) in one glance:

SymptomRead this first
Unknown / mis-placed flag, usage error (exit 2)commands/<group>.py — the arg & flag definitions
Request built wrong ($filter, path, missing header)core/<group>.py — the request builder
Auth fails, retry/throttle, paging, raw HTTPutils/d365_backend.py — the transport
Envelope shape, exit code, error categorycli.py → emit() & classify_d365_error()
Wrong profile / secret picked upcore/connection.py + core/session.py

05 Do it yourself ~6 min

A source-reading drill — no org needed. Open the repo on GitHub (or your local clone) and find each thing by navigating, not searching blindly:

  1. In crm/cli.py, find the _lazy_commands dictionary. Pick any group and read its "module:attr" target — that's the file the command layer lives in.
  2. Open that group's file under crm/commands/. Find one verb function and spot the line where it calls into crm/core/. That call is the command→core seam.
  3. Open the matching crm/core/ file. Find where it calls backend.get/post/patch/delete. That's the core→transport seam — the moment logic becomes HTTP.
  4. Back in cli.py, find CLIContext.emit. Confirm with your own eyes: this is where ok, data, and the exit code are decided.
✓
The habit this builds: you can now answer "what does crm X actually do?" by reading two or three short functions — command → core → backend — instead of treating the binary as a black box.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · Which file is the only one that actually performs an HTTP request to Dataverse?
Correct. The transport layer (d365_backend.py) owns all HTTP — auth, headers, retry, paging. Core builds the request and hands it down; it never opens a socket itself.
Not quite. Core builds the request and the command parses flags, but the actual HTTP call lives in the transport layer: crm/utils/d365_backend.py.
2 · For crm --json query odata contacts --filter "…", which file turns --filter into an OData $filter param and calls backend.get()?
Correct. Building the request — params, Prefer headers, validation — is the core layer's job. commands/query.py only parsed the flag and passed it in.
Re-check §2: the command file parses --filter into an argument, but building the $filter param and calling backend.get() happens in core/query.py.
3 · Where is the {ok, data, meta} envelope assembled and the exit code (0/1/2) decided?
Correct. emit() is the single chokepoint for output: it builds the envelope, and on ok:false exits 1 (usage errors exit 2). The L03 contract lives here.
Not quite. Core returns plain data; the envelope and exit code are produced in one place — CLIContext.emit() in cli.py.
4 · Why does crm import command modules lazily (via the _lazy_commands registry)?
Correct. Lazy import means only the one group you invoke gets loaded, so startup-only commands don't pay for the whole tree or its heavy dependencies.
It's about startup speed, not security or a Click rule: importing only the invoked group keeps --version/--help fast and avoids loading requests and every module.
