Milestone 04 / Schema Authoring / Lesson 03

One command,
whole table

Declare the table and every column in a single shorthand call — one publish, and a re-runnable report that's safe to repeat.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. L02's per-call publishing was the pain point. scaffold table is the first cure: one command builds the entity and all its columns in dependency order, publishes once, and reports each piece as applied/skipped/planned/failed. It's the middle rung between hand-typing verbs and the full declarative spec in L04.

Today's win: build the whole Project table in one line, re-run it to watch every piece say skipped, and learn which column kinds the shorthand can't express.

01 The shorthand

scaffold table takes a display name and one --column per column. Each column is Display:KIND[:opts]. The whole Project table, in a single call:scaffold

agent-cloud · exit 0crm --json scaffold table "Project" \
    --column "Code:string:max_length=20,required=ApplicationRequired" \
    --column "Budget:money" \
    --column "Active:boolean"

{ "ok": true, "data": {
    "applied": [
      {"kind":"entity",    "name":"itworx_Project"},
      {"kind":"attribute", "name":"itworx_Code"},
      {"kind":"attribute", "name":"itworx_Budget"},
      {"kind":"attribute", "name":"itworx_Active"}
    ], "skipped":[], "planned":[], "failed":[] },
  "meta": { "staged": false } }

One call, one publish (meta.staged: false = it published). The result is the same four-bucket report you saw in L01's dry-run — now with everything in applied. Internally it's L02's verbs in dependency order (entity first, then columns), with a single PublishAllXml at the end.

02 What the shorthand can — and can't — say

The :opts after the kind are comma-separated key=value pairs, and the set is deliberately small:

allowed opts
max_length · required · description · optionset_name · target_entity
not available
precision · format · min · max · inline picklist options

That gap has a sharp consequence the live CLI enforces: because there's no precision opt, decimal and double columns can't be scaffolded — they require precision, so they fail. (money works: it has a default precision.) Likewise a picklist in scaffold must point at an existing global option set via optionset_name= — there's no inline-local form here:

Want…In scaffold?Use instead
string max length, required:string:max_length=20,required=ApplicationRequired✓
money:money (default precision)✓
decimal / double✗ needs precisionL02 add-attribute --precision, or L04 spec
picklist (global):picklist:optionset_name=itworx_priority✓
picklist (inline local)✗L02 add-attribute --option, or L04 spec
lookup:lookup:target_entity=account✓ (→ M05)
✓
Pick the tool by what the column needs. Scaffold is fast for plain typed columns. The moment you need precision, a string format, a value range, or inline local options, reach for the imperative verb (L02) or the declarative spec (L04). They're complementary, not competing.

03 Re-running is safe: skipped

Every resource is created with if-exists = skip. Run the same scaffold table again and the entity and existing columns come back as skipped, while any new column is applied — so editing your command and re-running just fills the gap, never errors or duplicates:

agent-cloud · exit 0 · second run, +Weightcrm --json scaffold table "Project" \
    --column "Code:string:max_length=20,required=ApplicationRequired" \
    --column "Weight:integer" --column "Active:boolean"

{ "ok": true, "data": {
    "applied": [ {"kind":"attribute","name":"itworx_Weight"} ],        # the new one
    "skipped": [ {"kind":"entity","name":"itworx_Project"},
                 {"kind":"attribute","name":"itworx_Code"},
                 {"kind":"attribute","name":"itworx_Active"} ] } }
!
Partial apply on failure. Resources are created in order; if one fails, the ones before it stay. On a first run where Weight:decimal failed for missing precision, the entity and the column before it were already applied, Weight went to failed, and the rest never ran. Because re-running skips what exists, the fix is simply: correct the bad column and run again.

04 Preview and stage

Two global controls round out the verb:

FlagEffect
--dry-runEverything reported under planned, no creates — only one existence GET fires (you saw this in L01)
--stage-onlyCreates everything but suppresses the final publish → meta.staged: true; publish later yourself

So the verb spans the whole safety spectrum in one place: --dry-run to look, default to build-and-publish, --stage-only to build-but-hold. The same envelope shape carries all three states — read planned vs applied and meta.staged to know which you got.

05 Do it yourself ~7 min writes

# 1 · preview first — no writes
crm --json --dry-run scaffold table "Project" \
    --column "Code:string:max_length=20" --column "Budget:money" --column "Active:boolean"

# 2 · build it for real (drop --dry-run)
crm --json scaffold table "Project" \
    --column "Code:string:max_length=20" --column "Budget:money" --column "Active:boolean"

# 3 · re-run with one extra column — watch skip vs apply
crm --json scaffold table "Project" \
    --column "Code:string:max_length=20" --column "Budget:money" \
    --column "Active:boolean" --column "Notes:memo"

# 4 · clean up
crm --json metadata delete-entity itworx_project --yes
  1. In step 1, confirm planned lists 4 names and applied is empty. Nothing was created.
  2. In step 3, exactly one name is in applied (itworx_Notes) and three are in skipped. Re-running is a no-op for what already exists.
  3. Now try to break it: add --column "Score:decimal" and run. Read the failed entry — which opt is the shorthand missing?
  4. Re-run step 2 with --stage-only on a fresh display name. Check meta.staged. What's different from the default run?

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · How many publishes does one scaffold table with 5 columns run?
Correct. That's scaffold's whole point versus imperative verbs: build everything in order, then a single PublishAllXml. meta.staged: false confirms it published.
Scaffold publishes once, at the very end — not per column. That's the efficiency win over L02's per-call publishing.
2 · You scaffold a column "Score:decimal". What happens?
Correct. The shorthand's opts are max_length, required, description, optionset_name, target_entity — no precision. Since decimal/double require precision, they land in failed. Use add-attribute --precision or a spec instead.
Decimal/double require precision, but the scaffold shorthand can't express it — so the column goes to failed. Money works (default precision); decimal/double don't.
3 · You re-run the exact same scaffold table command. What do you get?
Correct. Every resource is if-exists=skip, so re-running is idempotent — existing pieces are skipped. Add a column to the command and only the new one is applied.
It's idempotent: existing resources are skipped, not duplicated or errored. Re-running to add a column only applies the new one.
4 · What does --stage-only change versus a default scaffold run?
Correct. --stage-only still writes the schema, but holds the publish — meta.staged: true. (--dry-run is the one that writes nothing.) You publish on your own schedule afterward.
Stage-only builds the schema but suppresses PublishAllXml (meta.staged: true). The no-write option is --dry-run; stage-only does write.
