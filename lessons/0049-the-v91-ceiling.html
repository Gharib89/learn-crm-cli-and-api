The v9.1
ceiling

On-prem caps at API v9.1. Ask for v9.2 and the server answers 501. Here's why that line exists, how the CLI dodges it for you, and what an older build genuinely lacks.

live on agent-on-prem  ·  crm.contoso.local/Contoso  ·  reads reversible probe
Why this matters: The version ceiling is the single most quoted on-prem fact — and the most misread. Once you know that v9.1 and v9.2 are behaviorally identical APIs, and that the cap is about which URL path the server answers (not which features the API has), you can tell a "this org is older" error apart from a real bug. That judgment is exactly what the mission asks for.

The API version is a segment in the URL path — /api/data/v9.1/ — not a feature switch. The cloud build serves up to v9.2; the on-prem build serves up to v9.1. Ask an on-prem server for a path it doesn't host, and it tells you so.

01 The version is a path, not a feature flag

Here is the part most people get wrong. Microsoft is explicit: the v9.0, v9.1, and v9.2 releases have identical Web API behaviors — none of them introduces breaking changes relative to the others.1 So the number in the path is not a capability dial. It exists so that, if Microsoft ever ships a breaking change, your code can pin to the version it was written against.

What actually differs between on-prem and cloud is the product build behind the API. The cloud is continuously updated and hosts the v9.2 path; the on-premises product tops out at the v9.1 path. When the CLI requests …/api/data/v9.2/… from an on-prem server, that route doesn't exist there, and the server returns HTTP 501 (Not Implemented).

KEY
Two different "differences" hide under "v9.1 vs v9.2." (1) The path difference — on-prem 501s on the v9.2 URL. (2) The build difference — the older on-prem product lacks some newer Dataverse messages (like CreateMultiple, §04). The version number causes (1); the product age causes (2). They are not the same thing.

02 The 501, live

You can see the ceiling directly. The on-prem profile is pinned to v9.1, so a normal call succeeds. Temporarily point it at v9.2 and re-run WhoAmI — the server refuses. (profile edit changes only the named field; it never touches the stored secret, so this is a safe, reversible probe.)

shell · reversible probe# 1 · force the on-prem profile to v9.2
crm --json profile edit agent-on-prem --api-version v9.2

# 2 · ask the on-prem server for the v9.2 path
crm --json --profile agent-on-prem connection whoami

  "ok": false,
  "error": "Requested API Version 'v9.2' is not available",

    "status": 501,

  }

shell · always revert the probe# 3 · put it back to v9.1
crm --json profile edit agent-on-prem --api-version v9.1
READ
The status is 501 and the CLI files it under category: "server_error" with retryable: true. That classification is generic 5xx handling — but a 501-for-version is not a transient fault. Retrying won't help; the path will never exist on this server. This is a case where you read the error string ("Requested API Version 'v9.2' is not available"), not just the category, to know the real cause.

03 Why you almost never see it — auto-negotiation

In normal use the 501 above never happens, because the CLI negotiates the version once, at profile add time. The logic lives in crm/core/connection.py:2

  1. The optimistic default is v9.2 (what cloud accepts).
  2. On profile add with no --api-version, the CLI probes with WhoAmI. If the server 501s, it downgrades one step to v9.1 and probes again.
  3. The negotiated value is written into the profile. From then on, every command uses the pinned v9.1 — no re-probe, no 501.

That is why the on-prem profile already shows "api_version": "v9.1" (you saw it in L01): it was negotiated when the profile was created.

GOTCHA
An explicit version is respected as-is — never auto-downgraded. If you pass --api-version v9.2 yourself (or set it with profile edit, as in §02), the CLI assumes you mean it and does not rescue you with a downgrade — even when it 501s. Auto-negotiation only kicks in when you omit the version and let the CLI choose. The fix for a hard-set bad version: re-run profile add without --api-version to re-negotiate.

04 What's genuinely missing: CreateMultiple & friends

Now the build difference. The high-throughput bulk messages CreateMultiple, UpdateMultiple, UpsertMultiple, and DeleteMultiple are newer Dataverse additions. They are cloud-only — not present on the on-prem v9.1 build.3 (Even on cloud they're selective: any custom table supports them, but not all stock tables like account or contact do.)

So how does crm data import move thousands of rows into an on-prem org? It doesn't use CreateMultiple at all. It routes every bulk write through the OData $batch endpoint — the universal mechanism that exists on both targets.4 In the CLI's own words, $batch is "the only on-prem bulk mechanism."

crm data import --helpBulk-import records from a JSONL/CSV file via $batch.
  --chunk-size INTEGER   Records per $batch call (each chunk is one
                       transactional changeset by default).
MechanismOn-premCloudUsed by crm
CreateMultiple / UpdateMultiple✗ absent✓ (some tables)never

The upshot for the mission: the platform-level CreateMultiple gap is real, but it does not change how you use the CLI. Because data import always uses $batch, bulk import behaves identically on on-prem and cloud. The gap only matters if you were calling those messages directly via action invoke — which would fail on-prem with a "message not found" error.

4 questions · click to reveal

1. A command against on-prem returns status: 501, "Requested API Version 'v9.2' is not available". meta.retryable is true. What should you do?

Correct. The retryable: true is generic 5xx classification, but a 501-for-version is permanent: the v9.2 path will never exist on an on-prem server. The error string is the real signal. The fix is to use v9.1 — re-negotiate by re-running profile add without an explicit version.
Don't trust retryable here — it's generic 5xx handling, but a 501-for-version is permanent. The v9.2 path doesn't exist on on-prem and never will; retrying or switching auth won't help. Read the error string and use v9.1 (re-negotiate via profile add).

2. True or false: because on-prem caps at v9.1, its Web API has fewer behaviors / operations than the v9.2 cloud API by virtue of the version number.

Correct. Microsoft documents that v9.0, v9.1, and v9.2 have identical Web API behaviors — no breaking changes between them. The version number isn't a capability dial. Real feature gaps (like CreateMultiple) come from the older on-prem product build, not from the version segment.
False. v9.0/9.1/9.2 are documented as behaviorally identical — the number doesn't add or remove operations. Feature gaps such as CreateMultiple come from the older on-prem product build, not from the version segment in the path.

3. When does the CLI auto-downgrade v9.2 → v9.1?

Correct. Negotiation runs once, at profile-add time, and only when you omit --api-version: it tries v9.2, downgrades to v9.1 on a 501, and persists the result. An explicit version is respected as-is and never auto-downgraded — that's why a hard-set v9.2 just 501s.
Auto-downgrade happens only at profile add when you omit --api-version — it probes, downgrades on 501, and persists v9.1. It does NOT run per-request, and an explicit version is never auto-downgraded.

4. You need to bulk-import 5,000 contacts into the on-prem org. CreateMultiple isn't available there. What does crm data import do?

Correct. data import never uses CreateMultiple on either target — it chunks the rows into OData $batch calls (set by --chunk-size). $batch exists on both on-prem and cloud, so bulk import behaves the same regardless of the CreateMultiple gap.
data import always uses OData $batch (the universal bulk mechanism present on both targets), chunked by --chunk-size. It never relies on CreateMultiple, so the on-prem gap doesn't affect it — and it doesn't degrade to one-request-per-row.
