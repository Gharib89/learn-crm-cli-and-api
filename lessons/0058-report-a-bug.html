Milestone 10 · Working Agentically · Lesson 06

Tool defect,
or you?

The M10 capstone. When a command misbehaves, sort it: your error, a documented limit — or a genuine crm defect. When it's the last one, file a high-quality issue upstream. The agent offers; it never files silently.

feedback loop  ·  upstream: Gharib89/crm  ·  capstone
Why this matters: The mission's final goal is to recognize when a failure is a tool defect — not user error — and feed a high-quality issue back to the upstream repo. A good report makes the CLI better for everyone and closes the agentic loop: you didn't just consume the tool, you improved it. This is the one milestone skill that points outward.

Most failures aren't bugs. The skill is telling the rare real defect apart from the far more common "the server said no" or "that's a documented limit" — and then writing the report that gets it fixed.

01 The triage line

Before reaching for the issue tracker, run the failure past this line. Most land on the left:

It's not a CLI bug if…It may be a CLI bug if…
The error names a server condition (auth, privilege, "not found", version)The CLI raises an internal traceback instead of a clean {ok,error,meta} envelope
The same inputs fail the same way the server would by hand--dry-run shows the CLI building a wrong request (bad URL/version/body)
It's a documented limit (a GUI-only component; a cloud-only message on on-prem)A version-agnostic command behaves differently for no documented reason
A retryable class actually recovers on retryThe envelope contract is violated (missing ok/meta, wrong category for the status)
SHORT FORM
A bug is the CLI's own behavior being wrong — a request it mis-built, a crash that breaks the contract, an unexplained inconsistency. A well-formed server "no" (a 403, a 501-for-version, a 429) is the API doing its job, reported faithfully. When in doubt, reproduce it with --dry-run and ask: did the CLI do something wrong, or did it correctly relay the server?

02 The agent offers — it never files silently

This is a firm rule from the skill itself: if crm misbehaves or lacks a capability, the agent tells you and offers to file an issue — it does not file on its own. Filing is an outward, public action; it needs your sign-off and your judgment on what to include.

CONSENT
The same posture you saw for GUI-only gaps (L02) applies here: surface, don't paper over — and surface, don't act unilaterally. The agent drafts the report; you approve it before anything is posted to Gharib89/crm.

03 Filing it well

On your approval, the issue goes to the upstream repo with the gh CLI, carrying the facts that make it reproducible — and nothing secret:

on approval# keep the needs-triage label so the maintainer sees it
gh issue create --repo Gharib89/crm --label needs-triage \
  --title "<short summary>" \
  --body-file <path>

The body follows the skill's template — the four things a maintainer needs to act:

issue body## What I was doing
<the exact crm command(s) run — scrubbed>

## Expected
<what should have happened>

## Actual
<the --json envelope or error output>

## Environment
- crm --version: 1.0.0
- target: on-prem (NTLM) | Dataverse online (OAuth)
- API version: v9.1 | v9.2
SCRUB
Before anything is posted, replace the real org/host with the example env (https://orgexample.crm.dynamics.com, http://crm.contoso.local/Contoso) and mask GUIDs. The repro must be faithful in shape, never in secrets. For a feature request, drop the Expected/Actual split — describe the capability and the workflow it unblocks — and keep the needs-triage label.

Lesson check

4 questions · click to reveal

1. Which is the strongest signal of a genuine crm defect?

Correct. A wrong request the CLI itself composed (seen in dry-run) or a crash that breaks the {ok,meta} contract is the tool misbehaving. A 403, a GUI-only gap, and a cloud-only message on on-prem are all documented/expected conditions reported faithfully — not bugs.
The defect signal is the CLI's own behavior being wrong — a malformed request (visible in dry-run) or a non-envelope traceback. 403s, GUI-only gaps, and cloud-only-message limits are expected conditions, correctly reported.

2. The agent concludes it has found a real bug. What does it do?

Correct. The skill's rule is explicit: surface it and offer to file — never file silently. Filing is a public action needing your sign-off. (And per the mission, this track files issues, not PRs — fixing the source is out of scope here.)
It surfaces the defect and offers to file, posting only on your approval — never silently, and never auto-filing. Fixing the source / opening a PR is out of scope for this track; a silent workaround hides the problem.

3. What must you do to the command and output before the issue is posted?

Correct. The repro must be faithful in shape but carry no secrets — replace the real org URL/host with the example env and mask GUIDs. The version, the full envelope, and the needs-triage label are exactly what the maintainer needs; keep them.
Scrub host/org to the example env and mask GUIDs — faithful shape, no secrets. Keep the version, the full envelope, and the needs-triage label; those are what make the report actionable.

4. In one phrase, what is the M10 agentic loop?

Correct. That six-step loop is the milestone: read the contract, judge the boundary, plan the work, de-risk with dependency checks, promote across environments, and feed defects back upstream. The others are useful fragments (the safe-write loop, the build cycle) but not the whole agentic workflow.
The loop is contract → boundary → plan → de-risk → promote → feedback. The other options are real but partial — the safe-write loop and the build cycle live inside this larger workflow.
