Milestone 10 · Working Agentically · Lesson 05

Two orgs,
one change

A customization is built in dev and lands in test, then prod. The profile is the switch between them, whoami confirms the target before every write, and a managed solution is the vehicle that carries the change downstream.

live on agent-cloud + agent-on-prem  ·  reads promotion
Why this matters: The mission requires handling the cloud/on-prem split correctly and delivering a requirement inside a solution that can move between environments. You never customize in prod; you build unmanaged in dev and ship managed downstream. Getting the switch, the direction, and the version ceiling right is what makes the agentic workflow safe across a real dev → test → prod chain.

The single most dangerous mistake in promotion is doing the right command against the wrong org. So the discipline starts there: the profile is the environment, and you confirm it every time before you write.

01 The profile is the switch

There's no env-var override and no ambient target — the active profile decides which org answers. profile list marks the active one; --profile <name> overrides per-run:

live · two profiles, two orgs$ crm --json profile list
[ { "name": "agent-cloud", "active": true,
    "target": "cloud", "url": "https://orgexample.crm.dynamics.com" },
  { "name": "agent-on-prem", "active": false,
    "target": "on-prem", "url": "http://crm.contoso.local/Contoso" } ]

Before any mutation, whoami confirms the live org actually answering — both targets respond with the same envelope, just different identities:

live · whoami on each target# cloud (active)            # on-prem (--profile agent-on-prem)
OrganizationId: 9102ced1-…     OrganizationId: b948cd5f-…
# same command, same shape — only the identity (and host) differ
RULE
whoami before every mutate. The agent never trusts "the active profile is probably right" — it confirms the target, because a schema write against prod-instead-of-dev is exactly the irreversible mistake this whole milestone is built to prevent.

02 Build unmanaged in dev, ship managed downstream

The ALM rule: customize in a dev org inside your own unmanaged solution, export it as managed, and import that managed zip into test then prod. The promotion pipeline, in real verbs:

dev → test → prod# in DEV — bump, then publish (unpublished == not exported)
crm solution set-version DealDesk --version 1.1.0.0
crm solution publish-all

# export MANAGED for downstream orgs
crm solution export DealDesk -o dealdesk_managed.zip --managed

# pre-flight, then import into TEST (repeat for PROD)
crm solution validate dealdesk_managed.zip --against-org
crm --profile test solution import dealdesk_managed.zip --yes
TWO TRAPS
Unpublished == not exported — a change you forgot to publish-all silently drops out of the zip. And never hand-edit a managed component downstream — that active unmanaged layer blocks future managed updates. The agent's job downstream is import only.

03 The version-ceiling trap

A managed zip carries the package version of the org it was built in, and an org rejects any zip newer than itself. That bites the cloud → on-prem direction directly: a cloud (v9.2) export will not import into on-prem (v9.1) — you get 0x80048068.

Build orgTarget orgResult
on-prem v9.1cloud v9.2imports (target is newer)
cloud v9.2on-prem v9.1rejected — 0x80048068
CATCH IT
solution validate --against-org compares the package's SolutionPackageVersion against the target org and catches this before the import (the offline validate is structural only and won't). The fix: build on the lowest version in your chain, or keep every tier on one platform. This is the cloud/on-prem split the mission asks you to handle, showing up at promotion time.

04 The agent's promotion discipline

Pulling it together — the rules the agent follows on every promotion, each one guarding against an irreversible slip:

  1. whoami before every mutate — confirm the live target, never trust the active profile blindly.
  2. Build unmanaged in dev only — prod and test receive managed imports, never hand edits.
  3. Publish before export — unpublished customizations don't travel.
  4. validate --against-org before import — catch the version ceiling and live form/view issues on paper.
  5. One solution per change, dry-run on the target — and verify components landed after import.
SAME VERBS
Notice the promotion commands are identical on cloud and on-prem — only the profile (auth + host + version) differs, exactly as M09 established. On-prem imports run synchronously; cloud returns an async job to watch. The workflow doesn't fork; the profile carries the difference.

Lesson check

4 questions · click to reveal

1. What selects which org a crm command runs against?

Correct. The profile is the switch — there's no env-var override and no ambient target. profile list shows the active one; --profile <name> overrides for a single run. That's why whoami-before-mutate matters: the profile, not a host flag, decides the target.
The active profile (or --profile per run) selects the org. There's no CRM_URL env var, no per-verb host flag, and nothing sticky about the last response.

2. You promote a change to prod. In which form does the solution travel downstream?

Correct. You build unmanaged in dev and ship managed downstream — managed layers are how test/prod receive changes without hand edits that would block future updates. Re-doing it by hand or copying records isn't ALM and doesn't carry metadata properly.
Ship a managed export into prod. Unmanaged downstream invites hand-edits that block updates; rebuilding by hand or bulk-copying records isn't solution ALM.

3. A cloud (v9.2) managed export fails to import into on-prem (v9.1) with 0x80048068. Why?

Correct. The zip carries its build org's package version, and a target rejects anything newer. So a v9.2 build won't land on a v9.1 org. validate --against-org catches it pre-import; the fix is to build on the lowest version in the dev→test→prod chain.
It's the version ceiling: a target org rejects a package newer than itself, so v9.2→v9.1 fails. On-prem does import managed solutions, it's not a privilege issue, and managed zips do move between orgs — just not downward in version.

4. What must happen in dev before solution export, or the change won't travel?

Correct. Only published customizations are included in the export zip, so a forgotten publish-all silently drops changes. Deleting the solution would remove it, switching to prod is wrong-direction, and whoami confirms a target but doesn't publish anything.
publish-all first — unpublished changes drop out of the export. Don't delete the solution, don't switch to prod to build, and whoami confirms a target without publishing.
