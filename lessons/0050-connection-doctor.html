Why this matters: M01 taught you to confirm identity before mutating. On-prem raises the stakes: the connection can break at the network layer (VPN down), the TLS layer, the version layer, or the auth layer — each with a different fix. connection doctor classifies which layer failed, so you act on the real cause instead of retrying blind. Reading its output is the on-prem operator's core skill.

Every other lesson in this milestone looks at one divergence. connection doctor shows them all at once — it's the single command that paints the full on-prem-vs-cloud picture, check by check.

01 What doctor checks

connection doctor (top-level alias: crm doctor) runs five checks in a fixed order, short-circuiting downstream checks when a prerequisite fails — there's no point testing TLS if TCP never connected. It's read-only: safe to run anytime, against any profile.

#CheckAnswers
1dns_tcpDoes the host resolve and accept a TCP connection on its port?
2tlsDoes the TLS handshake succeed? (skipped for plain http)
3versionWhich server version answers, and is the profile's api_version served?
4authDo the credentials authenticate? (returns the user GUID)
5rate_limitAre service-protection headers present, and what do they say?

The structure — these five checks, in this order — is identical on both targets. The details are where on-prem and cloud part ways.

TIP
Each failed check has a different fix. A dns_tcp failure on on-prem almost always means the VPN is down ("No route to host"). A version failure means the profile's api_version isn't served (re-negotiate). An auth failure means wrong domain/username/password (NTLM) or app registration (OAuth). doctor tells you which layer broke so you don't fix the wrong one.

04 The rate-limit check: service protection

The last check is the most cloud-specific. Dataverse online enforces service protection API limits: caps on requests-per-connection, concurrent connections, and execution time, evaluated over a rolling 5-minute window. Exceed them and the platform returns HTTP 429 (Too Many Requests) with a Retry-After header telling you how long to wait.1 Cloud advertises your remaining budget on every response via x-ms-ratelimit-* headers — which is what doctor surfaces.

On-prem emits none of these. There's no shared multi-tenant platform to protect, so doctor reports "no rate-limit headers present". Practically:

  • Cloud: a tight bulk loop can hit a 429. The CLI's retry layer honors Retry-After for idempotent verbs — but throttling is a real operational concern.2
  • On-prem: no service-protection throttling. Your ceiling is the server's own hardware, not a platform quota — heavy loops won't earn a 429, but they can still overload a modest box.
NOTE
Throttling-absence is not a license to hammer on-prem. The retryable classes you learned (429/5xx/transport) still apply; on-prem just won't produce the 429 class. Treat the difference as "no quota referee," not "infinite capacity."

3 questions · click to reveal

1. connection doctor against on-prem shows dns_tcp: ok=false, "No route to host" and every later check "not checked (network unreachable)". What's the most likely cause?

Correct. dns_tcp is the first check; "No route to host" is a network-layer failure, and for a VPN-gated on-prem host that almost always means the VPN isn't connected. The later checks are skipped because the connection never opened — so this isn't an auth, version, or throttling problem.
A dns_tcp / "No route to host" failure is network-layer — for a VPN-gated on-prem org it means the VPN is down. The downstream checks are skipped (can't test auth/version without a connection), so it isn't a password, version, or throttling issue.

2. Which doctor check returns "not applicable (plain http)" on a typical on-prem profile, and why?

Correct. On-prem orgs are commonly reached over plain http:// inside a trusted network, so there is no TLS handshake to verify and the tls check reports "not applicable (plain http)". Cloud is always https://, so its tls check does a real handshake.
It's the tls check. On-prem is typically plain http:// (no transport encryption to verify inside the VPN), so TLS is "not applicable." Cloud is always https and runs a real handshake.

3. Why does the cloud rate_limit check show x-ms-ratelimit-* values while on-prem shows none?

Correct. Service protection limits exist to keep one tenant from degrading a shared multi-tenant cloud — so Dataverse online advertises remaining budget via x-ms-ratelimit-* headers and returns 429 + Retry-After when exceeded. On-prem is single-org, so there's no platform quota and no such headers.
Cloud emits x-ms-ratelimit-* because service protection limits guard the shared platform (429 + Retry-After when exceeded). On-prem is a single dedicated org with no platform quota, so it genuinely sends no such headers — the CLI isn't hiding or renaming anything.
