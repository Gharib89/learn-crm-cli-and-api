Webhooks:
the no-DLL step

Register a webhook service endpoint and attach an event pipeline step — no .NET assembly required. The M08 done-when for plug-in step registration.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads + writes  ·  cleanup included
Why this matters: Not every integration needs compiled code. A webhook lets D365 POST the full execution context to any HTTP endpoint on every Create, Update, or Delete — in the same event pipeline as a plug-in, with the same stage and filtering options. The crm CLI can register, manage, and clean up webhooks without opening Power Apps maker. This is also the cleanest way to satisfy "register a plug-in step" without needing a .NET project.

A webhook is a service endpoint in the event pipeline. Instead of routing the message to a .NET class, D365 POSTs a JSON payload to your URL — same stages, same filtering, no assembly required. If you can accept an HTTP POST, you can receive D365 events.

01 Webhook vs plug-in: the comparison

Both live in the same event pipeline and share the same registration model. The difference is the handler — a .dll class vs an HTTP endpoint.

Dimension Plug-in Webhook
Handler Compiled .NET class in a .dll HTTP endpoint (any URL)
Fires via Event pipeline (pre-val / pre-op / post-op) Event pipeline (same stages)
Payload to handler IPluginExecutionContext (C# object) JSON RemoteExecutionContext (HTTP POST body)
Create via CLI register-assembly + register-step register-webhook + register-step
Needs .NET Yes No
Best for Complex logic, DB queries, other API calls Integrations, external systems, lightweight routing
TIP
Webhooks are the fastest path to a registered pipeline step. You need an HTTP endpoint and two CLI commands — no Visual Studio, no NuGet packages, no .dll compilation. For integration scenarios this is usually the right choice.

02 register-webhook — create the service endpoint

The first step is registering the webhook service endpoint — the URL D365 will POST to. The CLI stores the endpoint and its authentication configuration in the D365 database.

shellcrm --json plugin register-webhook \
    --name "itworx-contact-hook" \
    --url "https://webhook.site/a1b2c3d4-0046-test-hook" \
    --auth httpheader \
    --auth-value "X-Hook-Secret: demo-secret-123" \
    --solution itworx_dev
live output · agent-cloud{ "ok": true, "data": {
    "created": true,
    "name": "itworx-contact-hook",
    "url": "https://webhook.site/a1b2c3d4-0046-test-hook",
    "auth_type": "httpheader",
    "serviceendpointid": "a1b2c3d4-0046-0000-0000-000000000001",
    "solution": "itworx_dev"
}}

Authentication schemes

Schemeauthtype valueHow D365 authenticates
webhookkey 4 Appends ?code=<value> to the POST URL — Azure Functions style
httpheader 5 Sends <key>: <value> as a custom HTTP request header
httpquerystring 6 Appends ?key=value to the URL (custom query parameter)
WARN
--auth-value is write-only. The platform never returns the auth value in any read operation — this is a security design decision. If you lose the value, you must re-register the webhook with a new secret. Save it in your secrets manager at creation time.

03 register-step --service-endpoint — attach to the pipeline

With the webhook endpoint created, the next step is attaching it to the event pipeline — same command as a plug-in step, but with --service-endpoint instead of --plugin-type.

shellcrm --json plugin register-step \
    --message Create \
    --service-endpoint "itworx-contact-hook" \
    --entity contact \
    --stage postoperation \
    --mode async \
    --name "Notify on contact create" \
    --solution itworx_dev
live output · agent-cloud{ "ok": true, "data": {
    "created": true,
    "name": "Notify on contact create",
    "message": "Create",
    "entity": "contact",
    "stage": "postoperation",
    "mode": "async",
    "serviceendpointid": "a1b2c3d4-0046-0000-0000-000000000001",
    "stepid": "a1b2c3d4-0046-0000-0000-000000000002",
    "solution": "itworx_dev"
}}

--plugin-type and --service-endpoint are mutually exclusive — a step has exactly one event handler. Use one or the other, never both.

TIP
Standard webhook pattern: postoperation + async. Making an outbound HTTP call inside the synchronous DB transaction (pre-operation or sync post-operation) risks blocking the user's save on network latency or external service failures. Async post-operation queues the POST as a system job — the user's request completes immediately.

04 What the platform POSTs to your endpoint

On every matching event, D365 serializes the execution context to JSON and HTTP POSTs it to your registered URL. This is the same context a .NET plug-in receives as IPluginExecutionContext — just in JSON form.

RemoteExecutionContext · Create contact · example{
  "MessageName": "Create",
  "PrimaryEntityName": "contact",
  "PrimaryEntityId": "a1b2c3d4-1111-2222-3333-00155d467b91",
  "InputParameters": {
    "Target": {
      "firstname": "Ahmed",
      "lastname": "Gharib",
      "@odata.type": "#Microsoft.Dynamics.CRM.contact"
    }
  },
  "UserId": "bf40e4d2-0000-0000-0000-6045bd001b10",
  "CorrelationId": "65501fd3-a525-42c5-ad2a-976b5e63a1a5",
  "Depth": 1,
  "IsInTransaction": false
}

Key fields:

  • MessageName — the SDK message that fired (Create, Update, Delete, etc.)
  • PrimaryEntityName — logical name of the entity
  • PrimaryEntityId — GUID of the record that was created/updated/deleted
  • InputParameters.Target — the entity object from the operation payload (fields included in the operation, not necessarily all fields)
  • UserId — the user who triggered the operation
  • CorrelationId — use for tracing and idempotency checks
  • Depth — call depth; use to prevent infinite loops in recursive operations
  • IsInTransaction — false for async steps (already outside the transaction)
TIP
Use CorrelationId for idempotency. D365 may retry failed webhook deliveries. Store the CorrelationId in your endpoint and skip processing if you've already handled it — this prevents duplicate side effects on retry.

05 Manage and clean up

Webhook lifecycle management uses the same verbs as plug-in steps, with one important difference for teardown.

shell# Disable without deleting
crm --json plugin set-step-state <stepid> disable

# Re-enable
crm --json plugin set-step-state <stepid> enable

# Remove the step first (always before the endpoint)
crm --json plugin unregister-step <stepid> --yes

# No unregister-webhook command — delete service endpoint via entity delete
crm --json entity delete serviceendpoints <serviceendpointid> --yes
WARN
There is no crm plugin unregister-webhook command. To delete the webhook service endpoint, use crm entity delete serviceendpoints <id> --yes. Always unregister the step first — deleting an endpoint with active steps attached may leave orphaned registrations.

Full cleanup sequence for the lesson demo:

  1. Disable the step: crm plugin set-step-state <stepid> disable
  2. Unregister the step: crm plugin unregister-step <stepid> --yes
  3. Delete the service endpoint: crm entity delete serviceendpoints <serviceendpointid> --yes

Lesson check

4 questions · click to reveal

1. What is the key difference between --plugin-type and --service-endpoint in register-step?

Correct. --plugin-type points to a fully qualified .NET class name inside a registered assembly. --service-endpoint points to a webhook name (service endpoint). They are mutually exclusive — exactly one handler per step.
--plugin-type is the .NET assembly class path; --service-endpoint is the webhook name. They are not interchangeable and are mutually exclusive on a single step. Neither is tied to a specific execution mode or message type.

2. You want D365 to notify an external system on every contact create. Which combination is correct?

Correct. The standard webhook pattern is postoperation + async. This queues the HTTP POST as a system job — the user's Create completes immediately without waiting for your external system. Sync stages would block the user's request on your endpoint's response time.
The correct pattern is --stage postoperation --mode async --service-endpoint. Option C is invalid (async requires postoperation, not preoperation). Option A would make a sync HTTP call inside the save transaction — dangerous. Option D uses --plugin-type (requires a .dll) and is sync.

3. After register-webhook, you read the service endpoint back and notice auth_value is missing. Is this a bug?

Correct. --auth-value is intentionally write-only. The platform stores it hashed and never returns the raw secret in any API response. This is the same security design used by Azure Functions keys and most secrets APIs. Save it in your secrets manager at registration time.
This is expected behavior, not a bug. The auth value is write-only by platform design — it is never returned in any read or list operation. Option D is incorrect because it's not returned even on creation (the creation response only confirms auth_type). Save the secret externally.

4. How do you delete a registered webhook service endpoint?

Correct. There is no unregister-webhook command. Webhook service endpoints are deleted via the generic entity delete command targeting the serviceendpoints entity. Always unregister attached steps first.
Option A does not exist — there is no crm plugin unregister-webhook command. Option B is for workflow records. Option D removes plug-in assemblies. The correct approach is crm entity delete serviceendpoints <id> --yes after unregistering all attached steps.
