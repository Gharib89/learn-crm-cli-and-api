Resilience

The loop wrapped around that one HTTP line: when crm retries, how long it waits, when it refuses — and the one verb it will never retry for you.

source  github.com/Gharib89/crm  ·  @ main
Why now. L04 drew the transport as one box. Inside it, D365Backend.request() isn't a single call — it's a loop. Knowing what that loop retries (and what it deliberately won't) is the difference between "crm hung for a minute then failed" and "crm backed off five times on a 429, then surfaced a throttled error — expected." It's also the last piece of the M01 error categories: where retryable is decided.

Today's win: read request()'s retry loop and predict, for any failure, whether crm retries, how long it sleeps, and why a failed POST is treated differently from a failed GET.

01 The loop, not the call

Every verb you've met — get, post, patch, delete — funnels into one method, D365Backend.request(), and its body is a while True. Send, inspect, maybe sleep, send again:request

Three profile knobs set the shape of the backoff (defaults shown), so a slow or flaky org can be tuned without touching code:config

ConnectionProfile defaultsretry_max        = 5      # attempts after the first send
retry_base_delay = 1.0    # seconds; the "1s" in 2^attempt · 1s
retry_max_delay  = 60.0   # ceiling on any single sleep
retry_jitter     = True   # sleep = random.uniform(0, computed_delay)

Two details worth internalising. Full jitter: the actual sleep is a random value between zero and the exponential target — not the target itself — so a fleet of clients retrying after the same 429 don't stampede in lockstep. Retry-After wins: if Dataverse sends that header, crm honours it (capped at retry_max_delay) instead of computing its own backoff.

02 What gets retried — and the POST rule

"Retryable" isn't one rule; it's two checks — one for responses that came back, one for transport errors where nothing came back at all:retryable

FailureGET · PATCH · PUT · DELETEPOST (create / action)
429 throttledretriedno *
502 / 503 / 504retriedno * (503 only, with flag)
ConnectionError / Timeoutretriedno *
SSLErrornevernever
4xx (400/401/403/404/412)not transientnot transient

* unless you pass --retry-on-ambiguous (or set CRM_RETRY_ON_AMBIGUOUS).

!
Why POST is special. A POST creates a record or fires an action — it is not idempotent. If the response is lost (timeout, dropped connection), the write may already have committed on the server. Blindly re-sending could create a duplicate. So crm refuses to auto-retry a POST (issue #84); you opt into the re-send risk explicitly with --retry-on-ambiguous. SSLError is never retried either — a broken TLS handshake won't fix itself by waiting.

One more subtlety: a 412 (concurrency_conflict, the failed If-Match ETag) is tagged retryable in the envelope, but the backend does not auto-retry it — retrying only helps after you refetch a fresh ETag, which only the caller can do.

03 Paging and $batch: one logical op, many requests

Two more transport behaviours hide behind a single call you make.

Paging — get_collection() follows @odata.nextLink

Dataverse caps a page at (usually) 5,000 rows and hands back an @odata.nextLink for the rest. get_collection() loops that link, flattening each page's value array, until the server stops sending one — or until max_pages is hit. The next-link is an absolute URL with the query already baked in, so your $filter/$select apply only to the first GET; every subsequent page is whatever the server's link says:paging

core/query.py → backendrows = backend.get_collection("contacts",
    params={"$filter": "statecode eq 0"},
    max_pages=None)   # None = follow to exhaustion; 1 = single GET, no follow

$batch — many ops, one round-trip

backend.batch([...]) packs a list of operations into a single POST $batch. Two modes decide what happens when one op fails:batch

  • transactional=True (default) — all ops succeed or the whole set rolls back, like a database transaction.
  • continue_on_error=True — independent ops; a failure doesn't stop the rest. You read each BatchResult to see which succeeded.
  • content_id lets a later op reference a record created earlier in the same batch (e.g. create an account, then a contact pointing at it) — a server-side reference, no extra round-trip.
✓
The mission link: when a bulk operation behaves oddly, "was it batched? transactional or not?" is often the first question. Knowing crm has these two distinct modes tells you which knob to check — and which to mention in a bug report.

So retryable in the envelope is a hint, not a promise the loop already acted on it: for throttled/server_error it means "the loop already tried and gave up"; for concurrency_conflict it means "retry is possible, but it's your move." Reading the flag with that distinction is the skill.

05 See the loop yourself ~6 min

No org needed for the first two; the third needs a connection.

  1. Open d365_backend.py and find def request. Locate the while True: and trace the two exits — the return on a non-retryable response and the raise D365Error when attempt >= max_retries. That's Figure 1, in code.
  2. Find _is_response_retryable. Confirm with your eyes that the very first check short-circuits POST to False unless retry_on_ambiguous — the rule from §2.
  3. With a profile connected, set CRM_VERBOSE=1 and run a normal read. Watch stderr: when a response carries x-ms-ratelimit-* headers, crm logs your remaining request budget — the early-warning signs of throttling, before a 429.
✓
The habit this builds: "did crm retry, and should it have?" becomes a checkable question. A 60-second hang on a flaky network is now legible — five backoffs on retryable transport errors — not a mystery.

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · A crm entity create (a POST) times out — no response comes back. By default, what does request() do?
Correct. A POST isn't idempotent, so a lost response might mean the record was already created. Auto-retrying risks a duplicate, so crm refuses unless you pass --retry-on-ambiguous.
Re-read §2: POST is the exception. Because the create may have committed before the response was lost, crm will not auto-retry it — it raises, unless you opt in with --retry-on-ambiguous.
2 · With retry_jitter = True, how long does crm actually sleep before retry attempt?
Correct. "Full jitter": the computed exponential value is the ceiling, and the real sleep is random.uniform(0, that) — which spreads retries so many clients don't stampede together. (If Retry-After is sent, it wins instead — capped at 60s.)
It's B. Jitter means the exponential target is just the upper bound; the actual sleep is random.uniform(0, target). Retry-After overrides that when the server sends it, but jitter itself is the random-0-to-target rule.
3 · A request fails with 412. meta shows category: concurrency_conflict, retryable: true. Did the loop retry it?
Correct. concurrency_conflict is the one category where retryable: true is a hint to the caller, not a record of the loop's action. A blind re-send would just fail the same If-Match again — you must refetch the ETag first.
It's D. The loop does not auto-retry 412 (unlike 429/5xx). retryable: true here means "retry can work — but only after you fetch a fresh ETag," which is the caller's job.
4 · You read 12,000 contacts with one crm query. How many HTTP GETs likely hit Dataverse?
Correct. Dataverse pages results (~5,000/page), so ~12,000 rows ≈ three GETs: the first, then two @odata.nextLink follows. get_collection flattens them into one list for you.
It's A. Dataverse caps a page (~5,000 rows) and returns @odata.nextLink for the rest; get_collection follows it, so ~12,000 rows is roughly three GETs flattened into one result.
