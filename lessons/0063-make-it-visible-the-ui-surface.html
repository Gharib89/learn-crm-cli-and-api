Make it
visible

The data layer works, but no human can reach it yet. A custom table stays invisible until an app's sitemap points at it. Build the app, surface Deal Review through a subarea, drop Status on the form — then one publish lights the whole feature up.

live on agent-cloud  ·  writes your turn
Why this matters: Everything you built so far is real but unreachable — a sales manager can't open a table that no app surfaces. The brief's whole point is a human setting an approval status, so the UI is not optional polish; it's the last link that makes the feature usable. And the ordering trap here is the sneakiest: the table exists, the data is there, yet it's nowhere on screen until the subarea exists.

A model-driven app is mostly a container and a map. The container is the app; the map is the sitemap, whose subareas decide which tables show up in the nav. No subarea, no table — no matter how complete the schema is.

01 The runbook

Four writes: create the app, build a sitemap whose subarea binds Deal Review, add the Status field to the form, then publish and verify the whole feature end-to-end.

GOAL A Deal Desk app where a human can open Deal Reviews and set Status.

target: agent-cloud · all into DealDesk · one publish at the end

  1. 1. Create the model-driven app. An empty shell — clienttype 4, the modern unified-interface app — bound to the solution.

    step 1 · POST /appmodulescrm app create --name "Deal Desk" --unique-name itworx_dealdesk \
  --description "Deal review & approval" --solution DealDesk

    ✓ verify: the result returns an appmoduleid; note it for the sitemap.

  2. 2. Build the sitemap — the subarea is the magic. entity=itworx_dealreview binds the table into the nav. This is the line that makes it visible.

    step 2 · sitemap + subareacrm app build-sitemap itworx_dealdesk_sitemap \
  --area "sales:Sales" \
  --group "sales/deals:Deals" \
  --subarea "sales/deals:entity=itworx_dealreview:Deal Reviews" \
  --unique-name itworx_dealdesk --solution DealDesk

    ✓ verify: the sitemap links to the app (--unique-name); Deal Reviews now has a nav slot.

  3. 3. Put Status on the form. The column exists (L02) but isn't on any form yet — add it so a human can actually edit it.

    step 3 · form add-fieldcrm form add-field itworx_dealreview itworx_status \
  --solution DealDesk

    ✓ verify: crm form list itworx_dealreview --type main → the main form now contains itworx_status.

  4. 4. Publish once, then verify end-to-end. One publish-all covers app, sitemap and form together. Then open the app and round-trip a record.

    step 4 · publish + smoke testcrm solution publish-all
# open the app, then confirm a record shows with its status & account:
crm query odata itworx_dealreviews --select itworx_name,itworx_status \
  --expand "itworx_AccountId(\$select=name)" --top 5

    ✓ verify: the Deal Desk app opens, Deal Reviews is in the nav, a record shows its Status and linked Account. The feature is usable.

02 Invisible until the subarea

This is the trap that catches everyone: the table is created, published, full of data — and still nowhere a user can click. Three different things gate visibility, and they're independent:

It exists?On a form?In a sitemap subarea?Can a user reach it?
yes—noNo — invisible in every app
yesno fieldyesVisible, but can't edit Status
yesfield addedyesYes — fully usable
TWO MAPS
The sitemap subarea decides whether the table appears in the app's navigation at all. The form decides which fields show once you open a record. They're separate — you need both, which is why steps 2 and 3 are distinct writes.

03 Why one publish at the end

Earlier layers published per step to keep read-backs honest. The UI layer is different: app, sitemap and form are interdependent presentation changes, so a single publish-all after all three avoids three partial publishes and shows the finished surface at once.

PUBLISH OR INVISIBLE
Skip the publish and the sitemap/form edits stay in draft — the app won't show them, and (per M10 L05) they won't export in L06 either. The publish isn't optional polish; it's what moves the change from authored to live and shippable. Optional extra: a public view (crm view create itworx_dealreview …) gives Deal Reviews a tidy default list — nice, but not required for "usable".

Lesson check

4 questions · click to reveal

1. The Deal Review table is created, published, and has records — but users can't find it. Most likely cause?

Correct. Existence ≠ visibility. A custom table appears in an app only when a sitemap subarea (entity=itworx_dealreview) points at it. Without that, it's reachable by API but invisible in the UI.
A table is invisible in an app until a sitemap subarea binds it. Re-publishing, the relationship, and the primary column don't put it in the navigation.

2. The table is in the nav, but users can't edit Status. What's missing?

Correct. The sitemap got them to the table; the form decides which fields appear on a record. Status exists as a column but won't show until form add-field puts it on the main form.
Navigation and form are separate maps. They can reach the record via the subarea, but editing Status needs it added to the form with form add-field.

3. Why publish once at the end here instead of per step?

Correct. The three UI writes form one surface, so a single publish-all after them avoids partial publishes and reveals the finished app in one go. (Schema steps published per-step for honest read-backs — different goal.)
It's a pragmatic choice: the UI pieces are interdependent, so one publish at the end is cleanest. Mid-way publishing is allowed and forms do publish — there's no such restriction.

4. You finish the UI but skip publish-all. Beyond the app not updating, what else breaks?

Correct. Unpublished customizations don't travel in an export (M10 L05), so a forgotten publish silently drops your UI from the managed zip downstream. Publish makes it both live and shippable.
Unpublished changes don't export — your UI would be missing from the L06 managed zip. Records, the plug-in, and the version aren't affected by skipping publish.
