Milestone 06 / Solutions & ALM / Lesson 02

Read what's already there

Three commands give you a complete picture of any solution on the org.

target  agent-cloud  ·  orgexample.crm.dynamics.com  ·  read-only
Why now. Before you build a new solution or export an existing one, you need to know what's live. solution list, info, and components are your read-only toolkit. They're also the first thing an agent would run to discover the org's state — the same recon pattern you used with metadata describe in M03.

Three read commands, one pipeline: list all solutions → drill into one → see its components. The org already has itworx_dev — use it as the running example.

01 solution list: find what's on the org

Every solution installed on the org — system, ISV, and yours — shows up here. The key field is ismanaged: false = your editable solutions, true = locked system or ISV packages.

Many managed solutions are Microsoft system packages. The unmanaged ones you care about: Default and Active are org-wide catch-alls (anything not in a named solution lands in Default), and itworx_dev is yours.

✓
Avoid the Default Solution as a home for your components. Changes made directly in the Default Solution have no publisher prefix enforced and can't be exported cleanly. Always add components to a named solution like itworx_dev.

02 solution info: the full record

Once you've spotted a solution in the list, info fetches its complete metadata record — including the publisher GUID you'll need if you're scripting against the publisher.

agent-cloud · exit 0crm --json solution info itworx_dev

{ "ok": true, "data": {
    "uniquename":       "itworx_dev",
    "friendlyname":     "ITWorx Dev Solution",
    "version":          "1.0.0.0",
    "ismanaged":        false,
    "solutionid":       "34b1d791-a2b3-4c5d-8e9f-0a1b2c3d4e5f",
    "_publisherid_value": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "modifiedon":       "2026-06-18T11:30:00Z",
    "installedon":      "2026-06-18T09:12:00Z",
    "isapimanaged":     false,
    "solutiontype":     0
  } }
ismanaged
false = unmanaged (editable). true = managed (read-only after import).
solutiontype
0 = regular solution. 2 = patch solution (a targeted update on top of a base).
_publisherid_value
GUID of the publisher record. Look this up with solution info-publisher if you need the prefix or option-value offset.
installedon / modifiedon
Timestamps for when the solution was first installed vs last changed. Useful for drift detection.

03 solution components: what's inside

Every component registered in the solution — each with its type code and object GUID. This is how you see exactly what will travel when you export.

agent-cloud · exit 0crm --json solution components itworx_dev

{ "ok": true, "data": [
    {
      "componenttype":        1,
      "rootcomponentbehavior": 0,
      "objectid":             "246395a6-b1c2-4d3e-9f0a-1b2c3d4e5f6a",
      "solutioncomponentid":  "13b78ab2-c3d4-5e6f-7a8b-9c0d1e2f3a4b"
    }
  ], "meta": { "count": 1 } }
componenttype
1 = Entity (table). See the reference table below for the full list.
objectid
The MetadataId of the component — the same GUID you see in metadata describe output. Use this to identify which table, column, or view is registered.
rootcomponentbehavior
0 = include all subcomponents (all columns, views, etc. on the entity). 1 = include only selected subcomponents (you explicitly listed them).
solutioncomponentid
The registration record ID — unique per component-in-solution entry. Not the component's own ID.

04 Snapshot with --save and drift check with --diff

The real power of components is its persistence flags. Write the current inventory to disk, commit it to source control, and then let --diff catch GUI changes before they become a surprise in prod.

Save a snapshot

agent-cloud · exit 0crm --json solution components itworx_dev --save inventory.json

{ "ok": true, "data": { "saved": "inventory.json", "count": 1 } }

The file inventory.json contains the normalized component array — suitable for git:

[
  { "componenttype": 1, "objectid": "246395a6-b1c2-4d3e-9f0a-1b2c3d4e5f6a", "rootcomponentbehavior": 0 }
]

Check for drift

agent-cloud · exit 0 (no drift) or non-zero (drift detected)crm --json solution components itworx_dev --diff inventory.json
# exits 0 — no delta
# exits non-zero — prints the components that exist on the org but not in the snapshot
✓
Commit inventory.json to source control. When a teammate makes a GUI edit that sneaks a new column or form into itworx_dev, your CI pipeline can run --diff inventory.json and fail the build — alerting the team before that untracked component ends up in a prod export. This is the lightweight drift-check pattern for teams without full DevOps tooling.

05 Check yourself

Three questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · solution info itworx_dev returns ismanaged: false. What does that mean?
Correct. ismanaged: false means the solution is unmanaged — its components can be edited in place on this org. This is the normal state for a dev solution like itworx_dev.
ismanaged: false = unmanaged = editable. It says nothing about the number of components, the health of the record, or the publisher's prefix configuration.
2 · A component appears with componenttype: 1 and an objectid. What is the objectid?
Correct. For a componenttype: 1 component, objectid is the MetadataId of the entity — the same GUID you see in metadata describe output. It's the identity of the metadata object, not a data row.
objectid is the MetadataId of the component itself — for type 1 that's the entity's MetadataId. It's not the solution ID, the publisher GUID, or a data-layer primary key.
3 · You run solution components --save inventory.json, then a teammate adds a column to the table via the GUI. What does --diff inventory.json report?
Correct. --diff compares the live org's component list against the saved snapshot and reports what's on the org but not in the file — exactly the untracked changes introduced by GUI edits.
--diff reports components present on the live org that are absent from the saved snapshot — the drift. It works for any component type (not just relationships), and it never modifies or deletes your snapshot file.
Curious? Ask me: "What does componenttype: 300 (a model-driven app) look like in components output?" or "How do I look up a component's objectid from its logical name?"
