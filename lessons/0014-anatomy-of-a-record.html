Anatomy of a
Record

The same contact has two faces — the GUI form you click and the JSON the API returns. This lesson teaches you to read the second one.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. M01 got you connected and reading: you can entity get a record back. But the JSON that returns is full of things the form hides — gendercode: 2, statecode: 0, _ownerid_value holding a bare GUID. Every query you filter, every field you update for the rest of this milestone works on those values, not the friendly labels. Before you write or query records, you have to read one fluently. This is that Rosetta Stone.

Today's win: take one real contact and decode every field — map each GUID, code, and lookup back to what it is on the D365 form.

01 One row, two faces

A Dynamics record is just a row in a table. The form you see in the browser is not the record — it's a styled view of that row, with dropdowns turned into words and GUIDs turned into names. The Web API hands you the row underneath, unstyled. Same contact, two faces:

Full Name
Form shows the text box "Sara Mitchell"  →  API: "fullname": "Sara Mitchell"
Job Title
Form text box "Sales Manager"  →  API: "jobtitle": "Sales Manager"
Gender
Form dropdown reads "Female"  →  API: "gendercode": 2 (the label lives in metadata)
Status
Form badge "Active"  →  API: "statecode": 0 + "statuscode": 1
Owner
Form lookup chip with a person's name  →  API: "_ownerid_value": "<guid>"
(identity)
The form's URL has a record id  →  API: "contactid": "<guid>" — the primary key

The skill for this whole milestone is reading that right-hand column without flinching. Let's pull a real one and name every part.retrieve

02 Every table has two names

Before the fields, one naming split that trips up every beginner — you met it as a gotcha in M01 L06, here's the rule behind it. Each table carries both a plural and a singular name, and different commands want different ones:metadata

contacts
EntitySetName — the plural collection name. This is the one the Web API URL uses, so it's what entity get and query odata take.
contact
LogicalName — the singular, all-lowercase internal name. This is what metadata and query count take (M01 L06's 400 error was feeding contacts where contact was wanted).

Two more names from the same table definition are worth knowing, because they name the keys you just saw: the PrimaryIdAttribute is contactid (the GUID primary key) and the PrimaryNameAttribute is fullname (the column the GUI uses as the record's title). Pattern holds platform-wide: account → accountid/name, contact → contactid/fullname.metadata

✓
Rule of thumb. Plural contacts when you're addressing rows over the API (get/query). Singular contact when you're talking about the table (count, metadata). When in doubt, the entity-set is the plural in the record's _entity_id_url path (e.g. …/contacts(…)).

03 The four field shapes

Here's a real read from agent-cloud — one contact, asking for a column of each kind. --annotations tells the server to attach its human-readable labels (it's on by default for entity get; shown explicitly here):

agent-cloud · exit 0crm --json entity get contacts a1b2c3d4-1111-2222-3333-00155d467b91 --annotations \
    --select fullname,jobtitle,gendercode,statecode,statuscode,_ownerid_value,createdon

{ "ok": true, "data": {
  "fullname": "Sara Mitchell",                          # plain text
  "jobtitle": "Sales Manager",                         # plain text
  "createdon": "2026-05-05T12:12:12Z",                 # datetime (raw ISO)
  "createdon@OData.Community.Display.V1.FormattedValue": "5/5/2026 12:12 PM",

  "gendercode": 2,                                    # option-set: stored code
  "gendercode@OData.Community.Display.V1.FormattedValue": "Female",      # …its label
  "statecode": 0,  "statecode@…FormattedValue": "Active",
  "statuscode": 1, "statuscode@…FormattedValue": "Active",

  "_ownerid_value": "99887766-aaaa-bbbb-cccc-1234567890ab",          # lookup: related row's GUID
  "_ownerid_value@OData.Community.Display.V1.FormattedValue": "Dataverse Integration App",
  "_ownerid_value@Microsoft.Dynamics.CRM.lookuplogicalname": "systemuser",
  "_ownerid_value@Microsoft.Dynamics.CRM.associatednavigationproperty": "ownerid",

  "contactid": "a1b2c3d4-1111-2222-3333-00155d467b91"              # primary key
} }

Four shapes, and every column on every table is one of them:

Primary key
contactid — a GUID, the row's identity. Server-assigned at create (M01 L05). Always present, never something you set.
Plain column
fullname, jobtitle (text), createdon (datetime). The value is the value. Dates arrive as raw ISO Edm.DateTimeOffset; the FormattedValue twin is just locale-friendly display.formatted
Option-set (choice)
gendercode: 2 — the column stores a number; the word "Female" lives in metadata and rides along only as the @…FormattedValue twin. statecode/statuscode are the same shape (the record's lifecycle).formatted
Lookup (relationship)
_ownerid_value holds the GUID of a row in another table — the _<name>_value naming convention, a computed read-only field. Three annotations decode it: FormattedValue = the related row's display name, lookuplogicalname = which table it points to (systemuser), associatednavigationproperty = the name you'd use to $expand it.lookup
!
The lookup is the one that bites. _ownerid_value is not a name and not "the owner" — it's a foreign key: a GUID that only means something once you know its table. That's why lookuplogicalname: "systemuser" matters — the same GUID could otherwise be a contact, a team, or a queue. You'll set lookups in the next lesson, and you set them by GUID + table, never by name.

04 Two views: annotated vs --minimal

Those @… keys are the server doing you a favour — translating codes into labels at read time. Strip them with --minimal and you see what the database column actually stores:

annotated (default)crm --json entity get contacts <id> --select fullname,gendercode,_ownerid_value
{ "fullname": "Sara Mitchell",
  "gendercode": 2, "gendercode@…FormattedValue": "Female",
  "_ownerid_value": "99887766-…", "_ownerid_value@…lookuplogicalname": "systemuser", … }
--minimal · the stored rowcrm --json entity get contacts <id> --minimal --select fullname,gendercode,_ownerid_value
{ "_ownerid_value": "99887766-aaaa-bbbb-cccc-1234567890ab",
  "fullname": "Sara Mitchell",
  "contactid": "a1b2c3d4-1111-2222-3333-00155d467b91",
  "gendercode": 2 }

--minimal drops every key containing @ — the FormattedValues and the lookup annotations — and keeps the business fields, the _*_value lookup GUIDs, and the primary id. Two ways to think about the pair:skill

Annotated
The form's view — codes plus their labels, GUIDs plus their names and target tables. For human reading and reporting.
--minimal
The storage view — exactly what sits in the column. This is the shape you build filters against and send back on an update. For machine logic.
✓
Why this split is the whole point. The GUI only ever shows you the annotated face — so it's easy to think the record "contains" Female and Active. It doesn't: it contains 2 and 0/1. When you filter (--filter "gendercode eq 2") or update (gendercode: 2), you speak in the stored values. Reading the minimal view is reading the truth.

05 Do it yourself ~3 min

All read-only — pick any contact GUID from a quick list, then read it both ways:

# 0 · grab a contactid to work with
crm --json query odata contacts --top 3 --select fullname,gendercode

# 1 · annotated — the form's view (codes + labels + lookup target)
crm --json entity get contacts <paste-id> --annotations \
    --select fullname,gendercode,statecode,statuscode,_ownerid_value,createdon

# 2 · minimal — the stored row (codes + GUIDs only)
crm --json entity get contacts <paste-id> --minimal \
    --select fullname,gendercode,statecode,statuscode,_ownerid_value,createdon

Then, from your own output, answer:

  1. Find gendercode. What number is stored, and what label did the annotated view attach? Which of the two disappeared under --minimal?
  2. Find _ownerid_value. What table does lookuplogicalname say its GUID points to? (It's a foreign key — the GUID alone is meaningless without that.)
  3. You want "all active contacts." Which value would you filter on — statecode eq 0 or statecode eq 'Active'? Why? (Re-read §04.)

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · A contact returns gendercode: 2 alongside gendercode@…FormattedValue: "Female". What is actually stored in the database column?
Correct. The column stores the code 2. "Female" is the option-set's label, kept in metadata and attached only as the @…FormattedValue twin when you ask for annotations. Strip annotations (--minimal) and only 2 remains.
Not quite. The stored value is the code 2. "Female" is just its label (from metadata), riding along as the @…FormattedValue twin — it vanishes under --minimal. The column never stores the word.
2 · You see _ownerid_value: "99887766-…" and _ownerid_value@…lookuplogicalname: "systemuser". What does lookuplogicalname tell you?
Correct. A lookup is a foreign key — a GUID into another table. lookuplogicalname names that table (systemuser), which is what makes the GUID meaningful. (The display name is FormattedValue; the $expand handle is associatednavigationproperty.)
Re-read §03. The GUID in _ownerid_value is a foreign key; lookuplogicalname tells you which table it points into (systemuser). The display name is the FormattedValue annotation, not this one.
3 · query count rejected contacts but accepted contact, while entity get wants contacts. Which is which?
Correct. The plural contacts is the EntitySetName — it's in the Web API URL, so get/query odata use it. The singular contact is the LogicalName, used by metadata and query count. Same table, two names for two jobs.
Flip it. Plural contacts = EntitySetName (the URL/collection name → get/query). Singular contact = LogicalName (the internal name → metadata/count). Not transient, not an org split.
4 · You re-run the read with --minimal. The @…FormattedValue and @…lookuplogicalname keys are gone, leaving gendercode: 2 and a bare _ownerid_value GUID. What happened, and which view matches what's stored?
Correct. --minimal drops every @-key — the read-time labels and lookup metadata — and keeps the business fields, _*_value GUIDs, and primary id. That stripped-down shape is the storage view; annotations are added by the server on the way out, never deleted from the row.
No deletion happens — --minimal only changes what's shown. It drops the @ annotations (labels + lookup target table) the server adds at read time, leaving the actual stored values: the codes and GUIDs. That bare shape is what the column holds.
