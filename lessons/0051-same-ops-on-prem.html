Why this matters: This is the milestone's Done-when — "run the same ops on on-prem and explain each divergence from cloud." Everything so far built the mental model; now you exercise it. The payoff is confidence: once you've seen the round-trip come back identical, you can drive any M01–M08 workflow against on-prem without second-guessing whether it'll behave.

You already know these commands — you used every one on cloud in M01–M02. The point of running them on-prem isn't to learn new verbs. It's to watch what stays the same, so the one thing that changes stands out.

Create a contact, read it back, delete it, confirm it's gone — the M01 safe-write loop, pointed at on-prem with --profile agent-on-prem:

    "createdon": "2026-06-20T06:20:03Z",
    /* …every other contact column, defaulted… */
    "_entity_id": "ce249f12-0000-0000-0000-00155d467b90",

shell · get → delete → verify gone# read it back
crm --json --profile agent-on-prem entity get contacts <id> --select fullname,createdon
# → {"fullname":"M09 OnPremProbe", "contactid":"ce249f12-…", "_entity_id":"…"}

crm --json --profile agent-on-prem entity delete contacts <id> --yes

SAME
The create envelope returns the full created record with _entity_id and _entity_id_url — exactly the shape you saw on cloud. The id lives under contactid / _entity_id (the primary-key column), not a generic id field. not_found after delete is the same category cloud returns. Nothing about the result shape says "on-prem."

02 The one thing that changed: the URL

Where is the divergence, then? In the request line — and you can see it without writing anything, via --dry-run (which still composes the real request, it just doesn't send it):

crm --json --dry-run --profile agent-on-prem entity create contacts --data '{…}'{ "ok": true, "data": {
    "method": "POST",
    "url": "http://crm.contoso.local/Contoso/api/data/v9.1/contacts"
  }, "meta": { "dry_run": true } }

03 Queries, unchanged

Reads behave the same way. Both query styles run on on-prem with no syntax change:

shell · OData + FetchXML (on-prem)# OData — returns a bare data array
crm --json --profile agent-on-prem query odata contacts --select fullname --top 2
# → {"ok": true, "data": [ {…}, {…} ]}

# FetchXML
crm --json --profile agent-on-prem query fetchxml \
    --xml '<fetch top="1"><entity name="contact"><attribute name="fullname"/></entity></fetch>'
# → {"ok": true, "data": [ {…} ]}
RECALL
The data array is bare (the records sit directly under data, not under data.value), and meta.count is opt-in via --count — the same envelope shape you learned for cloud queries. The query layer is version-agnostic: OData v4 and FetchXML are identical on v9.1 and v9.2.

Lesson check

3 questions · click to reveal

1. You run entity create contacts against on-prem and against cloud with the same --data. What differs between the two requests?

Correct. Method (POST), body, entity set, and the returned envelope (including _entity_id) are all identical. The only difference is the request URL — http://crm.contoso.local/Contoso/api/data/v9.1/… vs https://orgexample.crm.dynamics.com/api/data/v9.2/… — both supplied by the profile.
Only the URL differs (scheme+host + v9.1/v9.2). The method is POST on both, the body is identical, and the envelope shape — including _entity_id / _entity_id_url — is the same. The profile carries the host and version.

2. After entity create contacts on on-prem, where is the new record's GUID in the envelope?

Correct. The create envelope returns the full record; the GUID is under the entity's primary-key column (contactid for contacts) and mirrored in the convenience field _entity_id (plus _entity_id_url). There's no generic data.id — same as cloud.
The id is under the primary-key column — contactid — and mirrored in _entity_id / _entity_id_url. There's no generic data.id field, and the full record is returned in the body (no need to re-query). This matches cloud exactly.

3. Does OData / FetchXML query syntax change when you target on-prem (v9.1) instead of cloud (v9.2)?

Correct. OData v4 and FetchXML are identical across v9.1 and v9.2 — same syntax, same flags, same bare data array, same opt-in --count. The query layer doesn't see the version difference at all.
No change. OData v4 and FetchXML behave identically on v9.1 and v9.2 — bare data array, opt-in --count, full --top/--select/--filter support. The query layer is version-agnostic.
