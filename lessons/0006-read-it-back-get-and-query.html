Read It Back:
get & query

Two ways to pull data out of the org — one record by its GUID, or a filtered list of many.

target  agent-cloud  ·  orgexample.crm.dynamics.com
Why now. In L05 you wrote a record and read it straight back from the create response. But most of the time the record already exists and you need to find it — to verify a change landed, to grab a GUID for the next command, to check what's there before you mutate. Reading is the half of the job you'll do most; these two verbs cover nearly all of it.

Today's win: retrieve one contact by its GUID, then list contacts with a filter — and read both shapes ($entity vs a value array) fluently.

01 Two shapes of read

Reading splits by whether you already know the record's GUID:

entity get
By identity. You have the GUID → one specific record. Returns a single object (an OData $entity).
query odata
By criteria. You have a filter → zero-or-more matching records. Returns an array under value, plus meta.count.

Both are GET requests to the same Web API you've been hitting all along — read-only, safe to run as often as you like.query

02 entity get — one record by GUID

agent-cloud · exit 0crm --json entity get contacts a1b2c3d4-1111-2222-3333-00155d467b91 --select fullname,jobtitle,emailaddress1

{
  "ok": true,
  "data": {
    "@odata.context": "https://orgexample.crm.dynamics.com/api/data/v9.2/$metadata#contacts(...)/$entity",
    "@odata.etag": "W/\"2503282\"",
    "fullname": "Sara Mitchell",
    "emailaddress1": "sara.mitchell@example.com",
    "jobtitle": "Sales Manager",
    "contactid": "a1b2c3d4-1111-2222-3333-00155d467b91"
  }
}

Two things to notice:

  • --select is not optional politeness — it's discipline. Omit it and you get every column on the entity (dozens, mostly empty). Always name the fields you need.query
  • @odata.etag (W/"2503282") is the row's version tag. Remember L04's If-Match: * on an update? That header is how the server detects whether the row changed under you; the etag is the value it compares. You're seeing the two halves of optimistic concurrency.update

03 query odata — find many

When you don't have a GUID, you describe what you want. The three workhorse options: --select (which columns), --filter (which rows, in OData syntax), --top (how many):

agent-cloud · exit 0crm --json query odata contacts --filter "jobtitle eq 'IT Director'" --select fullname,jobtitle --top 5

{
  "ok": true,
  "data": {
    "@odata.context": "...#contacts(fullname,jobtitle)",
    "value": [
      { "fullname": "James Okafor", "jobtitle": "IT Director",
        "contactid": "b2c3d4e5-2222-3333-4444-00155d467b92" }
    ]
  },
  "meta": { "entity_set": "contacts", "count": 1 }
}

Results are always an array under value — even one match, even zero (an empty list, still ok:true). meta.count tells you how many came back. Common --filter operators: eq, ne, gt/lt, contains(name,'x'), joined with and/or.query

✓
This is your verification tool. After any write, a quick query odata … --filter proves the change is really in the org — the live confirmation that closes the safe-write loop from L04/L05.

04 query count — and a real gotcha

To count rows without pulling them, there's query count. But it has two sharp edges, both worth meeting on safe ground now:

exit 1 · validationcrm --json query count contacts          # ← the SET name
{ "ok": false, "error": "Entity contacts was not found in the CRM system",
  "meta": { "status": 400, "category": "validation" } }
exit 0crm --json query count contact           # ← the LOGICAL name (singular)
{ "ok": true, "data": { "entity": "contact", "count": 0 } }
!
Two edges: (1) count wants the logical name contact, not the entity-set name contacts that get/query odata use — mixing them up gives the 400 above. (2) The count comes from RetrieveTotalRecordCount, which is cached and approximate — note it returned 0 even though the list in §3 found rows. Trust it for scale, never for exactness; for an exact filtered count, run the query.skill

The set-name-vs-logical-name split (contacts the set, contact the entity) runs through the whole platform — you'll meet it again in metadata. Filing it away now saves confusion later.

05 Do it yourself ~3 min

All read-only. Grab a real GUID from the first command and feed it to the second:

# 1 · list a few contacts — copy any contactid from the output
crm --json query odata contacts --top 3 --select fullname,jobtitle

# 2 · get that one record by its GUID
crm --json entity get contacts <paste-contactid> --select fullname,jobtitle,emailaddress1

# 3 · filter — change the title to one you saw in step 1
crm --json query odata contacts --filter "jobtitle eq 'Sales Manager'" --select fullname

Then answer, from your own output:

  1. In step 1, where do the records live — under data.value or directly under data? In step 2, which is it?
  2. What does meta.count say in step 3, and does it match the array length?
  3. Run crm --json query count contacts — what exit code and error? Now fix it. (One word changes.)

06 Check yourself

Four questions, instant feedback

tap an answer — no penalty for a wrong first try

1 · You have a contact's GUID and want just that one record. Which verb, and what shape comes back?
Correct. By-GUID retrieval is entity get, and it returns one object (an OData $entity) directly under data — no value array. Arrays are what queries return.
Not quite. By a known GUID you use entity get, and it returns a single object under data (the $entity shape) — not an array. The value array is what query odata returns.
2 · crm --json query count contacts returns ok:false, status:400, category:"validation". The fix?
Correct. count takes the logical name (contact), not the entity-set name (contacts) that get and query odata use. category:"validation" with retryable:false told you re-running won't help — it's a malformed argument, fix it.
No. category:"validation" / retryable:false means re-running is pointless and it's not about --top or the org. count wants the logical name contact, not the set name contacts.
3 · A query odata with a filter returns "value": [] and meta.count: 0, with ok:true, exit 0. What happened?
Correct. Zero matches is a perfectly successful query — ok:true, exit 0, an empty value array. "No rows match" is an answer, not a failure. (Contrast a real failure: ok:false + non-zero exit.)
Not quite. ok:true + exit 0 means success — the query ran fine and simply found nothing. An empty value array is a valid result, not an error.
4 · Why prefer RetrieveTotalRecordCount (query count) for "roughly how many?" but a filtered query odata for "exactly how many match X?"
Correct. query count reads a cached total for the whole entity — fast, but approximate (it can lag, even read 0). For an exact, filtered count you run the query itself and read meta.count off the live result.
Re-read §4. count is the cached, approximate, unfiltered total — fast but not exact. For an exact filtered number, run query odata --filter and read meta.count.
