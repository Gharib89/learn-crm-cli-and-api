Milestone 10 · Working Agentically · Lesson 04

Check the
wiring first

A plan is a set of assumptions. Before you mutate, verify them: what needs what, what blocks what, and is each target even ready. Three read-only tools turn a hopeful plan into a de-risked one.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads de-risk
Why this matters: "Dependency order bites" — a write that assumes a prereq which isn't there fails mid-build, leaving the org half-customized and hard to unwind. The mission wants requirements driven cleanly end-to-end; the cheapest way to get there is to confirm the wiring before any write, with read-only calls that cost nothing and catch the failure on paper instead of in production.

There are two questions dependencies answer, and they run in opposite directions: "what must exist before I build this?" and "what would break if I changed this?" One command answers both.

01 Two directions of dependency

metadata dependencies is read-only and points either way, by component --kind (entity, attribute, optionset, relationship):

FlagAnswersMaps to
--for dependents"What depends on this?" — what would break if you changed/removed itRetrieveDependentComponents
--for delete"What blocks deleting this?" — the components in the way of a teardownRetrieveDependenciesForDelete

Both return the same envelope shape: a can_delete verdict plus a blockers array. Here it is live against account on agent-cloud — note can_delete:false because real relationships point at it:

live · agent-cloud · reads only$ crm --json metadata dependencies account --kind entity --for delete
{ "ok": true,
  "data": {
    "can_delete": false,
    "blockers": [
      { "dependent_type": "Entity Relationship",
        "dependent_id": "b319a54f-0364-4ed1-a2f0-…",
        "required_type": "Entity",
        "dependency_type": 1 },
      … more blockers …
    ] } }
READ IT
can_delete:false + a non-empty blockers[] = "something depends on this; deal with those first." Each blocker names the dependent_type (here an Entity Relationship) so you know what stands in the way. For the Deal Desk teardown, this is how the agent removes components in the right (reverse) order instead of hitting 0x8004f01f.

02 Is the target even ready? metadata describe

Dependencies tell you about edges between components. metadata describe tells you about one component's write-readiness — does it exist, what's its real name, what can you write to it. It's the brief the agent reads before authoring against any table:

live · agent-cloud$ crm --json metadata describe account
{ "ok": true, "data": {
  "logical_name": "account",
  "entity_set_name": "accounts",
  "primary_id": "accountid",
  "primary_name": "name",
  "writable_attributes": [ … each with type, required_level, options … ] } }
USE
Before the Deal Desk step that adds a status column, the agent describes the target table to confirm the table exists, get its exact entity_set_name and primary_id (no guessing the three names), and check the column isn't already there. It's the write-readiness brief you met in M03 L05 — now used as a pre-flight gate.

03 The pre-write trio

Put together, three read-only checks de-risk any planned write — none of them mutate:

CheckCatches
metadata dependenciesA prereq that must exist first, or a blocker that stops a teardown/replace
metadata describeA target that doesn't exist, a wrong name, or a column already present
--dry-runA malformed request — it reports live facts (_exists, would_skip) and the exact URL/body it would send

And one more, after a write: a just-published metadata change takes a beat to propagate, so the agent gates the read-back with --expect ATTR=VALUE and retries on mismatch rather than trusting an instant read.

REMEMBER
--validate is record-write only. For the metadata, solution, and component writes that make up most of a customization, your pre-flight net is dependencies + describe + --dry-run + --expect — not --validate.

04 The Deal Desk check, before any write

Applied to the plan from L03, the dependency pass looks like this — all reads, zero mutations:

  1. Confirm the system anchors exist — describe account and describe opportunity: both are real, get their exact set/id names for the lookup and the step.
  2. Confirm the names are free — describe shows no ag_dealreview table and no ag_approvalstatus option set yet, so the create calls won't collide.
  3. Dry-run the dependency-sensitive writes — the 1:N create and the status column, to confirm the request shape (referenced/referencing entity, the optionset binding) before committing.
  4. Re-confirm the boundary — the rollup of approved value is still GUI-only (no --kind rollup), so it stays a flagged manual task, not a step that will fail.
ROLLUP FACTS
Even done by hand in the designer, the rollup carries platform constraints worth telling the user: a rollup can only aggregate over a 1:N relationship (not N:N), can't reference another rollup, recalculates on asynchronous scheduled jobs (so the Account total won't update the instant a review is approved), and there's a cap of 10 rollups per table.2 Surfacing that up front is part of a good dependency check.

Lesson check

4 questions · click to reveal

1. You want to know what would break if you removed a custom relationship. Which call?

Correct. --for dependents (RetrieveDependentComponents) lists what depends on the target — exactly "what would break if I changed this." describe reports one component's readiness, a dry-run previews a single request, and attributes just lists columns.
Use --for dependents — it returns the components that depend on the target. describe is single-component readiness, --dry-run previews one request, and attributes only lists columns.

2. metadata dependencies account --for delete returns can_delete:false with a non-empty blockers[]. This means:

Correct. ok:true means the read succeeded; can_delete:false + blockers is a normal, informative result — it's telling you which components stand in the way. It's not a failure, a privilege issue, or a missing entity.
It's a successful read (ok:true) reporting that dependents exist — handle them first. Nothing failed, there's no privilege problem, and account clearly exists.

3. For a metadata create-entity, which is not available as a pre-flight check?

Correct. --validate is record-write only (entity create/update) — it doesn't apply to metadata. For schema writes, the net is dependencies + describe + --dry-run + --expect.
--validate is record-write only and isn't available for metadata writes. The other three (dry-run, describe, expect-after-publish) are the schema-write pre-flight net.

4. Why does the dependency check re-confirm that the rollup is GUI-only?

Correct. The dependency pass is also where the agent re-verifies the automatable/GUI-only split against the live CLI, so a no-API component like the rollup never sneaks into the build sequence as a step that would error out. It's not about assemblies, dry-run can't create it, and cascade is a relationship concern.
It keeps the rollup as a manual task instead of a doomed build step — the dependency pass double-checks the boundary against the live CLI. Rollups don't depend on the assembly, dry-run can't create them, and cascade is unrelated.
