The agent
contract

When Claude Code drives crm, one flag turns human-readable output into a machine-readable promise: --json. Learn to read the envelope and the exit code, and you can trust — or distrust — every step.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  reads M10 start
Why this matters: The mission is to move D365 work onto a repeatable, reviewable agentic workflow. M01–M09 taught you the verbs; this milestone teaches the workflow that wraps them. It starts here, because every later step — planning, dependency-checking, promotion, bug-filing — rests on the agent and you reading the same structured signal from each command. That shared signal is the contract.

A human reads a wall of text and decides "looks fine." An agent can't — it needs a stable shape to branch on. --json is that shape: did it work, what came back, and what should I notice — in three fixed fields, every time.

01 Always --json — the three-field envelope

From an agent context you pass --json on every call. The reply is always the same skeleton: ok (did it work), data (the result), meta (diagnostics). Success fills data; failure replaces it with error.

success · exit 0$ crm --json connection whoami
{
  "ok": true,
  "data": {
    "BusinessUnitId": "4a617aa7-5a3d-f111-88b5-000d3a3b279b",
    "UserId": "bf40e4d2-ee3e-f111-88b4-6045bd001b10",
    "OrganizationId": "9102ced1-5c3e-f111-b31d-000d3a302819"
  }
}

When something goes wrong, the shape barely changes — ok:false, an error string, and a meta that classifies the failure so the agent can decide what to do next:

failure · exit 1$ crm --json entity get contacts 00000000-0000-0000-0000-000000000000
{
  "ok": false,
  "error": "Entity 'contact' With Id = 00000000-… Does Not Exist",
  "meta": {
    "status": 404, "code": "0x80040217",
    "category": "not_found", "retryable": false
  }
}
SCAN
meta.category is the one field worth branching on first — it's a closed enum (not_found, auth_failed, forbidden, validation, throttled, server_error, transport_error). And meta.warnings is an array of non-fatal advisories (staged-but-unpublished, created-but-read-back-failed) — always scan it even when ok:true.

02 The exit code is the first thing to check

Before parsing a single byte of JSON, the agent checks $?. The exit code answers "did the operation take effect?" in one integer — and it distinguishes a server failure from a you-typed-it-wrong failure:

ExitMeaningEnvelope
0Success — the operation happenedok: true
1Operational failure — server rejected it, validation failed, or you declined a promptok: false + classified meta
2Usage error — bad/unknown flag, missing argument, or a bare crm with no subcommandok: false, terse error (no meta)

The 1-vs-2 split is the useful part. Exit 1 is the server's verdict — your command was well-formed, D365 said no. Exit 2 is the CLI's verdict — you never reached the server:

usage error · exit 2$ crm --json entity get        # forgot the entity-set + id
{ "ok": false, "error": "Missing argument 'ENTITY_SET'." }
# note: no meta block — it never became a request
TRAP
Non-zero is non-zero, but the fix differs. On exit 2 the agent rewrites the command (it's a syntax problem — run --help). On exit 1 the agent reads meta.category and reacts to the server (retry a throttled, fix a validation, stop on a forbidden). Confusing the two means retrying a typo or rewriting a perfectly good command.

03 Three flags that make writes safe

Reads are cheap to get wrong — you just re-run. Writes aren't. Three flags are the agent's standing safety net before any mutation:

FlagWhat it doesScope
--dry-runBuilds the request and shows it (method/url/body) without sending the write. Reads still fire, so the preview reports live facts.every mutation
--validateServer-checks a record payload, turning raw OData noise into a clean unknown_fields verdict.record writes only (entity create/update)
--yesConfirms a destructive verb non-interactively. Without it, a delete in a non-TTY aborts safely.destructive verbs

A dry-run returns the exact HTTP it would have sent — and flags itself with meta.dry_run: true so the agent never mistakes a preview for a commit:

--dry-run · exit 0 · nothing written$ crm --json --dry-run entity create contacts \
      --data '{"firstname":"Test","lastname":"DryRun"}'
{ "ok": true,
  "data": { "_dry_run": true,
    "method": "POST",
    "url": "https://orgexample.crm.dynamics.com/api/data/v9.2/contacts",
    "body": { "firstname": "Test", "lastname": "DryRun" } },
  "meta": { "dry_run": true } }
KEY
The safe-write loop you met in M01 L04 is now the agent's default reflex: validate (records) → dry-run → read $? and the envelope → commit only if clean. --validate is record-only, so for metadata, solution, and component writes, --dry-run is your pre-flight net.

04 Never guess a flag — ask the binary

The CLI is versioned and drifts (you saw a result shape change between releases). So the agent's rule is: when unsure, interrogate the tool, don't recall from memory. Three discovery calls, none of which mutate anything:

  1. crm describe [group] — a machine-readable catalogue of every command, option, and choice. No connection needed. This is how the agent learns flags that are correct for the installed version.
  2. crm <group> --help — the exact options, choices, and defaults for one command.
  3. crm --json connection whoami — confirm which live org will answer before any mutation. The right command against the wrong org is still a mistake.
WHY
A coding agent's training data is frozen; your crm binary is not. Guessing a flag from memory is how you get a confident command that 404s or, worse, writes the wrong thing. describe + --help make the agent's knowledge match this machine's crm --version — the single most important habit in this milestone.

Lesson check

4 questions · click to reveal

1. A command returns exit 2 with {"ok":false,"error":"Missing argument 'ENTITY_SET'."} and no meta. What does the agent do?

Correct. Exit 2 is the CLI's verdict — a malformed command that never became a request (hence no meta). The fix is to rewrite the command, ideally after a quick --help. There's no server category to react to, nothing to retry, and nothing to confirm with --yes.
Exit 2 = usage error: the command was malformed and never reached the server (no meta to classify). The agent rewrites the command (check --help), rather than reading a category, retrying, or filing a bug.

2. Which field should an agent branch on first to decide how to react to a failure?

Correct. meta.category is a small closed set (not_found, auth_failed, forbidden, validation, throttled, server_error, transport_error) — stable to branch on. Status and hex codes are noisier, and the error string is for humans, not control flow.
Branch on meta.category — a closed enum built for exactly this. The HTTP status and hex code are lower-level, and the error string is meant for human reading, not reliable branching.

3. You want to confirm what a metadata create-entity will send, without creating anything. The right flag is:

Correct. --dry-run builds and shows the request without sending the mutation, flagged by meta.dry_run:true. --validate is record-write only — it doesn't apply to metadata, so dry-run is the pre-flight for schema writes. --yes only skips confirmation, and --json only sets output format.
Use --dry-run: it previews the request and writes nothing (meta.dry_run:true). --validate is record-write only (not metadata), --yes just suppresses a prompt, and --json only changes output format.

4. Why is "run crm describe / --help instead of recalling the flag" a core agent habit here?

Correct. The agent's training is frozen but the installed crm evolves (flags, output shapes, and defaults change between releases). describe and --help report what's true for this version, so the command matches the binary rather than a stale memory.
It's about correctness, not speed: crm is versioned and its flags/shapes drift between releases. Querying the binary (describe/--help) keeps the command aligned with the installed version instead of frozen training data.
