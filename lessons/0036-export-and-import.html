Milestone 06 · Solutions & ALM · Lesson 04

Pack it, ship it: export → import

The two commands that move customizations between orgs — and the ZIP that travels between them.

live on agent-cloud  ·  orgexample.crm.dynamics.com  ·  writes
Why this matters: Building in the org is only half the job. The other half is moving what you built. Export turns the solution into a ZIP you can commit, diff, or hand to another team. Import unpacks it on the target. These two operations are the whole ALM pipeline in CLI form.

Export is async — the server builds the ZIP server-side, then streams it down. Import is async too — the server processes it and reports progress. The CLI polls both until done.

01 Export the solution

Export packages the current state of the solution into a ZIP file. The operation is async: the server runs ExportSolutionAsync, the CLI polls the system job, and downloads the file when ready.

FlagPurpose
-o / --outputRequired. Local path for the downloaded ZIP file.
--managedExport as a managed solution. Default is unmanaged. See warning below.
--timeoutOverride the default poll timeout (milliseconds). Large solutions may need a longer timeout.
 
shellcrm --json solution export itworx_dev -o itworx_dev.zip

During the async poll you'll see progress on stderr:

export progress: 25% …
export progress: 75% …
export complete · downloading …

WARN
Add --managed to export a managed ZIP for production delivery. The default is unmanaged. You cannot convert managed → unmanaged — the managed flag is baked into the ZIP at export time. If you export unmanaged and import to prod, the components will be editable on the target (a security and governance problem).

02 What's inside the ZIP

Every solution ZIP has the same three files — regardless of how many components it contains.

shellunzip -l itworx_dev.zip

#      4075  2026-06-18  14:57  solution.xml

unzip -p itworx_dev.zip solution.xml  # see the manifest
FileContents
solution.xmlManifest: UniqueName, Version, Managed flag, Publisher info, and the RootComponents list. This is what you read to know "what's in this ZIP."
customizations.xmlThe actual component definitions: entity XML, form XML, view XML, ribbon XML, etc. The large file.
[Content_Types].xmlOPC/OOXML content type manifest. Boilerplate — you rarely need to read this.

TIP
Git tip: Commit the ZIP for binary storage, or run unzip -p itworx_dev.zip customizations.xml > customizations.xml and commit the extracted XML for meaningful diffs. The XML is the readable artifact; the ZIP is the deployable artifact.

03 Import the solution

Import unpacks the ZIP on the target org and runs ImportSolutionAsync server-side. The CLI polls the async job and streams progress lines to stderr.

FlagPurpose
--overwriteMaps to OverwriteUnmanagedCustomizations on the server. Without it, the server refuses to overwrite unmanaged customizations that already exist on the target.
--publishPublish customizations after import (equivalent to clicking "Publish All").
--skip-dependency-checkSkip dependency validation. Use with caution — missing dependencies will cause runtime errors.
--yesSkip the interactive confirmation prompt. Required when using --json (non-interactive mode).
--quietSuppress progress lines on stderr.
--timeoutOverride poll timeout. Large solutions take longer to import than to export.
 
shellcrm --json solution import itworx_dev.zip --overwrite --yes

04 The ALM pipeline: what just happened

You just proved the full dev-to-dev ALM cycle in one org (unusual but valid for lesson purposes). In a real project the same two commands span orgs:

  1. Dev org — build the customization. Run solution export itworx_dev -o itworx_dev.zip.
  2. Source control — commit the ZIP (or extracted customizations.xml) to Git. This is the artifact of record.
  3. CI pipeline — on PR merge, run solution import itworx_dev.zip --overwrite --yes targeting the test org.
  4. QA passes — run solution export itworx_dev -o itworx_dev_release.zip --managed.
  5. Prod — run solution import itworx_dev_release.zip --yes. The managed layer installs cleanly.
TIP
On-prem (M09): The same verbs, same envelopes, same flags. The only transport difference is NTLM auth vs. OAuth. When you get to M09, the export/import muscle memory you built here transfers directly.

Lesson check

4 questions · click to reveal

1. Why does solution export poll instead of returning immediately?

Correct. ExportSolutionAsync is always async — the server queues a system job, builds the ZIP in the background, and the CLI polls until the job reaches a terminal state before downloading.
Not quite. Export is always async regardless of size, data records are not scanned, and the CLI never compresses locally. The server builds the ZIP server-side; the CLI just polls and downloads.

2. The export envelope shows "managed": false. What does that mean for the import target?

Correct. managed: false means the ZIP carries an unmanaged solution. When imported, the components land as editable customizations on the target — appropriate for dev-to-dev moves, not for production delivery.
Not quite. The ZIP is valid. managed: false is the normal state for an unmanaged solution export — it simply means the target org will receive editable (unmanaged) components.

3. You're importing to a test org that already has an older itworx_dev. You omit --overwrite. What's the risk?

Correct. Without --overwrite (i.e. OverwriteUnmanagedCustomizations=false), the server protects existing unmanaged customizations on the target and rejects any component that would overwrite them.
Not quite. Missing --overwrite causes the server to fail the import when it finds conflicting unmanaged customizations — it doesn't skip silently, double, or roll back versions.

4. In a real dev → prod pipeline, which export flag do you add for the production import?

Correct. Managed solutions are the standard delivery vehicle for production. They install as a read-only layer, can be upgraded in place, and can be uninstalled cleanly. Never ship unmanaged to prod.
Not quite. --managed is the flag for production delivery. The other flags control verbosity, publishing, and dependency checks — not the managed/unmanaged nature of the ZIP.
